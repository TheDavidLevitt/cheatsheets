Computer Hardware Cheat Sheet

CPU · GPU · memory hierarchy · what matters for AI · started 2026-06-06 · twin of Tools/hardware_cheatsheet.md

0 · The mental model (read once)

Hardware = the physical machinery; software = the instructions it runs. Every design choice trades speed vs capacity vs cost — and in the AI era the bottleneck has moved from raw compute to memory bandwidth: how fast data can be fed to the compute.
The big picture — a few specialized blocks joined by buses (shared data highways; Latin omnibus, "for all" — many passengers, one road): 
        ┌───────────────────────────────────────────┐
        │                CPU package                │
        │ ┌────────┐ ┌────────┐    ┌─────────────┐  │
        │ │ Core 0 │ │ Core 1 │ …  │ L3 cache    │  │
        │ │ L1/L2  │ │ L1/L2  │    │(shared SRAM)│  │
        │ └────────┘ └────────┘    └─────────────┘  │
        └────────────────┬──────────────────────────┘
                         │ memory bus
              ┌──────────┴──────────┐
              │  RAM (DRAM)         │ ← working memory, volatile
              └──────────┬──────────┘
                         │ PCIe bus
     ┌──────────────┬────┴─────────┬───────────────┐
┌────┴─────┐  ┌─────┴──────┐  ┌────┴─────┐  ┌──────┴──────┐
│ GPU      │  │ NVMe SSD   │  │ Network  │  │ USB,        │
│ (+ VRAM) │  │(non-volat.)│  │ card     │  │ display, …  │
└──────────┘  └────────────┘  └──────────┘  └─────────────┘
CPU (Central Processing Unit) = the general-purpose brain. RAM (Random-Access Memory) = fast working memory, erased at power-off; "random access" = any address costs the same, unlike winding a tape. Storage (SSD/HDD) = slow but permanent. GPU (Graphics Processing Unit) = massively parallel accelerator with its own VRAM (Video RAM). Every bus has a bandwidth (GB/s) — often the real limit.
Von Neumann architecture (1945): programs and data share one memory; the processor fetches from it one stream at a time. Nearly every computer since follows it. Its famous flaw — the von Neumann bottleneck: the CPU↔memory link is the choke point. The whole memory hierarchy (§3) and AI accelerators (§2) are workarounds for this 80-year-old problem.

1 · CPUs — cores, clocks, caches, ISAs

Anatomy

Coreone complete processing unit; a "12-core CPU" = 12 mini-CPUs on one chip. Heat limits (~2005) forced going wide instead of fast.
Clock speedthe metronome, in GHz (billions of ticks/sec). Stuck near 3–5 GHz for 20 years. At 4 GHz, light travels only ~7.5 cm per tick.
Cachesmall fast on-chip memory (L1→L2→L3) so cores don't wait on RAM. French cacher, "to hide" — a stash near the worksite.
SMT / Hyper-ThreadingSimultaneous Multi-Threading: one core juggles 2 instruction streams, working on B while A stalls on memory. "8 cores / 16 threads" = 8 workers, 2 to-do lists each. Gain ~15–30%, not 2×.

ISA — Instruction Set Architecture

The vocabulary of machine instructions a CPU understands — the hardware/software contract. Code compiled for one ISA won't run natively on another.

x86-64descends from Intel's 8086 (1978). Chips made by Intel AND AMD — AMD (Advanced Micro Devices) is a competing vendor of x86 chips, not a separate ISA. Desktops, Windows laptops, classic servers.
AMD64 lorethe 64-bit extension is named after AMD because AMD designed it (Opteron, 2003) while Intel bet on the incompatible Itanium — Intel had to adopt its rival's design. x86-64 = AMD64 = x64.
ARMoriginally Acorn RISC Machine (UK, 1985). Arm Ltd. licenses designs; others build: Apple (M-series "Apple Silicon"), Qualcomm (phones), Amazon (AWS Graviton cloud servers).

RISC vs CISC

CISC (Complex Instruction Set Computer)rich instructions that each do a lot; made sense when memory was scarce and humans wrote assembly. Example: x86-64.
RISC (Reduced Instruction Set Computer)few simple uniform instructions; complexity moves to the compiler (Berkeley/Stanford, early 1980s). Examples: ARM, RISC-V (open-source, license-free).
Todayblurred: modern x86 decodes CISC instructions into RISC-like micro-ops internally. Practical difference = licensing + ecosystem.

2 · Accelerators — GPU, TPU, NPU, ASIC vs FPGA

Why GPUs took over AI: a CPU = few big clever cores tuned for latency; a GPU = thousands of small cores tuned for throughput — the same operation across huge arrays (SIMD: Single Instruction Multiple Data). Graphics applies the same math to millions of pixels; deep learning is the same shape — neural nets are mostly matrix multiplications. A video-game chip (made programmable by Nvidia's CUDA, 2007) became the AI engine by accidental fit.

The accelerator zoo

GPU — Graphics Processing Unitthousands of small cores + very-high-bandwidth memory; programmable (CUDA, ROCm, Metal). Gaming → AI datacenters.
TPU — Tensor Processing UnitGoogle's ASIC for tensor (multi-dimensional array) math — a hard-wired multiply-accumulate grid ("systolic array"). Less flexible, more efficient. Google Cloud only.
NPU — Neural Processing Unitsmall power-sipping neural accelerator inside consumer chips for on-device AI — phone SoCs, Apple "Neural Engine", Copilot+ PCs.

ASIC vs FPGA

ASIC — Application-Specific Integrated Circuitpermanently etched for one job. Fastest/most efficient; huge upfront cost, cheap at volume. TPUs, Bitcoin miners.
FPGA — Field-Programmable Gate Arraylogic blocks you can rewire after manufacture ("in the field"). Flexibility tax on speed; no fab cost, pricey per chip. Prototyping, evolving algorithms.
Why LLM work is memory-bandwidth hungry: generating each token reads essentially all model weights from memory (70B parameters × 2 bytes ≈ 140 GB per token). The math units finish long before the next weights arrive — the von Neumann bottleneck at industrial scale. Inference speed ≈ bandwidth ÷ model size, hence HBM on AI GPUs, and why GB/s beats TFLOPS (trillions of floating-point ops/sec) as the spec to watch. Training adds a fit problem (weights + gradients + optimizer state) → multi-GPU clusters.

3 · Memory hierarchy — the speed/size/cost pyramid

No memory is fast AND large AND cheap, so computers stack layers — tiny-fast near the core, huge-slow further out, each layer caching the one below.

  ▲ faster, smaller, $$$/byte
  │   registers      ~1 KB        (inside each core)
  │   L1 cache       ~64 KB/core    ┐
  │   L2 cache       ~1 MB/core     ├ SRAM
  │   L3 cache       ~32 MB shared  ┘
  │   RAM (DRAM)     8–192 GB
  │   SSD (NVMe)     0.5–8 TB
  │   HDD / network  ~∞
  ▼ slower, bigger, ¢/byte

SRAM vs DRAM — why caches are small

SRAM (Static RAM)6 transistors/bit in a stable flip-flop — "static" = holds state while powered. Very fast, 6× the silicon → expensive → kept small (L1/L2/L3).
DRAM (Dynamic RAM)1 transistor + 1 capacitor/bit — the charge leaks, needs constant refresh ("dynamic"). Dense and cheap → gigabytes, but slower.
DDR5Double Data Rate gen 5 — transfers on both clock edges (hence "double"); current CPU main memory.
HBMHigh Bandwidth Memory — DRAM dies stacked vertically right next to the GPU with a very wide connection; standard on AI accelerators.

Storage & volatility

SSD (Solid-State Drive)flash chips, no moving parts; NVMe (Non-Volatile Memory Express) attaches it straight to the PCIe bus, replacing disk-era SATA.
HDD (Hard Disk Drive)spinning magnetic platters + moving head — 1950s tech, still cheapest per TB; bulk/archive only.
Volatileregisters/SRAM/DRAM — contents fly away at power-off (Latin volare, "to fly").
Non-volatileSSD/HDD — survives power-off. Booting = copying the OS from non-volatile storage into volatile RAM.

The latency ladder — stretch 1 CPU cycle (~0.3 ns) to 1 second

1 CPU cycle~0.3 ns → 1 second
L1 / L2 / L3 cache hit~1 / 4 / 12 ns → ~3 s / 13 s / 40 s
RAM (DRAM) access~100 ns → ~5 minutes
NVMe SSD read~100 µs → ~4 days
HDD seek~10 ms → ~13 months
Network round-trip (same continent)~50 ms → ~5 years

A cache miss = the core "goes out for coffee"; a disk read = "a long vacation". This is why caches and SMT exist, and why data layout dominates real performance.

4 · Practical decoder — reading a spec sheet

Fictional listing: "M-series chip: 12-core CPU, 16-core GPU, 16-core Neural Engine, 64 GB unified memory, 400 GB/s bandwidth, 1 TB SSD."

Decoded

12-core CPU12 general cores, often split performance + efficiency cores (big fast vs small frugal).
16-core GPU16 GPU clusters of many small ALUs — vendors count differently; compare within a family only.
16-core Neural Enginethe NPU — on-device AI features, small-model inference.
64 GB unified memoryone RAM pool shared by CPU + GPU + NPU on the same package — no separate VRAM, no PCIe copying. Effectively a 64 GB "VRAM" budget, vs 8–24 GB on typical discrete GPU cards. The headline number for local LLMs.
400 GB/s bandwidthhow fast that pool can be read — the tokens-per-second throttle.
1 TB SSDaffects model loading and how many models fit, not generation speed.

What matters for local LLM inference — in order

1 · RAM/VRAM sizethe model must fit: parameters × bytes/parameter + ~20%. 8B model quantized (weights compressed to fewer bits) at 4-bit ≈ 5 GB; 70B at 4-bit ≈ 40 GB — fine in 64 GB unified memory, impossible in 16 GB.
2 · Memory bandwidthsets tokens/sec once it fits — every token re-reads the weights. 400 GB/s ÷ 40 GB model ≈ ~10 tokens/s ceiling.
3 · Compute (TFLOPS)matters for prompt ingestion (parallelizes well) more than generation.
4 · CPU cores, SSDeverything around the model: loading, tokenization, your other apps.

5 · Teardown / safe shutdown

shutdown beats pulling the plugthe OS flushes dirty caches (writes still sitting in RAM) to disk first — a power-cut can leave files half-written, forcing fsck/repair or corruption
Eject external drives before unpluggingwrites are write-back cached: "copy complete" may mean "in RAM, not yet on the disk." Eject forces the flush
Sleep vs hibernatesleep keeps RAM powered (it's volatile — battery dies, state gone); hibernate copies RAM to disk and powers off fully, so it survives an empty battery
SSD disposaldeleting files ≠ erasing them (data remanence); use the drive's secure-erase, or full-disk encryption from day one and destroy the key
