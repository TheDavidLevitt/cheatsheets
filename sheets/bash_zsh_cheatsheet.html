Bash & Zsh Cheat Sheet

The shell: pipes · variables · config · scripts · jobs · started 2026-06-06 · twin of Tools/bash_zsh_cheatsheet.md

0 · The mental model (read once)

Shell = the program that reads your typed commands, interprets them, and asks the kernel (the OS core that talks to hardware) to run them. The name is the metaphor: it's the outermost layer "wrapping" the kernel.

Terminal = the window program (Terminal.app, iTerm2, the VS Code panel) that draws text on screen and sends your keystrokes to a shell. Terminal = the phone; shell = the person answering. Historically a terminal was a physical device (keyboard + screen) wired to a mainframe — today's "terminal emulators" imitate one in software.
bash = Bourne Again SHell (1989) — a pun: it's the successor to Stephen Bourne's 1979 sh, "born again" as free GNU software. The lingua franca of scripting; nearly every Linux box and script assumes it.

zsh = Z SHell (Paul Falstad, 1990; named after a Yale professor's login ID zsh). A bash-compatible interactive shell with better completion, globbing, and theming (Oh My Zsh). Default on macOS since Catalina (2019).

CLI = Command-Line Interface — text in, text out; the opposite of a GUI.
You → terminal → shell → kernel: 
  you type        terminal          shell             kernel
  "ls -l"   →   (the window)  →  (bash/zsh:      →  (executes,
                 passes keys      parses, finds      talks to
                 to the shell     /bin/ls, runs      hardware)
                                  it, pipes I/O)

Two kinds of commands the shell can run

Built-insrun inside the shell process itself (cd, export, alias, history, exit). They must be built-in because they change the shell's own state — an external program couldn't change your directory.
External programsseparate binaries on disk, found via $PATH (ls, grep, cat, python3). Stored in /bin, /usr/bin, etc. Same on any shell; only the built-ins differ.
type <cmd>tells you which one you're dealing with (type cd → "shell builtin"; type ls → "/bin/ls")

bash vs zsh — what actually matters day-to-day

bash · zsh
Default prompt ends in$ · %
Default onLinux, scripts, servers · macOS (since 2019)
Config file~/.bashrc / ~/.bash_profile · ~/.zshrc / ~/.zprofile
Tab completionbasic · smarter (menus, typo-correction)
Extended globbing **opt-in (shopt -s globstar) · on by default
The headline: zsh runs ~all bash commands. Everything in this sheet works in both unless flagged. Write scripts in bash (#!/bin/bash) for portability; type into zsh on the Mac. (Root prompt is # in both — one reason comments also start with #: anything pasted from a root prompt becomes harmless.)

1 · Command grammar (read once)

Every command line follows: command [options/flags] [arguments]
• command — a built-in, or a program found on $PATH.
• flags — short -l or long --all; short flags bundle: ls -la = ls -l -a.
• arguments — what to act on (files, paths, strings).
• man <cmd> opens the manual (quit with q); cmd --help is the short version.

Recurring flag vocabulary (worth memorizing once)

-r / -Rrecursive — dir + everything inside (cp -r, rm -r, grep -r)
-fforce — skip prompts, ignore missing (rm -f, mv -f)
-vverbose — narrate every action (cp -v, rm -v)
-iinteractive — ask "are you sure?" (rm -i, mv -i)
-ndry run — show what would happen (rsync -n)
-qquiet — suppress output (grep -q, curl -q)

2 · Navigation & file ops (recap, concise)

Move & make

pwdprint working directory — "where am I?"
ls -lalist all (-a incl. dotfiles), long format (-l)
cd dir / cd .. / cd - / cdenter / up one / back to previous / home
mkdir -p a/b/cmake directory, -p creates parents as needed

Read & find

cat fileconcatenate — dump file(s) to screen
less filepage through a file (q quits) — "less is more" (it replaced more)
head -n 20 / tail -n 20first / last 20 lines; tail -f follows live logs

Path shorthand: ~ = home (/Users/<you>), . = here, .. = parent, - = previous dir (for cd), / = filesystem root.

3 · Pipes | and redirection > >> < 2> &>

The shell wires three standard streams to every command — each has a number (file descriptor): stdin (0, input), stdout (1, normal output), stderr (2, error output). Redirection re-plugs those wires.

Redirection grammar

cmd1 | cmd2pipe: stdout of cmd1 → stdin of cmd2 — history | grep ssh
cmd > filesend stdout to file, overwrite — ls > listing.txt
cmd >> filesend stdout to file, append — echo "note" >> log.txt
cmd < filefeed file in as stdin — sort < names.txt
cmd 2> filesend stderr (stream 2) to file — make 2> errors.txt
cmd 2>&1merge stderr into stdout (&1 = "where 1 points") — cmd > all.txt 2>&1
cmd &> fileboth streams to file (bash/zsh shortcut for the above) — cmd &> all.txt
cmd 2> /dev/nulldiscard errors — /dev/null is the system's black hole — find / -name x 2>/dev/null
cmd | tee filestdout to screen and file (a T-junction in the pipe) — make | tee build.log
Reading 2>&1: "redirect stream 2 to wherever stream 1 currently goes". Order matters — > f 2>&1 works; 2>&1 > f doesn't do what you want.
Pipes are the Unix idea (Doug McIlroy, 1973): small programs, each doing one thing, composed like plumbing: 
cat access.log | grep 404 | sort | uniq -c | sort -rn | head

4 · Chaining && || ; and globbing * ? ** [a-z]

Chaining — control flow on one line

a && brun b only if a succeeded (exit code 0) — "and then"
a || brun b only if a failed — "or else"
a ; brun b regardless — just a separator
a && b || crough if/else: b on success, c on failure

Every command exits with a status code: 0 = success, anything else = failure (Unix optimism: one way to succeed, many ways to fail). $? holds the last command's code — echo $? to inspect.

Globbing — wildcard filename expansion

*any string, any length (not hidden dotfiles) — *.py
?exactly one character — file?.txt
[a-z], [0-9], [abc]one character from the set/range
**recursive: this dir and all subdirs — ls **/*.md (zsh default; bash needs shopt -s globstar)
{a,b}brace expansion, literal alternatives — cp file.{txt,bak}

The shell (not the command) expands these before the command runs — rm *.txt becomes rm a.txt b.txt …. "Glob" comes from the original Unix helper /etc/glob ("global").

5 · Variables, $PATH, and export

NAME="value"        # set (NO spaces around = — the #1 beginner error)
echo $NAME          # read: $ means "the value of"
export NAME         # mark it for inheritance by child processes
export NAME="v"     # set + export in one line
unset NAME          # delete
Why export? A plain variable lives only in this shell. Programs you launch are child processes and inherit only exported (environment) variables. env lists them; echo $$ shows the current shell's process ID (PID).
$PATH = a colon-separated list of directories the shell searches, in order, left to right, to find a command: 
echo $PATH                              # see it
export PATH="$HOME/bin:$PATH"           # prepend your dir (this session only)
Grammar of that line: set PATH to your new dir + : + everything PATH already was.
Forgetting :$PATH wipes the old value — suddenly ls itself "command not found". Prepending means your dir wins ties; appending ($PATH:$HOME/bin) makes it the fallback.
To make it permanent, put the export line in a startup file (§9). macOS also seeds the system-wide PATH from /etc/paths and /etc/paths.d/ (one dir per line, assembled by path_helper — see man path_helper).

Useful built-in variables: $HOME $USER $SHELL (login shell) $PWD $? (last exit code) $$ (this shell's PID)

6 · Quoting rules — 'single' vs "double" vs $(command)

'single quotes'literal — nothing expands — echo '$HOME' → $HOME
"double quotes"variables expand, spaces preserved — echo "$HOME" → /Users/you
$(command)command substitution — run it, paste its output — echo "today: $(date)"
`command`old backtick syntax for the same — avoid (can't nest) — echo `date`
\xescape one character — echo \$HOME → $HOME
Rules of thumb:
• Default to double quotes around any variable: rm "$file" — protects against spaces in values (my file.txt would otherwise become two arguments).
• Single quotes when you want the text exactly as typed (regexes for grep, literal $).
• $(...) over backticks, always — it nests: $(dirname $(which python3)).

7 · History tricks — !!, !$, ctrl-r

The shell records commands (zsh: ~/.zsh_history; bash: ~/.bash_history). ! is the history expansion character ("bang").

Bang expansions

historylist recent commands (numbered)
!!rerun the whole last command — classic: sudo !! after a permission error
!$the last argument of the previous command — mkdir deep/path then cd !$
!*all arguments of the previous command
!42 / !greprerun command #42 / the last command starting with "grep"

Line-editing keys

ctrl rreverse-incremental search — type a fragment, keep pressing ctrl-r to cycle back; Enter runs, Esc edits
ctrl ckill the current foreground process (SIGINT)
ctrl a / ctrl ejump to start / end of line (Emacs keybindings)
ctrl u / ctrl kdelete to start / end of line
ctrl lclear screen (same as clear)
ctrl r alone replaces 90% of arrow-key archaeology. (sudo = superuser do — run one command as root.)

8 · Aliases & functions

alias gs='git status'           # alias = simple text substitution
alias ll='ls -lah'
alias ..='cd ..'
unalias gs                      # remove
alias                           # list all

mkcd() { mkdir -p "$1" && cd "$1"; }    # function: takes arguments ($1)
Alias = dumb prefix replacement — fine for shortening a fixed command. Function = a real mini-script — use one the moment you need arguments or logic (aliases can't take $1). Both live only in the current session until you put them in your config file (next section).

9 · Config files decoded — which runs when

The suffix rc = "run commands" (from the 1960s CTSS runcom facility) — a script the shell runs at startup. The split that confuses everyone:

• Login shell — the first shell of a session (SSH login, and every new macOS Terminal tab, which macOS treats as a login).
• Interactive non-login shell — a shell started from a shell (typing bash/zsh, most Linux terminal tabs).

Which file, read by what, for what

~/.zshrcevery interactive zsh — ← your stuff: aliases, functions, PATH, prompt (macOS: this is the one)
~/.zprofilezsh login shells only — one-time-per-login env setup (Homebrew puts its PATH line here)
~/.bashrcinteractive non-login bash — aliases/PATH on Linux
~/.bash_profilebash login shells only — macOS-bash legacy; convention: have it source ~/.bashrc so both paths load one file
/etc/zshrc, /etc/profilesystem-wide versions, all users (root-owned; rarely touched)
Practical rules: on the Mac, edit ~/.zshrc; in bash-land, edit ~/.bashrc and let ~/.bash_profile source it. Changes apply to new shells — load them into the current one with source ~/.zshrc (. is a synonym: . ~/.zshrc). cat /etc/shells lists installed shells; chsh -s /bin/bash changes your login shell.

10 · Minimal scripting — shebang, $1, one-liners, chmod +x

A script = commands in a text file. Three steps:

#!/bin/bash                      # 1. shebang: which interpreter runs this file
echo "Hello, $1!"                # 2. the script ($1 = first argument)
chmod +x greet.sh                # 3. make it executable (+x = add execute permission)
./greet.sh Ada                 # run it → Hello, Ada!
Shebang = #! — "sharp" (#) + "bang" (!). Must be line 1: the kernel reads it to pick the interpreter (#!/bin/bash, #!/usr/bin/env python3 — the env form finds the interpreter via PATH, more portable). The ./ is needed because the current dir is deliberately not on $PATH (a security choice). chmod = change mode (the file's permission bits).
Positional parameters: $1 $2 … = arguments; $0 = script name; $# = how many; "$@" = all of them, properly quoted. 
# if — note the spaces inside [ ] are mandatory ([ is itself a command)
if [ -f config.txt ]; then echo "found"; else echo "missing"; fi

# for — loop over a glob
for f in *.txt; do echo "processing $f"; done

# common tests: -f file exists, -d dir exists, -z string empty,
#               = string equal, -eq numbers equal
Top of every serious bash script: set -euo pipefail — die on any error (-e), on unset variables (-u), and on failures inside pipes.

11 · Job control — ctrl-z, bg, fg, &, jobs, kill

The shell can juggle several processes ("jobs") in one terminal:

cmd &start in the background — prompt returns immediately
ctrl zsuspend (pause) the foreground job (SIGTSTP) — it's frozen, not dead
jobslist this shell's jobs with numbers [1], [2]…
bg / bg %2resume a suspended job, in the background
fg / fg %2bring a job back to the foreground
kill %1terminate job 1 (politely: SIGTERM)
kill -9 <PID>force-kill by process ID (SIGKILL — can't be ignored; last resort)
ps aux | grep namefind a process's PID system-wide
cmd & disown / nohup cmd &detach a job so it survives closing the terminal
The classic move: started a long command without &? → ctrl z then bg — same result, after the fact.
Note ctrl z ≠ ctrl c: suspend vs kill. Background jobs die with their terminal unless disowned/nohup'd (nohup = "no hangup" — ignore the terminal-closed signal, named for the days when hanging up a modem killed your processes).

12 · Teardown — getting out

exit / ctrl dleave the shell — ctrl d = EOF on stdin ("no more input, we're done")
ctrl ckill the foreground process (SIGINT) — not ctrl z, which only suspends (§11)
ctrl z then kill %1a suspended job still holds resources — finish it off (jobs lists the numbers)
kill <PID>SIGTERM — polite "please shut down"; the process can clean up first
kill -9 <PID>SIGKILL — the kernel removes the process; can't be caught or ignored, no cleanup. Last resort
history -cwipe this session's in-memory history (bash; zsh: history -p) — the history file is untouched
␣cmd (leading space)keep a command out of history — needs HISTCONTROL=ignorespace (bash) / setopt HIST_IGNORE_SPACE (zsh). Handy for secrets
Stuck terminal tab? Escalate in order: ctrl c → ctrl d → close the tab. Closing sends SIGHUP to its jobs — anything not disowned/nohup'd dies with it (§11).
