1 · The compute spectrum — managed → control

The one rule that answers half the exam questions: pick the most managed option that still fits the requirements. Only move right on the spectrum (toward raw VMs) when the requirement forces you — a kernel module, a specific OS, a license tied to physical hardware, a legacy daemon that can't be containerized. 
 most managed ◄──────────────────────────────────────────────► most control

"Serverless" defined properly: there are servers — like "wireless" networks still have wires somewhere. Serverless means you don't manage servers: the platform autoscales to zero (no traffic → no instances → no bill), bills per use (per request / per 100ms of CPU), and handles patching, capacity, and load balancing. The trade: less control over the runtime environment.

Cloud Functions (now "Cloud Run functions")

FaaS (Function as a Service): you write a single function in a supported language (Python, Node.js, Go…); GCP wraps, deploys, and scales it. Triggered by HTTP requests or events (a file lands in a Cloud Storage bucket, a Pub/Sub message arrives). Smallest unit of compute — glue code, webhooks, lightweight ETL (Extract-Transform-Load).

docs: cloud.google.com/functions

Cloud Run — serverless containers

You hand GCP a container image (any language, any binary — if it fits in a container, it runs); GCP runs it serverlessly: scale-to-zero, scale-out under load, pay-per-use. Two shapes:

Servicelistens for HTTP requests; stays up while traffic flows; the default web-app/API answer
Jobruns to completion then exits — batch work, migrations, scheduled tasks; no HTTP endpoint

docs: cloud.google.com/run

App Engine — the legacy note

GCP's original PaaS (Platform as a Service, 2008): deploy source code, platform does the rest. Two flavors (Standard = sandboxed runtimes, Flexible = container on a managed VM). Still supported and still on the exam, but for new projects Google steers you to Cloud Run, which does the same job with fewer constraints. Recognize it; don't reach for it first.

docs: cloud.google.com/appengine

GKE — Google Kubernetes Engine

Managed Kubernetes (the open-source container orchestrator Google created — Greek κυβερνήτης, "helmsman"; abbreviated K8s, K + 8 letters + s). Choose it when one container isn't enough: many cooperating services, custom networking, fine-grained scheduling, or portability to other clouds. Autopilot mode manages the nodes for you; Standard mode gives you the node VMs. If Cloud Run fits, prefer Cloud Run — GKE is power with operational cost.

docs: cloud.google.com/kubernetes-engine

Compute Engine — full VMs (IaaS)

IaaS (Infrastructure as a Service): virtual machines with your choice of CPU, memory, disks, and OS. You get root; you also get the patching, scaling, and 3-a.m. pages. The answer when you need OS-level control, lift-and-shift of existing servers, GPUs with custom drivers, or software that can't be containerized. The rest of this sheet drills into it.

docs: cloud.google.com/compute

Exam decision reflex

"event-driven", "trigger on upload"Cloud Functions
"containerized", "stateless HTTP", "scale to zero"Cloud Run
"microservices", "orchestration", "portability"GKE
"lift and shift", "custom OS/kernel", "specific license"Compute Engine

2 · Compute Engine machinery — machine types, images, Spot

Machine families & types — decoding e2-small

A machine type name reads family-shape (sometimes family-shape-vCPUs). The letter says the optimization, the digit says the generation:

PrefixMnemonic & use
e2, n2, n4General-purpose. E = economy/efficient (cheapest, dev/test, small services); N = normal balanced workhorse. e2-micro, e2-small, e2-medium are shared-core sizes for learning labs.
c3, c4Compute-optimized: highest sustained CPU performance — gaming servers, HPC (High-Performance Computing), media transcoding.
m3, m4Memory-optimized: huge RAM-to-CPU ratios — in-memory databases like SAP HANA.
a3, g2Accelerator / GPU-attached: machine learning training and inference.
t2a, c4aArm-based (a-suffix): Arm64 CPUs (Ampere, Google Axion) — cheaper per core if your software has arm64 builds.

Example: n2-standard-4 = N family, gen 2, standard CPU:RAM ratio, 4 vCPUs. Custom machine types let you dial exact vCPU/RAM when no preset fits.

docs: machine families resource guide

Boot images & image families

Every VM boots from an image — a prebuilt disk containing an OS. Images live in per-OS public projects (debian-cloud, ubuntu-os-cloud, rocky-linux-cloud, windows-cloud…). Specific images get dated names (debian-12-bookworm-v20250610) and eventually deprecate; an image family is a moving pointer to the newest non-deprecated image in a line:

# family → always-current image, the right thing to reference in scripts
gcloud compute instances create my-instance-01 \
  --image-family=debian-12 --image-project=debian-cloud

# Ubuntu family names encode version + arch:
#   ubuntu-2404-lts-amd64   ← x86-64 build
#   ubuntu-2404-lts-arm64   ← Arm build (must match an Arm machine type!)

Default: omit the image flags and you get the latest Debian LTS (Long-Term Support) from debian-cloud — Debian is Compute Engine's house default. The amd64/arm64 suffix is the CPU architecture; amd64 = x86-64 (AMD shipped the 64-bit x86 extension first, hence the name — it runs on Intel too).

docs: OS images & families

Spot VMs (née preemptible)

Spot VMs use Google's spare capacity at a 60–91% discount — but GCP can preempt (reclaim) them at any time with a 30-second warning. The older preemptible VM model is the same idea with a hard 24-hour maximum lifetime; Spot is its successor (no 24h cap). Both terms appear on the exam.

Good forbatch jobs, CI (Continuous Integration) runners, fault-tolerant data processing, anything restartable
Never fordatabases, single-instance production services, anything that can't die mid-task
Flag--provisioning-model=SPOT on instances create

docs: Spot VMs

3 · VM-creation abstractions — instance vs template vs machine image vs snapshot

Five things that all sound like "a copy of a VM" but aren't. This table is exam gold.

ThingWhat it actually isContains data?When it's the answer
VM instance the live, running (or stopped) machine itself yes — it is the data you need… a computer
Instance template a recipe: machine type, image, disks-to-create, network, metadata — config only. Free, immutable (edit = create new version), global resource. no — zero disk data, just settings feeding a MIG (§4); stamping out identical VMs; "standardize VM creation"
Machine image a full clone: configuration plus the data of all attached disks, in one resource yes — every disk back up or migrate an entire VM, multi-disk included; "restore the whole machine"
Disk snapshot point-in-time copy of one disk; incremental (later snapshots store only changed blocks); can be scheduled yes — one disk routine disk backup; "scheduled backups of the data disk"
Custom (disk) image a reusable boot-disk golden image: one disk, prepared (software baked in), used to create new VMs yes — one disk "pre-installed software on every new VM" — bake once, boot many
Sole-tenant node a dedicated physical host — your VMs share the hardware with no other customer n/a (it's a host) compliance isolation; BYOL (Bring Your Own License) software licensed per physical core/socket
Memory hooks: template = recipe (no food in a recipe). Machine image = the whole frozen meal. Snapshot = one frozen dish. Custom image = a dish you mass-produce from. Sole-tenant = private kitchen.
Classic exam trap: "copy a VM including its data" → machine image, not instance template (templates carry no data). "Standardize config for an autoscaled group" → instance template. "Windows Server license tied to physical cores" → sole-tenant node.

4 · MIGs — Managed Instance Groups

A MIG is a pool of identical VMs created from one instance template, managed as a unit. It's the standard exam answer for "scalable, resilient, stateless workload on VMs."

What a MIG gives you

Autoscalingadd/remove instances on CPU utilization, load-balancer load, or custom metrics; set min/max size
Autohealinga health check probes each VM; failed VMs are deleted and recreated from the template (not rebooted — recreated, so don't store state on them)
Rolling updatesswap in a new template gradually (rolling-action start-update); canary a subset first
Regional MIGsspread instances across multiple zones in a region — survives a zone outage
Load balancinga MIG is the natural backend of a GCP load balancer

Minimal MIG, end to end

# 1. the recipe
gcloud compute instance-templates create web-template \
  --machine-type=e2-small \
  --image-family=debian-12 --image-project=debian-cloud \
  --tags=http-server

# 2. the group (regional, 2–10 instances)
gcloud compute instance-groups managed create web-mig \
  --template=web-template --size=2 --region=europe-west1

# 3. autoscale on CPU
gcloud compute instance-groups managed set-autoscaling web-mig \
  --region=europe-west1 --min-num-replicas=2 \
  --max-num-replicas=10 --target-cpu-utilization=0.6
Unmanaged instance groups also exist: a bag of heterogeneous, hand-made VMs grouped only for load balancing. No template, no autoscaling, no autohealing. If the exam says "existing dissimilar VMs behind one load balancer," that's the one — otherwise the answer is always the managed kind.

docs: instance groups

5 · Customization at boot — metadata, startup scripts, cloud-init

How a freshly cloned image becomes your server, without baking a custom image.

Instance metadata — the VM's key-value config store

Every VM gets a metadata server — an internal HTTP endpoint (metadata.google.internal, IP 169.254.169.254) serving key-value pairs about and for the instance. Two layers: project-wide metadata (applies to all VMs in the project) and instance metadata (this VM only; wins on conflict). GCP itself uses it for SSH keys (§6) and service-account tokens (§7); you use it to pass config:

gcloud compute instances add-metadata my-instance-01 \
  --metadata=env=staging,owner=alex@example.com

# read it from inside the VM (the header is mandatory):
curl -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/attributes/env"

Startup scripts — run as root, every boot

A script stored in metadata under the key startup-script; the guest environment runs it as root on every boot (not just the first). Make it idempotent — safe to re-run.

# inline
gcloud compute instances create my-instance-01 \
  --metadata=startup-script='#!/bin/bash
apt-get update && apt-get install -y nginx'

# or from a local file
gcloud compute instances create my-instance-01 \
  --metadata-from-file=startup-script=startup.sh

# debug: where did my script's output go?
sudo journalctl -u google-startup-scripts.service

There's a matching shutdown-script key (best-effort, ~90s budget on normal stops — and only the 30s warning window on Spot preemption).

cloud-init — declarative first-boot config

cloud-init is the cross-cloud standard for first-boot VM configuration (Ubuntu and Container-Optimized OS images run it natively). Instead of an imperative script, you declare users, packages, and files in YAML, passed via the metadata key user-data:

gcloud compute instances create my-instance-01 \
  --image-family=ubuntu-2404-lts-amd64 \
  --image-project=ubuntu-os-cloud \
  --metadata-from-file=user-data=cloud-init.yaml

# cloud-init.yaml
#cloud-config
packages: [nginx]
runcmd:
  - systemctl enable --now nginx

Which runs when: cloud-init's config phases run on first boot (per instance); GCP startup-scripts run every boot. For ACE, startup scripts are the expected answer; cloud-init is the portable power tool.

6 · Connecting — gcloud ssh and IAP tunnels

What gcloud compute ssh actually does

It's not magic — it's automated key plumbing. First run:

gcloud compute ssh my-instance-01 --zone=europe-west1-b
  1. Keygen: generates an SSH keypair at ~/.ssh/google_compute_engine if none exists.
  2. Metadata push: uploads the public key into project (or instance) metadata under ssh-keys.
  3. Guest agent: the guest environment (an agent baked into Google's images) watches metadata, sees the new key, creates a matching Linux user on the VM and installs the key in its authorized_keys.
  4. Connect: plain ssh to the VM's external IP as that user.

That's why SSH "just works" with no password setup — and why it doesn't work on images without the guest agent. OS Login (enable-oslogin=TRUE metadata) is the modern upgrade: keys live on your Google identity and access is controlled by IAM (Identity and Access Management) roles instead of raw metadata.

IAP tunneling — SSH with no external IP

IAP (Identity-Aware Proxy) lets Google's frontend proxy your SSH/RDP traffic to a VM's internal IP after checking your Google identity. The security win: VMs need no external IP at all — no internet-facing port 22.

gcloud compute ssh my-instance-01 --zone=europe-west1-b \
  --tunnel-through-iap

Three prerequisites (each one is an exam question):

1 · APIenable the IAP API (iap.googleapis.com) in the project
2 · IAM rolecaller needs roles/iap.tunnelResourceAccessor (plus normal instance access)
3 · Firewallallow ingress from 35.235.240.0/20 (Google's fixed IAP source range) on tcp:22
gcloud compute firewall-rules create allow-iap-ssh \
  --direction=INGRESS --action=ALLOW --rules=tcp:22 \
  --source-ranges=35.235.240.0/20

docs: IAP TCP forwarding

Memory hook: the IAP range 35.235.240.0/20 — "35, 235: same digits, one borrows a 2." It's the only CIDR (Classless Inter-Domain Routing) block worth memorizing for the ACE.

7 · Identity — every VM is somebody

A VM doesn't call GCP APIs as you; it calls them as its attached service account — a robot identity with its own email and IAM roles. Code on the VM gets tokens from the metadata server, no key files needed:

# from inside the VM — who am I?
curl -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/email"
# → vm-runtime-sa@example-project-123.iam.gserviceaccount.com

# get an OAuth access token (what client libraries do under the hood)
curl -H "Metadata-Flavor: Google" \
  "http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token"
Default Compute Engine service account caution. New VMs default to PROJECT_NUMBER-compute@developer.gserviceaccount.com, which historically carries the project-wide Editor role — far too much power. One compromised VM = an editor of your whole project. Best practice (and the exam answer): create a dedicated least-privilege service account per workload and attach it explicitly: 
gcloud iam service-accounts create app-runtime-sa
gcloud compute instances create my-instance-01 \
  --service-account=app-runtime-sa@example-project-123.iam.gserviceaccount.com \
  --scopes=cloud-platform   # let IAM roles, not legacy scopes, do the limiting

Two-layer model worth keeping straight: access scopes (legacy, per-VM OAuth ceiling) ∩ IAM roles (the real permissions on the service account) = what the VM can actually do. Modern practice: scope cloud-platform (wide open) and constrain with tight IAM roles.

docs: service accounts for instances

8 · Command quick reference

Grammar reminder: gcloud compute <resource> <verb> [name] [--flags]. Add --zone/--region or set defaults with gcloud config set compute/zone.

gcloud compute instances create my-instance-01 --machine-type=e2-small --zone=europe-west1-bcreate a VM (default = latest Debian, default service account)
gcloud compute instances listall VMs in the project, every zone
gcloud compute instances describe my-instance-01full config dump (IPs, disks, metadata, service account)
gcloud compute instances stop / start my-instance-01stop = no CPU/RAM billing (disks still bill); start it again later
gcloud compute instances delete my-instance-01gone (boot disk too, unless --keep-disks)
gcloud compute ssh my-instance-01 --zone=…SSH with automatic key plumbing (§6); add --tunnel-through-iap for no-external-IP VMs
gcloud compute instance-templates create web-template --machine-type=e2-small …create the reusable recipe (config only, free)
gcloud compute instance-groups managed create web-mig --template=web-template --size=3MIG from a template
gcloud compute machine-images create my-backup --source-instance=my-instance-01full-VM clone: config + all disk data
gcloud compute disks snapshot my-data-disk --snapshot-names=data-snap-01point-in-time copy of one disk
gcloud compute images list --filter="family:debian"browse public boot images; --filter takes key:value or expressions like family~ubuntu-24
gcloud compute machine-types list --zones=europe-west1-b --filter="name~e2"what machine types exist in a zone
Wallet hook: stop pauses compute billing but disks and static IPs keep billing; delete is the only true zero. For practice projects: create, play, delete.
