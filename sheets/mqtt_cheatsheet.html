MQTT Cheat Sheet

Lightweight pub/sub for IoT & sensor telemetry · started 2026-06-12 · twin of Tools/mqtt_cheatsheet.md

0 · The mental model (read once)

MQTT (Message Queuing Telemetry Transport) = a lightweight, binary publish/subscribe protocol over TCP (Transmission Control Protocol). Built for telemetry: many small messages from many devices over slow/flaky networks. The "Queuing" is legacy naming — there's no real queue in the core protocol today. IoT (Internet of Things) = networked physical devices/sensors; MQTT is a common protocol beneath it.
Publish/subscribe through a central broker — the single most important idea: 
PUBLISHERS            BROKER             SUBSCRIBERS
(data sources)     (the post office)    (consumers)

sensor ──publish──► ┌────────────┐ ──deliver──► dashboard
sensor ──publish──► │  topic =   │ ──deliver──► database
gateway ─publish──► │ rendezvous │ ──deliver──► alarm svc
                    └────────────┘
          fans each message out to every matching subscriber
A device publishes to a named topic (plant/well07/pressure); clients subscribe to it; the broker fans every message out. Nobody addresses anybody directly — the topic is the rendezvous.
Why pub/sub, not HTTP request/response? Thousands of battery sensors on a flaky radio link. With HTTP (HyperText Transfer Protocol) each device must know every consumer's address, open a heavy connection, retry. With MQTT each device keeps one light long-lived connection to the broker and fires small messages at a topic. Add a new consumer? It just subscribes — the devices change nothing. That decoupling (in space and time) is the whole payoff.

1 · History

1999invented by Andy Stanford-Clark (IBM) & Arlen Nipper for SCADA (Supervisory Control And Data Acquisition) monitoring of oil & gas pipelines over costly low-bandwidth satellite links — remote-equipment telemetry was the original use case.
"MQ" namenods to IBM's MQ Series message-queue product it descended from; modern MQTT has no durable core queue, so the name is historical.
2013standardisation moves to OASIS (Organization for the Advancement of Structured Information Standards), an open-standards body.
3.1.1 / 5MQTT 3.1.1 (2014) is the long-time workhorse; MQTT 5 (2019) is current — richer errors, message properties, shared subscriptions. Check which version a broker/client speaks.

2 · Core concepts

Actors & routing

Brokercentral server: accepts connections, receives every publish, matches topics, fans out to subscribers (Mosquitto, HiveMQ, EMQX)
Clientany device/app connected to the broker — can publish, subscribe, or both
TopicUTF-8 label that routes a stream; created implicitly just by publishing to it
Topic hierarchyslash-delimited levels like a path: plant/area3/well07/pressure; no leading slash by convention

Verbs & data

Publishsend a message (the payload) to a topic
Subscriberegister interest; broker then pushes matching messages to you
Payloadthe body — content-agnostic: 3200, {"psi":3200}, or binary. You pick the encoding
Sessionbroker's memory of a client (subscriptions + undelivered msgs); clean or persistent (see §5)

3 · Topic wildcards (subscribing only)

Wildcards match many topics in one subscription. Never valid when publishing.

The two wildcards

+ single levelmatches exactly one level. plant/+/pressure → plant/well07/pressure but NOT plant/area3/well07/pressure
# multi levelmatches this level & everything below; must be last. plant/# → plant, plant/well07/pressure, plant/area3/well07/temp, …

Mnemonic

+ = single-slot (one level); # = "everything from here down." Subscribing to # alone fans the entire broker to you — great for debugging, dangerous in production.

+ → one level # → all below

4 · QoS — Quality of Service (the delivery tradeoff)

QoS (Quality of Service) = how hard the protocol works to deliver. Higher = stronger guarantee but more round-trips. Chosen per publish and per subscription.

0 — at most once
("fire & forget")		sent once, no ack; may be lost, never duplicated. Cheapest — fine for high-rate readings where one dropped sample is harmless
1 — at least onceacked & resent until confirmed; guaranteed to arrive but may duplicate — make consumers idempotent. Good default
2 — exactly oncefour-step handshake: delivered with no duplicates. Strongest & slowest — use where a duplicate would harm (e.g. "open valve" twice)
Rule of thumb: stream high-rate telemetry at QoS 0/1; reserve QoS 2 for the rare message whose duplication would actually cause harm.

5 · Reliability features for flaky links

State & last-value

Retained messageflag telling the broker to keep the last message on a topic and hand it to any future subscriber immediately — "last known value" for a slow sensor (latest tank level, even if hourly)
Clean vs persistent sessionclean = broker forgets the client on disconnect; persistent = remembers subscriptions & queues missed QoS 1/2 messages, delivering on reconnect. MQTT 5 reframes this as session expiry interval

Liveness

LWT (Last Will & Testament)message registered at connect; broker auto-publishes it if the client drops unexpectedly — e.g. publish offline to plant/well07/status so dashboards know the sensor died vs. the reading being merely stale
Keep-alive (heartbeat)timer set at connect; client must send something (a tiny PINGREQ if idle) within it. Silence past ~1.5× the interval → broker declares the client dead and fires its LWT. Catches silent network drops TCP alone may miss for ages

6 · Why MQTT is tiny + ports

Frugal by design

2-byte min headersmallest control packet is 2 bytes of overhead — vs. hundreds of bytes of text headers in a typical HTTP request
Binary wire formatnot text — compact, fast to parse on an 8-bit microcontroller
One long-lived TCP connreused for everything, vs. setting up/tearing down a connection per message

Ports & transport

1883MQTT over plain TCP — unencrypted; local testing only, never expose publicly
8883MQTT over TLS (Transport Layer Security — the encryption behind HTTPS); the standard for real deployments
8080/8081MQTT over WebSockets so a browser dashboard can subscribe directly

7 · Practical CLI with Mosquitto

Mosquitto = Eclipse's small open-source broker; ships mosquitto_sub (subscribe) and mosquitto_pub (publish). Commands below are typed in a terminal/shell.

# Terminal — subscribe: every topic under "plant/"
mosquitto_sub -h test.example.com -t 'plant/#' -v
#   -h host   -t topic   -v = verbose (prints "topic payload")

# Terminal (another window) — publish a wellhead pressure reading
mosquitto_pub -h test.example.com -t plant/well07/pressure -m 3200
#   -t topic   -m message (the payload)

# Publish a RETAINED "last known value" for late subscribers
mosquitto_pub -h test.example.com -t plant/well07/pressure -m 3200 -r
#   -r = retain

# Subscribe at QoS 1 (at-least-once)
mosquitto_sub -h test.example.com -t 'plant/+/pressure' -q 1

# Over TLS on 8883 with credentials (real deployments)
mosquitto_sub -h broker.example.com -p 8883 --cafile ca.crt \
              -u sensoruser -P 's3cret' -t 'plant/#' -v
The two-terminal demo (one mosquitto_sub -t 'plant/#', one mosquitto_pub) is the fastest way to see pub/sub: publish in one window, watch it appear in the other — no direct connection between them.

Common brokers

8 · MQTT vs HTTP vs AMQP

MQTTpub/sub via a broker; tiny binary frames over TCP → massive fleets of constrained, intermittently-connected devices pushing telemetry
HTTPrequest/response, client → server, text headers → web pages, REST APIs, occasional calls. Heavy for high-frequency device telemetry
AMQP (Advanced Message Queuing Protocol)broker messaging with rich durable queues, exchanges, routing → enterprise back-end buses where complex routing + guaranteed queued delivery beat device frugality
Shorthand: HTTP = "ask and wait", MQTT = "publish and forget (to a topic)", AMQP = "enterprise message plumbing with real queues."

9 · Security — the default is wide open

A bare broker on 1883 has no encryption and often no authentication — anyone who can reach it can subscribe to everything (#) or publish anything. Treat an open broker like an open database. Never expose 1883 to the public internet.
EncryptTLS on 8883 so payloads & credentials aren't in cleartext
Authenticateusername/password at minimum; ideally per-device X.509 client certificates (digital identity certs)
Authorisetopic-level ACLs (access-control lists) so a sensor publishes only to its topic and reads nothing else

10 · Teardown / lifecycle — leaving cleanly

Disconnect paths

Clean disconnectclient sends DISCONNECT, then closes TCP. Broker sees it as intentional and does NOT fire the LWT
Unexpected dropnetwork dies / battery fails / process crashes. Broker notices when keep-alive lapses (~1.5×) and publishes the LWT to announce it's gone
Session cleanupclean session → broker discards subscriptions + queued msgs on disconnect; persistent → keeps them (until session-expiry) so the client resumes on reconnect

Design tip

Pair a retained status topic with an LWT: publish online (retained) on connect, register offline as the will. Subscribers then always see the true current state — even if they connect after the device already died.
