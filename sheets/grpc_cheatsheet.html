gRPC Cheat Sheet

Services calling services: RPC · protobuf · HTTP/2 · started 2026-06-12 · twin of Tools/grpc_cheatsheet.md

0 · The mental model (read once)

gRPC = a framework for calling a function that runs inside another program on another machine, as if it were local. Official expansion: gRPC Remote Procedure Calls — a recursive acronym (the name contains itself); the "g" first hinted at Google, which open-sourced it in 2015 (it grew from Google's internal system Stubby). RPC (Remote Procedure Call) = the older core idea: invoke a procedure that executes elsewhere while your code looks like an ordinary local call; the network plumbing is generated and hidden.
Call a function, don't fetch a URL. REST makes you think in resources/URLs; gRPC flips back to functions: 
REST mindset                  gRPC mindset
───────────                   ────────────
GET /users/42        →        user = client.GetUser(id=42)
(fetch a document)            (call a remote method, get a typed object)
The workflow — first step is the whole point: 1) Write a contract (a .proto file declaring methods + message shapes — the language-independent source of truth). 2) Generate code with the protoc compiler → a client stub (local stand-in whose methods secretly make network calls) + server skeleton, in any target language. 3) Call remote methods like local ones; gRPC marshals args over HTTP/2 and returns a typed result.
Why it matters for GCP-ACE: internal microservices, Kubernetes (K8s) service-to-service traffic, and service meshes (Istio, Anthos Service Mesh) lean on gRPC — it's fast, strongly typed, and polyglot (many languages, one contract).

2 · Pillar one — Protocol Buffers (protobuf)

Protocol Buffers does two jobs: a serialization format (how data becomes wire bytes — binary + schema-driven, so bytes carry compact tag numbers, not field names) and an IDL (the language you define messages/services in). You write a .proto, then run the protoc compiler to generate code (with the gRPC plugin: client stub + server skeleton) in any language.

protobuf vs JSON — why binary + schema wins for internal traffic

Trade-off in one line: protobuf buys smaller + faster + type-checked at the cost of human-readability. Public API → often JSON; chatty internal microservices → protobuf. See formats_cheatsheet.md for the JSON side.

3 · Pillar two — HTTP/2 as the transport

gRPC runs on HTTP/2 (the 2015 revision of the Hypertext Transfer Protocol), not HTTP/1.1 — its streaming and speed depend on these features. TCP (Transmission Control Protocol) = the reliable ordered byte-stream HTTP sits on.

Features gRPC relies on

Multiplexingmany concurrent requests share one TCP connection as independent streams — no head-of-line blocking
Streamseach call is a long-lived bidirectional stream — the basis for gRPC's streaming call types
Binary framingHTTP/2 already splits traffic into binary frames; protobuf bytes ride inside them
HPACKheader compression — repeated metadata (auth tokens) cut down on chatty traffic

Browser caveat → gRPC-Web

Browsers cannot speak raw gRPC (JavaScript doesn't get the needed HTTP/2 frame control). gRPC-Web — a variant + proxy — lets browser apps reach gRPC backends. This is the main reason public, browser-facing APIs still tend to be REST/JSON. See http_cheatsheet.md for HTTP/2 detail.

4 · The four call types (the big differentiator vs REST)

A REST call is one request → one response. Because gRPC sits on HTTP/2 streams it offers four shapes; the streaming ones are what REST/JSON can't do cleanly. In the .proto, the stream keyword on the request and/or response picks the shape.

TypeClientServerUse
Unaryone msgone msgthe classic call — GetUser(id) → User. Like REST.
Server-streamingone msgstreamask once, server pushes many — price updates, paged list as a stream
Client-streamingstreamone msgupload many, one reply — stream sensor readings, get one summary
Bidirectionalstreamstreamboth send freely over one connection — chat, live telemetry, games

5 · A minimal .proto, annotated

syntax = "proto3";              // use proto3, the current dialect

package user.v1;               // namespace; "v1" lets you version the API

// A "service" groups related remote methods (RPCs).
service UserService {
  // unary: one request message in, one response message out
  rpc GetUser (GetUserRequest) returns (User);

  // server-streaming: one request in, a STREAM of Users back
  rpc ListUsers (ListUsersRequest) returns (stream User);
}

// A "message" defines a typed record. Each field has a type,
// a name, and a unique field NUMBER (the tag used on the wire —
// numbers, not names, are what get serialized).
message GetUserRequest {
  int64 id = 1;                 // field 1
}

message User {
  int64  id    = 1;
  string name  = 2;
  string email = 3;
}

message ListUsersRequest {
  int32 page_size = 1;
}

Then compile it on the command line (CLI):

# CLI (your shell): generate Python client+server code from the .proto
python -m grpc_tools.protoc -I. --python_out=. --grpc_python_out=. user.proto
Field numbers (= 1, = 2) are a field's identity on the wire. Rename a field freely, but never reuse or change a number — that breaks compatibility with already-serialized data.

6 · When to use gRPC vs REST/JSON

Reach for gRPC when…

internal microservices talking to each other
low latency + high throughput matter
polyglot stack — one contract, many languages
you need streaming (any of the 3 streaming modes)
strong typing / generated clients are valued

Reach for REST/JSON when…

API is public / third-party consumed
clients are browsers (can't speak raw gRPC)
you want easy human debugging (curl, paste JSON)
simple CRUD over caching-friendly HTTP/1.1
maximum ecosystem familiarity / lowest friction
Rule of thumb: gRPC behind the wall (service↔service inside your cluster), REST at the front door (anything a browser or outside party hits). When a browser must reach a gRPC backend, put gRPC-Web + a proxy in between.

7 · Ecosystem one-liners

Reliability & plumbing

Deadline / timeoutclient sets a deadline ("answer by T") that propagates across hops; if it passes, the call is cancelled everywhere. Prefer deadlines over per-hop timeouts — they don't pile up.
Interceptormiddleware wrapping every call — auth, logging, metrics, retries — without touching business logic. The gRPC analog of HTTP middleware.
Health checksa standard Health service (Check/Watch) that load balancers + K8s probes use to ask "are you ready?"

Building blocks & where it lives

Channelthe client's long-lived connection to a server (manages HTTP/2 + load balancing). Create one, make many calls.
Metadatakey-value pairs sent alongside a call (like HTTP headers) — auth tokens, request IDs.
Where it shows upKubernetes internal traffic, service meshes (Istio, Anthos Service Mesh), GCP APIs, high-throughput data systems.

8 · Teardown — closing down cleanly

Close the channelwhen done, the client closes the channel to free the HTTP/2 connection + sockets. Leaking channels leaks connections — create one per server target and reuse it.
Cancel via deadlineif a call exceeds its deadline, gRPC cancels it and propagates the cancellation server-ward, so no machine keeps doing dead work. The graceful "give up" path.
Graceful server shutdowna server can stop accepting new calls but let in-flight RPCs finish before exiting (e.g. Go's GracefulStop), avoiding abrupt stream breaks.
