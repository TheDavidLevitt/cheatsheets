gcloud Command Grammar Cheat Sheet

The predictable shape of every gcloud command — service group → resource → verb → flags — so you can recognize the right one on sight. Pure CLI syntax, ACE-exam depth.

The grammar rule The big gotcha Service trees list vs describe Common flags Zonal/regional/global Canonical commands Traps Glossary

1 · The grammar rule (read once, recognize forever)

Almost every gcloud command is the same sentence with the words swapped. Learn the slots, not the commands:

gcloud  <SERVICE-GROUP>  <RESOURCE>  <VERB>  [POSITIONAL]  [--FLAGS]
service-group = the product (compute, iam, storage…) resource = the noun, usually plural (instances, buckets, roles…) verb = the action (create, list, describe…)

Worked example, slot by slot

gcloud compute instances create my-instance --zone=us-central1-a
   │      │        │        │        │           │
   │      │        │        │        │           └ --FLAG (scope)
   │      │        │        │        └ POSITIONAL (the name)
   │      │        │        └ VERB    (create)
   │      │        └ RESOURCE (instances — plural!)
   │      └ SERVICE-GROUP (compute)
   └ the binary

Read it as a sentence: "In compute, take the instances collection and create one called my-instance in this zone." Every command parses the same way.

Etymology: it mirrors REST

The grammar isn't arbitrary — it's a REST (Representational State Transfer) URL turned inside out:

RESTgcloud
/compute/.../instancesservice → resource collection (plural)
POST / GET / DELETEthe verb: create / describe / delete
path parameterthe positional name argument

That's why resources are plural: you're naming a collection and then acting on it, exactly like an API endpoint.

The full global structure (release track is optional and goes before the service group): 
gcloud [alpha|beta] <service-group> <resource> <verb> [POSITIONAL] [--FLAGS]
        └─ release track │
           omit = GA (stable)         everything to the right = the sentence above
gcloud compute … stable   gcloud beta compute … pre-release surface   gcloud alpha … earliest, may break. Service groups can nest one level deeper for sub-collections: gcloud compute networks subnets create … (resource = networks subnets).
Discovery trio — when you forget a command, the CLI tells you. Append nothing and read the help: gcloud compute lists its resources; gcloud compute instances lists its verbs; gcloud compute instances create --help lists the flags. The grammar makes the help browsable.

2 · The big gotcha — almost all infrastructure lives under gcloud compute

The single most common wrong answer on the exam: reaching for gcloud networks … or gcloud storage … for networking infrastructure. There is no top-level gcloud networks group. VPC networks, subnets, firewall rules, routers, static IPs, disks, snapshots, load-balancer pieces — all of it hangs off gcloud compute, because Compute Engine is where the underlying network and disk primitives live.

Under gcloud compute (the surprise residents)

compute networksVPC networks — not gcloud networks
compute networks subnetssubnets (a sub-collection of networks)
compute firewall-rulesfirewall rules
compute routersCloud Router (BGP, Cloud NAT)
compute addressesreserved static IP addresses
compute disks / snapshots / imagespersistent disks & their copies
compute instancesVMs
compute instance-templatesVM blueprints for MIGs
compute instance-groups managedmanaged instance groups (MIGs)
compute forwarding-rules / target-pools / url-mapsload-balancer building blocks

What is not under compute

gcloud storagebuckets & objects (the GCS surface): buckets, cp, ls, rm, rsync
gcloud iamroles, service accounts, workload-identity pools
gcloud projectsthe project resource itself + its IAM policy bindings
gcloud dnsCloud DNS managed zones & record sets
gcloud containerGKE clusters & node pools

Mental rule: if it's a low-level plumbing primitive (a wire, an IP, a disk, a VM), look under compute first. If it's a managed product with its own name (Storage, DNS, SQL, Pub/Sub, GKE), it has its own group.

IAM grants are split on purpose: role definitions and identities live under gcloud iam, but granting a role on a project is gcloud projects add-iam-policy-binding — because the policy belongs to the resource (the project), not to IAM. Ask "whose object is this binding attached to?" → projects.

3 · Service trees — service → resource → common verbs

Each group fans out the same way. The verbs in green recur in almost every group (create · list · describe · update · delete); the rest are resource-specific.

gcloud compute — the big one

compute
├─ instances ........ create list describe delete
│                      start stop reset ssh scp
├─ disks ............ create resize snapshot delete
├─ snapshots ........ create list delete
├─ images .......... create list describe
├─ networks ........ create list describe delete
│   └─ subnets ...... create list describe
│                      expand-ip-range
├─ firewall-rules .. create list describe delete
├─ routers ......... create add-bgp-peer
│                      update-bgp-peer describe
├─ addresses ....... create list delete   (static IPs)
├─ instance-templates ......... create list delete
├─ instance-groups managed .... create list
│                      resize set-autoscaling
└─ forwarding-rules / target-pools / url-maps

gcloud iam

iam
├─ roles ........... describe create list
│                     update delete
│   (NO get-permissions verb — describe
│    shows the permission list)
├─ service-accounts  create list describe delete
│   └─ keys ........ create list delete
│                     (key files = anti-pattern)
└─ workload-identity-pools
                     create describe   (+ providers)

Trap: there is no gcloud iam roles get-permissions. To see a role's permissions you describe it.

gcloud projects

projects
├─ create / describe / list / delete
├─ get-iam-policy                (dump bindings)
├─ add-iam-policy-binding        (grant a role)
└─ remove-iam-policy-binding     (revoke a role)

Grants live here because the allow policy is attached to the project resource. Same add/remove-iam-policy-binding verbs appear on buckets, instances, SAs — anywhere a policy can attach.

gcloud container — GKE

container
├─ clusters ........ create describe list delete
│                     get-credentials   (wire up kubectl)
│                     upgrade resize
└─ node-pools ...... create list delete

Memorize: after clusters create, you run clusters get-credentials NAME to point kubectl at it. Without it, kubectl has no context.

gcloud storage — GCS

storage
├─ buckets ......... create describe update
│                     list delete
├─ cp     (copy objects, local ↔ gs://)
├─ ls     (list buckets / objects)
├─ rm     (delete objects)
└─ rsync  (sync a tree to/from a bucket)

Modern replacement for gsutil. Note the object verbs (cp, ls, rm, rsync) are filesystem-style, not objects create.

gcloud sql

sql
├─ instances ....... create describe list
│                     patch delete
├─ databases ....... create list delete
└─ backups ......... create list describe

gcloud pubsub

pubsub
├─ topics .......... create list publish delete
└─ subscriptions ... create list pull delete

gcloud dns

dns
├─ managed-zones ... create list describe delete
└─ record-sets ..... list create
                      transaction start/add/execute

Serverless & config

run
├─ deploy           (deploy a container service)
└─ services ........ list describe delete

functions
└─ deploy           (deploy a Cloud Function)

config
├─ list / set / get-value / unset
└─ configurations .. create activate list

logging
├─ read             (query log entries)
└─ sinks ........... create list describe

4 · list vs describe — a heavily-tested split

list — many, shallow

Enumerates all resources of a type in scope, as a compact table of a few key columns. Takes no name — it's the collection.

gcloud compute instances list
NAME         ZONE           STATUS
my-instance  us-central1-a  RUNNING
web-2        us-central1-b  RUNNING

describe — one, deep

Dumps the full configuration of a single named resource (every field, in YAML). Requires the positional name.

gcloud compute instances describe my-instance \
  --zone=us-central1-a
# → full YAML: disks, networkInterfaces,
#   serviceAccounts, metadata, scheduling…
Recognition rule: question says "see all X" / "which X exist" → list (no name). Question says "the configuration / details / full settings of this X" → describe NAME. If a name appears in the answer, it's almost never list.

5 · Common flags, grouped

Scope — --project / --zone / --region

--project=PROJECT_IDwhich project (defaults to config value)
--zone=us-central1-arequired for zonal resources (VMs, zonal disks)
--region=us-central1required for regional resources (subnets, regional addresses)

Omit the scope flag and gcloud either uses your config default or prompts you to pick. See §6 for which resources are zonal vs regional vs global.

Output — --format

--format=jsonmachine-readable, full structure
--format=yamlsame as describe's default look
--format=table(name,status)pick exact columns
--format="value(name)"bare values, no header — perfect for scripts/$(…)

Filter — --filter

gcloud compute instances list \
  --filter="status=RUNNING"

gcloud compute instances list \
  --filter="name~^web"      # ~ = regex match

The ~ operator is regex match (and !~ is "doesn't match"). Combine with AND / OR. Server-side where supported, so it scales.

Identity & behavior

--impersonate-service-account=SA_EMAILrun this one command as an SA (needs TokenCreator)
--account=alex@example.comuse a specific logged-in account for this command
--quiet / -qskip all confirmation prompts (scripting; dangerous with delete)
Resource-specific flags worth memorizing (these are the ones exams hinge on):
--subnet-mode=customon compute networks create — custom vs auto subnets. This lives on networks create, not subnets.
--range=10.0.0.0/24on compute networks subnets create — the subnet's CIDR
--machine-type=e2-mediumon instances create — VM size
--image-family=debian-12 --image-project=debian-cloudon instances create — boot image (latest in a family)
--boot-disk-size=50GBon instances create — boot disk size
--allow=tcp:22,tcp:443 --source-ranges=0.0.0.0/0on firewall-rules create — what + from where
--size=200GBon disks resize — grow a persistent disk (can't shrink)

6 · Zonal vs regional vs global — which flag the resource demands

Pick the wrong scope flag and the command errors or finds nothing. The scope is a property of the resource type:

--zone Zonal

Lives in one zone; needs --zone.

VM instances
Standard persistent disks (zonal)
Zonal MIGs

--region Regional

Spans zones in a region; needs --region.

Subnets
Regional static addresses
Regional MIGs, Cloud Router, Cloud NAT
Cloud SQL (regional service)

global Global

No location flag — the resource is project-wide.

VPC networks (the network is global; its subnets are regional)
Firewall rules
Images, global addresses, URL maps
The classic question: "A subnet is created with which scope flag?" → --region. "A VM?" → --zone. "The VPC network itself?" → neither, it's global. Remember the nesting: global network ⊃ regional subnet ⊃ zonal VM.

8 · Traps — wrong → right

✗ Wrong (looks plausible)✓ Right
✗ gcloud iam roles get-permissions roles/compute.viewer ✓ gcloud iam roles describe roles/compute.viewer — there is no get-permissions verb; describe shows the permission list.
✗ gcloud networks create MY_VPC ✓ gcloud compute networks create MY_VPC — there is no top-level gcloud networks; networks live under compute.
✗ gcloud compute networks create MY_VPC --type=custom
✗ gcloud compute subnets create … --type=custom		 ✓ gcloud compute networks create MY_VPC --subnet-mode=custom — the flag is --subnet-mode (values auto/custom) and it lives on networks create, not subnets.
✗ gcloud storage buckets update gs://my-disk --capacity=200GB ✓ gcloud compute disks resize MY_DISK --size=200GB --zone=us-central1-a — a disk is a compute resource, not a storage bucket.
✗ gcloud compute subnets create … (no --region) ✓ add --region=us-central1 — subnets are regional; without the flag the command errors or prompts.
✗ gcloud iam add-iam-policy-binding PROJECT_ID … ✓ gcloud projects add-iam-policy-binding PROJECT_ID … — the binding attaches to the project, so the verb lives under projects.
✗ gcloud projects add-iam-policy-binding PROJECT_ID --member="alex@example.com" --role="compute.viewer" ✓ --member="user:alex@example.com" --role="roles/compute.viewer" — member needs a typed prefix (user:), role needs the full roles/ name.
✗ gcloud container clusters create … then expecting kubectl to work ✓ run gcloud container clusters get-credentials MY_CLUSTER --zone=… first — that writes the kubeconfig context.
✗ gcloud storage objects create gs://my-bucket/file.txt ✓ gcloud storage cp file.txt gs://my-bucket/ — object I/O uses filesystem-style verbs (cp/ls/rm/rsync), not objects create.
One rule rescues most of these: when unsure, parse the answer as gcloud SERVICE RESOURCE VERB and ask (a) is the service really where this resource lives? (compute owns the plumbing), (b) is the resource plural?, (c) does that verb actually exist for it? Mismatched slots are how distractor answers are built.

9 · Glossary recap

Service groupthe product namespace right after gcloud (compute, iam, storage, projects…)
Resourcethe noun the command acts on, usually plural (instances, buckets, roles); can nest (networks subnets)
Verbthe action; recurring set = create / list / describe / update / delete (+ add-iam-policy-binding)
Positionalthe unflagged name argument naming a specific resource
Flag--key=value modifier — scope, format, filter, or resource-specific options
Release trackoptional alpha/beta before the service group; omit for GA (stable)
list vs describemany+shallow (no name) vs one+deep (named, full YAML)
Zonal / regional / globalresource location scope → which flag it needs (--zone / --region / none)
MIGManaged Instance Group — compute instance-groups managed
get-credentialsthe container clusters verb that wires kubectl to a GKE cluster
--subnet-modeauto/custom flag on compute networks create (not on subnets)
~ operatorregex match in --filter (e.g. name~^web)
