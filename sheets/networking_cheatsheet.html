0 В· The mental model (read once)

The internet = a network of networks, descended from ARPANET (Advanced Research Projects Agency Network, 1969) вҖ” built so messages could route around damaged links. Core idea, unchanged since: chop data into independent packets, let each find its own way, reassemble at the far end. A protocol = agreed rules for talking (who speaks first, message format, how to say "resend that").
The postal system for packets: 
your message            вҶ’  cut into postcards  вҶ’  each gets an address
(an HTTP request, say)     (packets)              (IP headers)
      в”Ӯ                          в”Ӯ                       в”Ӯ
postcards may take different routes, arrive out of order, or get
lost вҖ” FINE, because the receiver reorders & asks for resends (TCP)
Packet = one postcard (chunk of data + addressing). Router = postal sorting office: reads the destination, forwards one hop closer. Your message crosses ~10вҖ“20 routers it never sees.

The layer stack вҖ” simplified 4-layer TCP/IP model (envelopes inside envelopes)

Application вҖ” HTTP, DNS, SSH, SMTPthe actual message: "GET this page", "what's this name's IP?"
Transport вҖ” TCP, UDPport numbers + (TCP) ordering, resends, the "connection" illusion
Internet вҖ” IPsource + destination IP addresses; routing across networks
Link вҖ” Ethernet, Wi-FiMAC addresses; crossing ONE physical hop

TCP (Transmission Control Protocol) = registered mail: numbered, acknowledged, resent, in order (HTTP, SSH, email). UDP (User Datagram Protocol) = plain postcard: fire-and-forget, low overhead (DNS, video calls, games). The textbook 7-layer OSI model (Open Systems Interconnection) splits the same idea more finely; this 4-layer view is what's actually deployed.

The OSI model вҖ” where the "layer" numbers come from

The OSI model (Open Systems Interconnection вҖ” ISO standard, 1984) is a 7-layer reference model that lost the deployment war to TCP/IP but won the vocabulary war: when cloud docs or network engineers say "layer 7" or "L4", they're counting OSI layers. Universal shorthand across networking, cloud platforms, and certifications.

L7 ApplicationHTTP, DNS, SSH вҖ” the content. "L7 load balancer" routes on URLs/headers (e.g. GCP's HTTP(S) LB)
L6 Presentationencoding/encryption (TLS, roughly) вҖ” rarely named in practice
L5 Sessionconnection management вҖ” rarely named in practice
L4 TransportTCP/UDP + ports. "L4 load balancer" routes on IP:port, blind to content
L3 NetworkIP, routing. "L3 switch" = a switch that also routes
L2 Data linkEthernet/Wi-Fi, MAC addresses. "L2 segment" = one local network
L1 Physicalcables, radio, light pulses

Mapping to the 4-layer view above: L7(+6+5) = Application, L4 = Transport, L3 = Internet, L2+L1 = Link. L5/L6 mostly exist in textbooks вҖ” real protocols fold them into the application. The numbers you'll actually hear: L2, L3, L4, L7. (Office joke: "layer 8" = the user.)

1 В· Addresses вҖ” IP, NAT, ports, MAC

An IP (Internet Protocol) address identifies a machine. IPv4 = Internet Protocol version 4 вҖ” the 1981 standard still carrying most traffic: four numbers 0вҖ“255 (32 bits, ~4.3 billion addresses вҖ” which ran out, hence NAT and IPv6). The successor is version 6 because the number 5 was already taken by an experimental 1980s streaming protocol (ST) that never shipped.

  right вүҲ "which host on it" (split point varies)

Private ranges (never routed publicly; everyone reuses them)

192.168.x.xhome routers вҖ” your laptop is probably 192.168.1.x
10.x.x.xcorporate networks, cloud VPCs
172.16вҖ“31.x.xthe forgotten middle child (Docker uses it)
127.0.0.1localhost вҖ” "this machine talking to itself"; never leaves the box

Docs (incl. this sheet) use 203.0.113.x вҖ” reserved for documentation, can never collide with a real machine.

Famous ports вҖ” the apartment number

IP gets you to the building; the port (0вҖ“65535) is the apartment вҖ” which program gets the packet. One IP can run web + mail + SSH side by side.

22 вҖ” SSH (Secure Shell)remote login вҖ” see the SSH sheet
53 вҖ” DNSthe phone-book lookups (section 2)
80 вҖ” HTTPplain web
443 вҖ” HTTPSencrypted web (the padlock)

example.com:8080 = "this host, port 8080". Browsers silently assume :80 for http, :443 for https.

NAT (Network Address Translation) вҖ” your router owns one public IP from your ISP (Internet Service Provider) and shares it among every device. Outbound, it rewrites private source addresses (192.168.1.7) to the public one and remembers the mapping; replies get translated back. Side effects: inside devices are unreachable from outside by default (a free crude firewall), and "what's my IP?" differs inside vs. outside.
IPv6 (Internet Protocol version 6) = the 128-bit successor (eight colon-separated hex groups, e.g. 2001:db8::1; :: compresses zeros). Enough addresses that every device can be public вҖ” no NAT needed; adoption is gradual.
MAC (Media Access Control) address = hardware serial number of your network card (aa:bb:cc:dd:ee:ff) вҖ” used only for the final local-network hop; never travels across the internet.

2 В· Names вҖ” DNS, the internet's phone book

DNS (Domain Name System) translates names (example.com) into IPs (203.0.113.42). A distributed, hierarchical database вҖ” no single machine holds the whole book.

1. browser вҶ’ OS stub вҶ’ RESOLVER       "what's www.example.com?"
     (resolver = your ISP's, or public: 1.1.1.1 / 8.8.8.8)
2. resolver вҶ’ ROOT server             root: "ask the .com servers" вҶ’
3. resolver вҶ’ .com TLD server         TLD:  "ask example.com's servers" вҶ’
4. resolver вҶ’ AUTHORITATIVE server    "it's 203.0.113.42, cache 1h (TTL)"
5. resolver вҶ’ browser                 answer + cached for next time

Do a DNS lookup yourself CLI

These are terminal commands вҖ” type them at a shell prompt (Terminal on Mac/Linux, where dig is preinstalled; on Windows use nslookup in PowerShell). Same convention everywhere on this sheet: anything in a command table or starting with $ runs in your terminal.

dig example.comfull answer with TTLs (the pro tool; "domain information groper")
dig +short example.comjust the IP
dig MX example.comask for a specific record type
nslookup example.comolder, everywhere (incl. Windows)

TLD (Top-Level Domain) = the last label (.com, .fr). Authoritative = the server that owns the answer. TTL (Time To Live) = cache duration вҖ” why DNS changes "take a while to propagate".

Record types worth knowing

A DNS record = one typed row in a domain's zone file: name вҶ’ typed value, served by the authoritative server. You create/edit these in your registrar or DNS console (Cloudflare, Route 53, Cloud DNSвҖҰ) when you set up a website or email. The type tells clients what kind of answer it is:

A ("address")name вҶ’ IPv4 address. The workhorse: example.com вҶ’ 203.0.113.42. A browser asks for this first.
AAAA ("quad-A")same, but вҶ’ IPv6. Named AAAA because IPv6 addresses are 4Г— the size of an A record's IPv4.
CNAME (Canonical Name)alias: name вҶ’ another name, which is then looked up in turn. E.g. www.example.com вҶ’ example.com, or a GitHub Pages custom domain вҶ’ username.github.io. Lets many names follow one moving target.
MX (Mail eXchange)where email for @example.com should be delivered вҖ” the hostname of the mail server, with a priority number. No MX = no inbound mail.
TXT (text)free-form text attached to the name. Two big uses: proving you own a domain (Google/Microsoft make you paste a token here) and email anti-spoofing policies (SPF lists who may send as you; DKIM publishes a signing key).
NS (Name Server)which servers are authoritative for the domain вҖ” the pointer the TLD hands out in step 3 of the lookup walk above.

3 В· URI vs URL vs URN вҖ” and URL anatomy

URI (Uniform Resource Identifier) = umbrella term: any string identifying a resource. URL (Uniform Resource Locator) = a URI that also says where/how to fetch it вҖ” almost everything you use. URN (Uniform Resource Name) = names without locating, e.g. urn:isbn:0451450523 identifies a book but not where to get it. Every URL is a URI; not every URI is a URL.

4 В· HTTP & HTTPS вҖ” what TLS adds

HTTP (HyperText Transfer Protocol) = the web's request/response conversation: client sends GET /docs/intro.html + headers, server answers with a status code (200 OK, 404 Not Found), headers, body. вҶ’ Deeper coverage (verbs, status codes, REST) in formats_cheatsheet.

HTTPS = HTTP inside a TLS (Transport Layer Security) tunnel вҖ” successor to SSL (Secure Sockets Layer; the old name sticks, e.g. "SSL certificate"). TLS adds: 1. Encryption вҖ” eavesdroppers see gibberish. 2. Authentication вҖ” a certificate proves the server really is example.com, vouched for by a CA (Certificate Authority) your browser already trusts (Let's Encrypt, DigiCertвҖҰ) вҖ” that's the padlock. 3. Integrity вҖ” tampered packets are detected and rejected. 
The TLS handshake in 4 steps
1. ClientHello  вҶ’  "I speak TLS 1.3; here are my cipher suites + a random"
2. ServerHello  вҶҗ  "use this cipher; here's my CERTIFICATE + key material"
3. Verify+keys     client checks cert chain up to a trusted CA; both sides
                   derive the same session keys (key-exchange math lets them
                   agree on a secret even while eavesdropped)
4. Finished     вҮ„  everything from here is encrypted вҖ” HTTP flows inside
Why plain HTTP is fine on localhost but never on Wi-Fi: traffic to 127.0.0.1 never leaves your machine вҖ” no wire to tap. On any shared network (cafГ© Wi-Fi especially), every plain-HTTP byte вҖ” cookies, passwords, pages вҖ” is readable and modifiable by anyone on it. (The cookie is named after the "magic cookie" of 1970s programming lore вҖ” an opaque token a program hands you to give back later, allegedly via fortune cookies.)

5 В· The full journey вҖ” type a URL, press Enter

What happens for https://www.example.com/ with cold caches:

 1. DHCP      (earlier, on joining the network) the router's DHCP server вҖ”
              Dynamic Host Configuration Protocol вҖ” handed your laptop an IP
              (192.168.1.7), the gateway, and a DNS resolver to use
 2. Parse     browser splits the URL: scheme https вҶ’ port 443, host www.example.com
 3. DNS       resolver walk (root вҶ’ TLD вҶ’ authoritative) returns 203.0.113.42
 4. TCP       three-way handshake to 203.0.113.42:443 вҖ” SYN вҶ’ SYN-ACK вҶ’ ACK
              ("can you hear me?" / "yes, can you?" / "yes")
 5. TLS       4-step handshake above: verify certificate, derive session keys
 6. Request   browser sends "GET / HTTP/1.1" + headers (encrypted inside TLS)
 7. NAT/route packets NAT-rewritten at your router, hop across ~10вҖ“20 routers,
              reassembled at the server
 8. Response  server replies "200 OK" + headers + the HTML body
 9. Subfetch  browser parses HTML, discovers CSS/JS/images, fires more requests
              (reusing the open connection where possible)
10. Render    HTML вҶ’ DOM tree, CSS applied, JS runs, pixels appear вҖ”
              typically well under a second for the whole journey

6 В· Diagnostic tools вҖ” first-aid kit CLI

All terminal commands вҖ” Terminal app on Mac/Linux, PowerShell on Windows (names vary slightly there).

ping example.com"is it alive, how far?" вҖ” round-trip ms. Named after sonar's ping: send a pulse, time the echo.
traceroute example.com"which routers does my packet cross?" вҖ” every hop with timing; shows where a path breaks
dig example.com"what does DNS say?" вҖ” records, TTLs, which server answered
curl -v https://example.com"show the whole HTTP conversation" вҖ” DNS, TCP, TLS cert, request + response headers
ifconfig / ip addr"what's my IP & interfaces?" (macOS/BSD vs. modern Linux)
netstat -an / lsof -i"which ports are open / which program owns that connection?" (lsof -i :443)
Triage order when "the internet is broken": ping 1.1.1.1 (raw connectivity, skipping DNS) вҶ’ ping example.com (DNS working?) вҶ’ traceroute (where does it die?) вҶ’ curl -v (server's fault or mine?).

port still "busy" after the kill?normal: the OS holds the closed socket in TIME_WAIT ~30вҖ“60 s to absorb late packets вҖ” wait it out, or restart on another port
