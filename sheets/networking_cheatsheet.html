Networking Cheat Sheet — how the internet works

Layers · IP · DNS · URLs · HTTPS · the full journey · started 2026-06-06 · twin of Tools/networking_cheatsheet.md

0 · The mental model (read once)

The internet = a network of networks, descended from ARPANET (Advanced Research Projects Agency Network, 1969) — built so messages could route around damaged links. Core idea, unchanged since: chop data into independent packets, let each find its own way, reassemble at the far end. A protocol = agreed rules for talking (who speaks first, message format, how to say "resend that").
The postal system for packets: 
your message            →  cut into postcards  →  each gets an address
(an HTTP request, say)     (packets)              (IP headers)
      │                          │                       │
postcards may take different routes, arrive out of order, or get
lost — FINE, because the receiver reorders & asks for resends (TCP)
Packet = one postcard (chunk of data + addressing). Router = postal sorting office: reads the destination, forwards one hop closer. Your message crosses ~10–20 routers it never sees.

The layer stack — simplified 4-layer TCP/IP model (envelopes inside envelopes)

Application — HTTP, DNS, SSH, SMTPthe actual message: "GET this page", "what's this name's IP?"
Transport — TCP, UDPport numbers + (TCP) ordering, resends, the "connection" illusion
Internet — IPsource + destination IP addresses; routing across networks
Link — Ethernet, Wi-FiMAC addresses; crossing ONE physical hop

TCP (Transmission Control Protocol) = registered mail: numbered, acknowledged, resent, in order (HTTP, SSH, email). UDP (User Datagram Protocol) = plain postcard: fire-and-forget, low overhead (DNS, video calls, games). The textbook 7-layer OSI model (Open Systems Interconnection) splits the same idea more finely; this 4-layer view is what's actually deployed.

"Layer 7", "L4" — the OSI numbering everyone actually uses

The OSI model (Open Systems Interconnection — ISO standard, 1984) is a 7-layer reference model that lost the deployment war to TCP/IP but won the vocabulary war: its layer numbers are universal shorthand across networking, cloud docs, and certifications.

L7 ApplicationHTTP, DNS, SSH — the content. "L7 load balancer" routes on URLs/headers (e.g. GCP's HTTP(S) LB)
L6 Presentationencoding/encryption (TLS, roughly) — rarely named in practice
L5 Sessionconnection management — rarely named in practice
L4 TransportTCP/UDP + ports. "L4 load balancer" routes on IP:port, blind to content
L3 NetworkIP, routing. "L3 switch" = a switch that also routes
L2 Data linkEthernet/Wi-Fi, MAC addresses. "L2 segment" = one local network
L1 Physicalcables, radio, light pulses

Mapping to the 4-layer view above: L7(+6+5) = Application, L4 = Transport, L3 = Internet, L2+L1 = Link. L5/L6 mostly exist in textbooks — real protocols fold them into the application. The numbers you'll actually hear: L2, L3, L4, L7. (Office joke: "layer 8" = the user.)

1 · Addresses — IP, NAT, ports, MAC

An IP (Internet Protocol) address identifies a machine. IPv4 = four numbers 0–255 (32 bits, ~4.3 billion addresses — which ran out, hence NAT and IPv6).

 203.0.113.42
 └┬┘ └┬┘└┬┘└┬┘
  each part = one byte (0–255); left ≈ "which network",
  right ≈ "which host on it" (split point varies)

Private ranges (never routed publicly; everyone reuses them)

192.168.x.xhome routers — your laptop is probably 192.168.1.x
10.x.x.xcorporate networks, cloud VPCs
172.16–31.x.xthe forgotten middle child (Docker uses it)
127.0.0.1localhost — "this machine talking to itself"; never leaves the box

Docs (incl. this sheet) use 203.0.113.x — reserved for documentation, can never collide with a real machine.

Famous ports — the apartment number

IP gets you to the building; the port (0–65535) is the apartment — which program gets the packet. One IP can run web + mail + SSH side by side.

22 — SSH (Secure Shell)remote login — see the SSH sheet
53 — DNSthe phone-book lookups (section 2)
80 — HTTPplain web
443 — HTTPSencrypted web (the padlock)

example.com:8080 = "this host, port 8080". Browsers silently assume :80 for http, :443 for https.

NAT (Network Address Translation) — your router owns one public IP from your ISP (Internet Service Provider) and shares it among every device. Outbound, it rewrites private source addresses (192.168.1.7) to the public one and remembers the mapping; replies get translated back. Side effects: inside devices are unreachable from outside by default (a free crude firewall), and "what's my IP?" differs inside vs. outside.
IPv6 = the 128-bit successor (eight colon-separated hex groups, e.g. 2001:db8::1; :: compresses zeros). Enough addresses that every device can be public — no NAT needed; adoption is gradual.
MAC (Media Access Control) address = hardware serial number of your network card (aa:bb:cc:dd:ee:ff) — used only for the final local-network hop; never travels across the internet.

2 · Names — DNS, the internet's phone book

DNS (Domain Name System) translates names (example.com) into IPs (203.0.113.42). A distributed, hierarchical database — no single machine holds the whole book.

1. browser → OS stub → RESOLVER       "what's www.example.com?"
     (resolver = your ISP's, or public: 1.1.1.1 / 8.8.8.8)
2. resolver → ROOT server             root: "ask the .com servers" →
3. resolver → .com TLD server         TLD:  "ask example.com's servers" →
4. resolver → AUTHORITATIVE server    "it's 203.0.113.42, cache 1h (TTL)"
5. resolver → browser                 answer + cached for next time

Look it up yourself

dig example.comfull answer with TTLs (the pro tool)
dig +short example.comjust the IP
dig MX example.comask for a specific record type
nslookup example.comolder, everywhere (incl. Windows)

TLD (Top-Level Domain) = the last label (.com, .fr). Authoritative = the server that owns the answer. TTL (Time To Live) = cache duration — why DNS changes "take a while to propagate".

Record types worth knowing

Aname → IPv4 address
AAAAname → IPv6 ("quad-A" — 4× the bits of an A)
CNAMEalias: name → another name (www → example.com)
MXMail eXchange — where @example.com email goes
TXTarbitrary text: ownership proofs, email anti-spoofing (SPF/DKIM)

3 · URI vs URL vs URN — and URL anatomy

URI (Uniform Resource Identifier) = umbrella term: any string identifying a resource. URL (Uniform Resource Locator) = a URI that also says where/how to fetch it — almost everything you use. URN (Uniform Resource Name) = names without locating, e.g. urn:isbn:0451450523 identifies a book but not where to get it. Every URL is a URI; not every URI is a URL. 
https://alice@www.example.com:443/docs/intro.html?lang=en&v=2#section-3
└─┬─┘   └─┬─┘ └──────┬───────┘└┬┘└───────┬──────┘└────┬────┘ └───┬───┘
scheme   user       host      port      path        query     fragment

URL parts decoded

schemeprotocol: https, http, ssh, mailto, file
user@identity for the connection (rare in browsers; common in SSH/Git URLs)
hostdomain name (DNS-resolved) or literal IP
:portapartment number; omitted = scheme default (80/443)
/pathwhich resource on the server (looks like a file path; usually isn't)
?querykey=value pairs joined by & — parameters for the server
#fragmentjump-to anchor within the page — never sent to the server at all

4 · HTTP & HTTPS — what TLS adds

HTTP (HyperText Transfer Protocol) = the web's request/response conversation: client sends GET /docs/intro.html + headers, server answers with a status code (200 OK, 404 Not Found), headers, body. → Deeper coverage (verbs, status codes, REST) in formats_cheatsheet.

HTTPS = HTTP inside a TLS (Transport Layer Security) tunnel — successor to SSL (Secure Sockets Layer; the old name sticks, e.g. "SSL certificate"). TLS adds: 1. Encryption — eavesdroppers see gibberish. 2. Authentication — a certificate proves the server really is example.com, vouched for by a CA (Certificate Authority) your browser already trusts (Let's Encrypt, DigiCert…) — that's the padlock. 3. Integrity — tampered packets are detected and rejected. 
The TLS handshake in 4 steps
1. ClientHello  →  "I speak TLS 1.3; here are my cipher suites + a random"
2. ServerHello  ←  "use this cipher; here's my CERTIFICATE + key material"
3. Verify+keys     client checks cert chain up to a trusted CA; both sides
                   derive the same session keys (key-exchange math lets them
                   agree on a secret even while eavesdropped)
4. Finished     ⇄  everything from here is encrypted — HTTP flows inside
Why plain HTTP is fine on localhost but never on Wi-Fi: traffic to 127.0.0.1 never leaves your machine — no wire to tap. On any shared network (café Wi-Fi especially), every plain-HTTP byte — cookies, passwords, pages — is readable and modifiable by anyone on it. (The cookie is named after the "magic cookie" of 1970s programming lore — an opaque token a program hands you to give back later, allegedly via fortune cookies.)

5 · The full journey — type a URL, press Enter

What happens for https://www.example.com/ with cold caches:

 1. DHCP      (earlier, on joining the network) the router's DHCP server —
              Dynamic Host Configuration Protocol — handed your laptop an IP
              (192.168.1.7), the gateway, and a DNS resolver to use
 2. Parse     browser splits the URL: scheme https → port 443, host www.example.com
 3. DNS       resolver walk (root → TLD → authoritative) returns 203.0.113.42
 4. TCP       three-way handshake to 203.0.113.42:443 — SYN → SYN-ACK → ACK
              ("can you hear me?" / "yes, can you?" / "yes")
 5. TLS       4-step handshake above: verify certificate, derive session keys
 6. Request   browser sends "GET / HTTP/1.1" + headers (encrypted inside TLS)
 7. NAT/route packets NAT-rewritten at your router, hop across ~10–20 routers,
              reassembled at the server
 8. Response  server replies "200 OK" + headers + the HTML body
 9. Subfetch  browser parses HTML, discovers CSS/JS/images, fires more requests
              (reusing the open connection where possible)
10. Render    HTML → DOM tree, CSS applied, JS runs, pixels appear —
              typically well under a second for the whole journey

6 · Diagnostic tools — first-aid kit

ping example.com"is it alive, how far?" — round-trip ms. Named after sonar's ping: send a pulse, time the echo.
traceroute example.com"which routers does my packet cross?" — every hop with timing; shows where a path breaks
dig example.com"what does DNS say?" — records, TTLs, which server answered
curl -v https://example.com"show the whole HTTP conversation" — DNS, TCP, TLS cert, request + response headers
ifconfig / ip addr"what's my IP & interfaces?" (macOS/BSD vs. modern Linux)
netstat -an / lsof -i"which ports are open / which program owns that connection?" (lsof -i :443)
Triage order when "the internet is broken": ping 1.1.1.1 (raw connectivity, skipping DNS) → ping example.com (DNS working?) → traceroute (where does it die?) → curl -v (server's fault or mine?).

7 · Teardown / closing connections

lsof -i :8888 → kill <PID>find what's holding a port, then kill it (escalate to kill -9 only if it ignores the polite one)
port still "busy" after the kill?normal: the OS holds the closed socket in TIME_WAIT ~30–60 s to absorb late packets — wait it out, or restart on another port
flush the DNS cachemacOS: sudo dscacheutil -flushcache; sudo killall -HUP mDNSResponder · Windows: ipconfig /flushdns — forces fresh lookups after a DNS change
renew a DHCP leasemacOS: sudo ipconfig set en0 DHCP (or System Settings ▸ Network ▸ TCP/IP ▸ Renew DHCP Lease) · Windows: ipconfig /release then ipconfig /renew
disconnect VPN / Wi-Fi cleanlyuse the VPN client's own Disconnect (restores your routes and DNS); toggle Wi-Fi off before switching networks to drop the stale lease
