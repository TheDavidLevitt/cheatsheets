SSH Cheat Sheet

Remote shells: keys · config · scp · tunnels · started 2026-06-06 · twin of Tools/ssh_cheatsheet.md

0 · The mental model (read once)

SSH (Secure Shell) = a protocol for an encrypted connection to another machine: a remote terminal, plus file transfer (scp/sftp/rsync) and tunnels, all over the same pipe. Created in 1995 by Tatu Ylönen after a password-sniffing attack on his university network — it replaced telnet and rlogin, which sent passwords in plain text. Standard implementation: OpenSSH, preinstalled on macOS/Linux.
You get a shell on another machine, through an encrypted pipe: 
  your laptop                         remote server
  ┌──────────┐   encrypted pipe      ┌──────────────┐
  │ terminal │ ═══════════════════► │ shell (bash) │
  │  (ssh)   │ ◄═══════════════════ │  runs THERE  │
  └──────────┘     port 22          └──────────────┘
Every command in an ssh session executes on the remote machine. Default port 22 (telnet was 23 — SSH took the slot next door). Three uses of the pipe: terminal, file transfer, tunnels.

1 · Key-based auth — the core skill

Padlock vs. key. A keypair = two linked files. The public key (.pub) is a padlock — safe to copy onto any server; it can only lock (verify), never unlock. The private key is the key that opens it — it never leaves your machine. Login = the server challenges you to prove you hold the private key matching a padlock in its list; no password crosses the wire.

Step 1 — generate (once per machine)

ssh-keygen -t ed25519 -C "laptop-2026"
-t ed25519key type — modern elliptic-curve algorithm; shorter, faster, ≥ as secure as RSA
-C "label"Comment embedded in the public key so you can tell keys apart later (machine name or email)
~/.ssh/id_ed25519private key — never leaves this machine
~/.ssh/id_ed25519.pubpublic key — the shareable padlock (one line of text)

Passphrase: set one — it encrypts the private key on disk (stolen laptop ≠ stolen identity). The agent (§4) means you type it once per session. Ed25519 vs RSA one-liner: use ed25519 unless a legacy server forces -t rsa -b 4096 (RSA = Rivest–Shamir–Adleman, the older default).

Step 2 & 3 — install, then log in

ssh-copy-id alice@server.example.com   # once, with password
ssh alice@server.example.com           # forever after, no password

ssh-copy-id logs in once with your password and appends your public key to ~/.ssh/authorized_keys on the server — the server's list of accepted padlocks (one per line) — fixing permissions as it goes. That's all "installing a key" means.

Never share, email, paste, or commit the private key. Only the .pub ever leaves your machine. Leaked? Generate a new pair and purge the old public key from every server's authorized_keys.
Etymology — why "ed25519"? Decode it as Ed + 25519. Ed = EdDSA (Edwards-curve Digital Signature Algorithm) — "Edwards" honors mathematician Harold Edwards, who described this curve family in 2007. 25519 names the underlying prime field: the curve lives over the prime 2²⁵⁵ − 19 — the number is literally in the name. The same prime powers Curve25519/X25519 (key exchange, Daniel J. Bernstein, 2005); Ed25519 (Bernstein et al., 2011) is its signature sibling. So: "Edwards-curve signatures over the 2²⁵⁵−19 prime."

2 · Everyday commands

Connect & run

ssh alice@server.example.comremote shell as user alice
ssh -p 2222 alice@server.example.comnon-default port (default 22)
ssh alice@server.example.com 'df -h'run one remote command, print output, exit

Move files

scp report.pdf alice@server.example.com:/tmp/secure copy to server (: before remote path)
scp alice@server.example.com:/var/log/app.log .copy from server to here
scp -r ./site alice@server.example.com:~/www/directory, recursively
sftp alice@server.example.cominteractive session (SFTP = SSH File Transfer Protocol): ls/cd, get, put, exit
rsync -avz ./project/ alice@server.example.com:~/project/sync over SSH — transfers only what changed (-a archive, -v verbose, -z compress)
Mnemonic: in scp/rsync, a path with host: in front is remote, without is local — source first, destination second, same as cp.

3 · ~/.ssh/config — stop typing incantations

A per-user file that gives connections names. Every SSH-speaking tool (ssh, scp, rsync, git, VS Code Remote) understands the alias.

# ~/.ssh/config
Host myserver                       # the alias you'll type
    HostName 203.0.113.10           # real address (IP or DNS name)
    User alice                      # login user (drop alice@)
    Port 2222                       # non-default port, if any
    IdentityFile ~/.ssh/id_ed25519  # which private key to offer

Now ssh myserver replaces ssh -p 2222 -i ~/.ssh/id_ed25519 alice@203.0.113.10 — and scp file myserver:/tmp/ works too. Stack one Host block per server; Host * at the bottom sets defaults.

4 · ssh-agent — type the passphrase once

A small background process holding your decrypted private keys in memory for the session: unlock once, every later ssh/scp/git asks the agent instead of you.

ssh-addload default key into the agent (passphrase prompt, once)
ssh-add -llist keys the agent currently holds
ssh-add --apple-use-keychainmacOS: store passphrase in Keychain so the key auto-loads after reboot (pair with UseKeychain yes + AddKeysToAgent yes in config)

5 · Tunnels / port forwarding

The encrypted pipe can carry any port. Two directions:

Local forward -L — the one you'll use

A port on your machine reaches a service near the server. Classic: Jupyter on a remote box, listening only on its own localhost:8888.

ssh -L 8888:localhost:8888 alice@server.example.com
browser → localhost:8888 ══(encrypted)══►
  server → its localhost:8888 (jupyter)

Now http://localhost:8888 is the remote notebook. Remote PostgreSQL: -L 5433:localhost:5432, point your client at 5433.

Remote forward -R — the mirror image

A port on the server reaches a service on your machine — e.g. show your local dev server to someone on the remote side.

ssh -R 9000:localhost:3000 alice@server.example.com
someone on server → server:9000 ══(encrypted)══►
  you → your localhost:3000

-N = no remote command — tunnel only, no shell opens. Handy with -L/-R.

6 · SSH for GitHub

GitHub uses SSH keys for git push/pull over git@github.com: URLs — same keypair mechanics; paste your .pub into GitHub → Settings → SSH keys. Test the connection:

ssh -T git@github.com

-T disables the terminal request (GitHub gives no shell); success = "Hi username! You've successfully authenticated…". Full setup and SSH-vs-HTTPS remotes: GitHub sheet.

7 · Troubleshooting

Permission denied (publickey)server only accepts keys and didn't accept yours: wrong user, key not in authorized_keys, or wrong key offered — debug with -v, pin the key with IdentityFile
UNPROTECTED PRIVATE KEY FILEpermissions too open; fix: chmod 600 ~/.ssh/id_ed25519 (and chmod 700 ~/.ssh)
REMOTE HOST IDENTIFICATION HAS CHANGED!server's host key ≠ the one cached in ~/.ssh/known_hosts. Either the server was rebuilt, or a MITM (man-in-the-middle — an attacker relaying traffic between you and the server). Verify out-of-band before ssh-keygen -R hostname
ssh -v alice@server.example.comverbose: shows the whole negotiation — keys offered, auth methods tried. -vv/-vvv for more
Host-key fingerprint & TOFU. On first connect, SSH shows the server's key fingerprint and asks "are you sure?". Yes = cached in known_hosts; every later connect silently verifies it's still the same machine. This first-connect leap of faith = TOFU — trust on first use. The careful version: compare the fingerprint with one from your hosting console before accepting.

8 · Ending a session — teardown

Normal exit

exit / logout / Ctrl-Dleave the remote shell (Ctrl-D = end-of-file)
exit the carrying shellcloses a foreground tunnel; Ctrl-C if running with -N
ps aux | grep "ssh -" → kill <PID>find/kill a background (-f) tunnel

Hung session & cleanup

Enter then ~.kill a frozen session — client-side escape, works when the remote is dead
Enter then ~?list all escape sequences
ssh-add -Dforget loaded keys (shared machine)
ssh-keygen -R hostdrop a server from your known_hosts
edit server's authorized_keysrevoke your own access: delete your line
The ~ escapes are read by your local ssh client, not the remote shell — that's why they work even when the far side is unresponsive. They're only recognized at the start of a line, hence Enter first.

9 · Glossary recap

SSHSecure Shell — encrypted terminal, file transfer, tunnels (Ylönen, 1995; replaced telnet/rlogin)
OpenSSHthe standard implementation (OpenBSD project)
Keypairpublic key (shareable padlock) + private key (never leaves your machine)
Ed25519 / RSAkey algorithms; prefer Ed25519, RSA-4096 for legacy
EdDSAEdwards-curve Digital Signature Algorithm — the family Ed25519 belongs to (over the 2²⁵⁵−19 prime)
SCP / SFTPSecure Copy (one-shot, cp-like) / SSH File Transfer Protocol (interactive browse-and-transfer) — both ride the SSH pipe
rsyncdelta-syncing file copier over SSH — transfers only what changed (deploys, backups)
telnet / rloginthe pre-SSH plaintext remote-login tools SSH replaced (passwords sent in the clear)
authorized_keysserver-side list of accepted public keys
known_hostsclient-side cache of server host keys (the TOFU record)
ssh-agentsession daemon holding decrypted keys — passphrase once
Port forwarding-L (local→remote) and -R (remote→local) tunnels
MITMman-in-the-middle — attacker relaying traffic between you and the server
TOFUtrust on first use — accept a fingerprint once, verify it forever after
