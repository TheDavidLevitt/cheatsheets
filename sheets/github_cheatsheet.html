0 · Git vs. GitHub + the mental model

Git = the version-control tool on your computer (see git cheat sheet). GitHub = a website/hosting service (owned by Microsoft) that stores Git repos in the cloud for sharing. GitLab & Bitbucket are competitors. Git works offline; GitHub is just a copy others can reach. Remote = a named bookmark to a hosted copy; the default is called origin.
local ⇄ remote 
  GitHub (remote "origin")          Your machine (local)
  ┌────────────────────┐           ┌─────────────────────────┐
  │  remote repository │           │ local repo │ working tree│
  └────────────────────┘           └─────────────────────────┘
          ▲   │                          ▲          │
   push   │   │  fetch / pull            │          │ add / commit
          │   ▼                          │          ▼  edits
push = upload your commits fetch = download, don't merge pull = fetch + merge clone = fresh local copy
Memory hook: push work up to share; pull others' work down to catch up. fetch is the cautious cousin of pull.
Grammar: git <verb> [remote] [branch] [--flags] — e.g. git push origin main = "push to remote origin, branch main."
SSH vs. HTTPS — two ways to authenticate to GitHub. HTTPS (https://github.com/user/repo.git) uses a PAT (Personal Access Token — a random string replacing your password; plain passwords disabled 2021). SSH (git@github.com:user/repo.git) uses an SSH (Secure Shell) key pair you generate once — no prompts after setup, preferred long-term.

1 · git clone — copy a remote to your machine

Makes a full local copy (files + entire history) and auto-sets a remote origin pointing back.

Clone

git clone <url>copy into a folder named after the repo
git clone <url> myfolderclone into a folder you name
git clone --depth 1 <url>shallow: latest snapshot only (faster)
git clone https://github.com/yourname/yourrepo.git

git remote — manage bookmarks

git remote -vlist remotes + URLs (-v = verbose)
git remote add origin <url>link a local repo to GitHub
git remote set-url origin <url>change URL (e.g. HTTPS → SSH)
git remote remove originunlink a remote

First push of a brand-new local repo:

git remote add origin git@github.com:yourname/yourrepo.git
git push -u origin main

2 · git push — upload your commits

Push

git pushpush current branch to its upstream
git push -u origin mainpush + set upstream (-u); do once per new branch
git push origin <branch>push a specific branch
git push --force-with-leasesafe force: refuses if remote moved (prefer over --force)

What's "upstream"?

Upstream = the specific remote branch your local branch tracks. -u wires it up so bare push/pull know where to go, and git status can say "ahead 2 / behind 1".

3 · git fetch vs. git pull — getting others' work

The pair most worth understanding precisely. fetch downloads but never touches your files; pull downloads and integrates.

Commands

git fetchdownload into origin/* — files unchanged, nothing merged. Safe anytime.
git fetch --prunealso drop refs to branches deleted on GitHub
git pullfetch then merge into current branch
git pull --rebasefetch, then replay your commits on top → linear history

Good habit + rebase note

Habit: git fetch first, inspect with git log origin/main, then merge — avoids surprise merges from a blind pull.

Rebase = move your commits to start from a different base, replaying them one by one → tidy straight-line history vs. merge's graph. Rule: rebase local/unshared work; merge anything already public.

4 · Branch + Pull Request flow

Branch = a movable pointer to a line of development — work on a feature without disturbing main. Pull Request (PR) = GitHub's web feature to propose merging one branch into another (diff + review + checks bundled). GitLab calls it a Merge Request. You're requesting the maintainer pull your branch in. CI (Continuous Integration) = automated tests/builds run on each push/PR (e.g. GitHub Actions). 
git switch -c feature-login      # 1. create + switch to a new branch (-c = create)
#   ... edit files, git add, git commit ...
git push -u origin feature-login # 2. publish the branch to GitHub
#   3. on github.com: "Compare & pull request", describe, open the PR
#   4. teammates review; CI runs; you push fixes (they auto-update the PR)
#   5. click "Merge pull request" once approved
git switch main                  # 6. back to main locally
git pull                         # 7. bring the merged work down
git branch -d feature-login      # 8. delete the merged local branch (-d = delete)

Branch commands

git branchlist local branches (* = current)
git switch <branch>move to existing branch (older: git checkout)
git switch -c <branch>create + switch
git merge <branch>merge another branch into current
git branch -d <branch>delete a fully-merged branch

5 · Fork vs. clone · the gh CLI

Fork vs. clone (open source)

Clone = local copy of a repo you can usually write to. Fork = a GitHub-side copy of someone else's repo under your account, so you can push freely then PR back.

Flow: Fork on GitHub → clone your fork → branch → push → PR to upstream. Keep current with a second remote:

git remote add upstream <original-url>
git fetch upstream
git merge upstream/main

gh — GitHub's CLI (optional)

Official command-line tool for GitHub-web actions (separate from git).

gh auth loginauthenticate the CLI (one-time)
gh repo clone user/repoclone by shorthand
gh repo createmake a new GitHub repo
gh pr createopen a PR from the shell
gh pr statusPRs relevant to you

6 · Teardown — disconnecting & cleaning up

Repo-side

git remote remove originunlink local repo from GitHub — nothing deleted on either side
git push origin --delete mybranchdelete a branch on GitHub (local copy survives; git branch -d for that)
git fetch --prunedrop stale local origin/* refs to deleted branches
gh repo delete user/repodelete a whole repo from the CLI (confirms; needs delete_repo scope). Web: Settings → Danger Zone

Site & credentials

Settings → Pages → Unpublish sitetake a Pages site offline without deleting the repo (deleting the repo also kills the site)
Settings → Developer settings → Tokens / SSH keysrevoke a PAT or remove an SSH key — that credential's access dies instantly
gh auth logoutsign the gh CLI out (removes its stored token; git unaffected)
