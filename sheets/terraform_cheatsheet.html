Terraform Cheat Sheet

Infrastructure as Code on GCP: write · init · plan · apply · started 2026-06-06 · twin of Tools/terraform_cheatsheet.md

0 · The mental model (read once)

Terraform = the Infrastructure-as-Code tool HashiCorp released in 2014. You declare the cloud resources you want in text files; Terraform talks to the cloud provider's API (Application Programming Interface) to make reality match the files. It is declarative — you describe the end state ("one bucket, EU region"), not the steps to get there. Terraform computes the steps.
IaC (Infrastructure as Code) = managing infrastructure (VMs, buckets, networks, IAM — Identity and Access Management) through machine-readable definition files instead of manual console operations. HCL (HashiCorp Configuration Language) = the language Terraform files (.tf) are written in. Looks like JSON's friendlier cousin: blocks, labels, key = value pairs.
Why code instead of clicks? Clicking in the Cloud Console creates infrastructure, but the recipe lives only in your memory. Analogy from the Vim sheet: the console is Vim's insert mode — live edits with no saved file. Terraform is the saved file: the .tf config is the durable artifact, and the cloud is just its current render. Lose the project? terraform apply re-renders it. (Even better: with the Git sheet, the config is a committed file — diffable, reviewable, revertible.)

What IaC buys you over console clicking

Reproduciblesame files → same infra, in dev, staging, prod, or after a disaster
Reviewableinfra changes go through pull requests like any code
Diffableterraform plan shows exactly what would change before it changes
Self-documentingthe config is the up-to-date inventory; no stale wiki page
The core loop: 
   write .tf files ──► terraform init ──► terraform plan ──► terraform apply
   (declare intent)    (download the      (preview diff:     (make the cloud
        ▲               GCP provider,      + ~ - symbols)     match the files)
        │               set up backend)         │                  │
        └───────── edit & re-plan ◄─────────────┘                  │
                                                                   ▼
                                                        terraform destroy
                                                        (tear it all down)
init — once per project (and after adding providers/modules/backends). plan → apply — every change cycle; plan is free and safe, apply mutates. destroy — the whole point of practice projects: clean teardown, zero forgotten-VM bills.
Memory hook: "I plan applies" — init, plan, apply. Init once, plan always, apply deliberately.
Command grammar: terraform <verb> [options], run from the directory containing your .tf files (the "root module"). Terraform reads all .tf files in that directory and merges them — file names are for humans (main.tf, variables.tf, outputs.tf is the convention, not a rule).

1 · HCL anatomy — the five block types you need

HCL is built from blocks: TYPE "label" "label" { body }.

provider — which cloud, which credentials

provider "google" {
  project = "my-gcp-project-id"
  region  = "europe-west1"
}

A provider is the plugin that translates Terraform's plan into real API calls. init downloads it from the Terraform Registry. The google provider covers GCP; there are providers for AWS, Azure, GitHub, even Spotify.

resource — the thing you want to exist (the key grammar)

resource "google_storage_bucket" "learning" {
  name     = "demo-learning-bucket"
  location = "EU"
}

Read the header as resource "TYPE" "LOCAL_NAME":

"google_storage_bucket"the resource type, fixed vocabulary defined by the provider (always <provider>_<product>). You look these up in the registry docs; you can't invent them.
"learning"the local name, your nickname for this block, used only inside your config to refer to it: google_storage_bucket.learning.name. It is not the bucket's name in GCP — that's the name argument inside.

variable — parameterize, reference with var.

variable "project_id" {
  type        = string
  description = "GCP project to deploy into"
}
# used elsewhere as:  project = var.project_id

Values come from (in priority order): -var flags > *.tfvars files > TF_VAR_project_id environment variables > the default in the block.

output — print useful values after apply

output "bucket_url" {
  value = google_storage_bucket.learning.url
}

Shown at the end of apply, retrievable any time with terraform output. Outputs are how modules hand values to their callers.

data — read existing infrastructure (look, don't touch)

data "google_project" "current" {
  project_id = var.project_id
}
# read-only reference:  data.google_project.current.number

resource manages a thing; data only queries a thing that already exists (often created by someone else or by the console). Note the data. prefix when referencing.

Memory hook: type = the species (set by the provider), local name = the pet name (set by you). GCP sees the species and the name argument; only your config knows the pet name.

2 · State — the sacred file

terraform.tfstate is Terraform's memory: a JSON map from your config blocks to the real resource IDs in GCP. plan works by comparing three things: config (desired) vs. state (last known) vs. reality (refreshed).

Why it's sacred:
  • Delete it and Terraform forgets it ever created anything — next apply tries to create duplicates, and destroy can't clean up.
  • It contains resource attributes in plaintext — potentially secrets (generated passwords, keys). Treat it like a credentials file.
  • Never commit terraform.tfstate to Git. Add it to .gitignore along with .terraform/ (the provider download cache).

Remote state in a GCS bucket (the GCP-native answer)

Local state breaks down with teammates or multiple machines. Store it in GCS (Google Cloud Storage) instead, via a backend block:

terraform {
  backend "gcs" {
    bucket = "my-tfstate-bucket"   # create this one manually (or console) —
    prefix = "tf/state"      # chicken-and-egg: state can't store itself
  }
}

Re-run terraform init after adding it; Terraform offers to migrate existing state. GCS backends also give you state locking (two applies can't stomp each other) and versioning if enabled on the bucket.

3 · Command table

terraform initdownload providers/modules, configure backend. Run first, and after changing any of those.
terraform validatesyntax + internal consistency check (no API calls)
terraform fmtauto-format .tf files to canonical style (like gofmt)
terraform plandry-run diff: what would apply do? Always read it.
terraform plan -out=tf.plansave the plan so apply tf.plan executes exactly what you reviewed
terraform applyplan, ask for confirmation (yes), then execute
terraform apply -auto-approveskip the prompt — for CI (Continuous Integration), not for fingers
terraform destroydelete everything this config manages. The free-tier student's best friend.
terraform state listlist every resource currently tracked in state
terraform showhuman-readable dump of state (or of a saved plan)
terraform outputre-print output values from state
terraform import <addr> <id>adopt an existing (console-made) resource into state — write the matching block first
Memory hook for the daily four: i v p a — "init, validate, plan, apply" — alphabetical flow except init leads.

4 · Minimal complete GCP example (annotated)

One file, main.tf — a provider, a bucket, an output:

terraform {
  required_providers {
    google = {
      source  = "hashicorp/google"   # registry address of the provider
      version = "~> 6.0"             # "~>" = allow 6.x, block 7.0 surprises
    }
  }
}

provider "google" {
  project = var.project_id           # comes from terraform.tfvars
  region  = "europe-west1"
}

variable "project_id" {
  type        = string
  description = "GCP project to deploy into"
}

resource "google_storage_bucket" "demo" {   # type + local name
  name                        = "${var.project_id}-tf-demo"  # globally unique
  location                    = "EU"
  force_destroy               = true        # allow destroy even if non-empty
  uniform_bucket_level_access = true        # IAM-only access (best practice)
}

output "bucket_url" {
  value = google_storage_bucket.demo.url    # printed after apply
}

Run it:

gcloud auth application-default login   # how Terraform gets GCP credentials
terraform init                          # downloads the google provider
terraform plan                          # expect: Plan: 1 to add, 0 to change, 0 to destroy
terraform apply                         # type "yes" → bucket exists
terraform destroy                       # type "yes" → bucket gone, wallet safe

5 · Reading plan output — the symbols

plan prefixes every affected resource with a symbol. Always scan for - and -/+ before typing yes.
+create — new resource · safe
~update in-place — change an argument, resource survives · usually safe
-destroy — resource will be deleted · ⚠ data loss possible
-/+replace — destroy then re-create (a "forced replacement"; the plan marks the guilty argument with # forces replacement) · ⚠⚠ downtime + data loss
<=read a data source · harmless

The summary line is your sanity check: Plan: 1 to add, 0 to change, 0 to destroy. — if you expected a rename and see 1 to destroy, stop and read why.

6 · Modules in brief

A module is a folder of .tf files used as a reusable unit — a function for infrastructure. Your working directory is already the root module; calling another one looks like:

module "network" {
  source  = "terraform-google-modules/network/google"  # from the registry
  version = "~> 9.0"
  project_id   = var.project_id    # module's input variables
  network_name = "my-vpc"
}
# consume its outputs:  module.network.network_self_link

source can be a registry address, a Git URL, or a local path (./modules/x). Run terraform init again after adding one. For ACE-level learning: use well-known Google modules before writing your own.

7 · .tfvars and secrets hygiene

terraform.tfvars (auto-loaded) or anything.auto.tfvars supply variable values, keeping environment-specific data out of the config:

# terraform.tfvars
project_id = "my-gcp-project-id"
Rules that prevent a very bad week:
  • Never commit credentials. No service-account JSON keys in the repo, no secrets in .tf or .tfvars files that get committed. Prefer gcloud auth application-default login locally and Workload Identity in CI.
  • Never commit state (terraform.tfstate*) — it can echo secrets in plaintext.
  • Minimum .gitignore: .terraform/ *.tfstate *.tfstate.* *.tfvars (commit a sanitized terraform.tfvars.example instead), and any *.json key files.
  • Mark sensitive variables sensitive = true so plan output redacts them — but remember they still land in state.
Licensing note (checked 2026-06): in August 2023 HashiCorp moved Terraform from the open-source MPL (Mozilla Public License) to the BSL (Business Source License) — source-available, but restricting competing commercial use. The community response was OpenTofu, an MPL-2.0 fork now under the Linux Foundation — a near drop-in replacement (tofu init/plan/apply, same HCL, same providers). IBM acquired HashiCorp in 2025; the BSL remains in place as of mid-2026. For learning and for the ACE exam mindset, the two are interchangeable; just know both names exist and why.

8 · Teardown

destroy closes the loop the sheet opened with — but read before you type yes.

terraform destroydestroy everything this config manages — shows a destroy plan first; read it (every line is a -) before approving
terraform destroy -target=google_storage_bucket.demodestroy one resource only — surgical; use sparingly (targeting fights the declarative model)
terraform state rm <addr>forget a resource without destroying it — removes it from state; the real resource lives on, unmanaged
What NOT to dohand-delete resources in the console → state drift: state says it exists, reality disagrees; next plan gets confused. Let Terraform do the deleting. ⚠
Safe to delete.terraform/ (provider cache — init rebuilds it) and *.tfplan files
Never deleteterraform.tfstate — the sacred file (§2). Deleting it orphans every resource it tracks. ⚠
Memory hook: destroy through Terraform, never around it — the state file only knows about deletions it performed itself.
