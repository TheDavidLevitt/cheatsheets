WebSockets Cheat Sheet

The persistent two-way connection behind live apps · started 2026-06-12 · twin of Tools/websockets_cheatsheet.md

0 · The mental model (read once)

WebSocket = a protocol (standardized as RFC 6455 — Request for Comments 6455, the document defining it) that keeps one connection open so browser and server can send messages to each other anytime, in either direction. Full-duplex = both ends can talk at once, like a phone call, not strict turns.
The problem it solves: HTTP (HyperText Transfer Protocol) is strictly request/response — the client must ask before the server can answer; the server can't push. For chat, dashboards, multiplayer, trading tickers, and collaborative editing, faking server-push with HTTP means ugly polling (asking on a timer).
A letter vs. a phone call — the core image: 
HTTP                          WebSocket
────                          ─────────
client: "any news?"  ─────►   one handshake, then the line stays open
server: "here you go" ◄────
(connection closes)           client ───► server   anytime
client: "any news now?" ──►   server ───► client   anytime
server: "here you go" ◄────   (no re-asking, no re-connecting)
HTTP is mailing letters — each ask/reply is a fresh envelope; to learn what changed you mail again (polling). WebSocket is a phone call left off the hook — dial once (the handshake), then either side speaks whenever there's news.
Two URL schemes: ws:// = plain (like http://) wss:// = over TLS (like https://) — USE THIS
TLS = Transport Layer Security, the encryption behind HTTPS. wss:// is encrypted and far more likely to pass corporate proxies.

1 · How it starts — the HTTP Upgrade handshake

A WebSocket is born as a normal HTTP request. The client sends a GET with headers asking to switch protocols; if the server agrees it replies 101 Switching Protocols and the same connection becomes a WebSocket.

Client asks → server agrees

GET /chat HTTP/1.1
Host: example.com
Upgrade: websocket          ← "switch me"
Connection: Upgrade
Sec-WebSocket-Key: <random>
Sec-WebSocket-Version: 13

HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: <hash>

Why it's clever

Upgrade handshakea normal HTTP request that converts the connection — reuses existing web infrastructure
101 Switching ProtocolsHTTP status meaning "agreed — no longer HTTP from here"
Same TCP connectionthe same TCP (Transmission Control Protocol, the reliable byte-stream layer) socket is reused — nothing reconnects
Ports 80 / 443starts as HTTP, so rides standard web ports (80 plain, 443 TLS) and slips through firewalls
Why the disguise pays off: firewalls, proxies, and load balancers already handle HTTP on 80/443. Starting as HTTP, a WebSocket inherits that compatibility for free.

2 · After the handshake — frames & keep-alive

Once upgraded, HTTP is gone. What flows is a thin framing layer over TCP — small headers wrapping each chunk. You rarely touch frames directly (the browser hands you whole messages), but the frame types explain the behavior.

Texta UTF-8 string (JSON, plain text) — the common case
Binaryraw bytes (images, audio, protobuf)
Ping / Pongkeep-alive heartbeat: one side pings, the other must pong — confirms the line is live and stops idle proxies killing it
Closea polite "hanging up" frame that begins the close handshake (§7)

So a WebSocket is just: handshake once → exchange text/binary frames freely → ping/pong to stay alive → close frame to end.

3 · A tiny browser example (JavaScript)

The browser ships a built-in WebSocket object. The four events — open, message, error, close — plus send and close are essentially the whole surface. The magic line is onmessage: it fires whenever the server pushes, which plain HTTP could never do.

// 1. Dial. Always wss:// (encrypted) in production.
const socket = new WebSocket('wss://example.com/chat');

// 2. The line is open and ready.
socket.onopen = () => {
  socket.send('hello server');        // send a message anytime
};

// 3. Server PUSHED us a message — no asking required.
socket.onmessage = (event) => {
  console.log('got:', event.data);    // event.data is the message
};

// 4. Something broke (network drop, server gone).
socket.onerror = (err) => {
  console.error('socket error', err);
};

// 5. The line closed. event.code tells you why (see §7).
socket.onclose = (event) => {
  console.log('closed:', event.code, event.reason);
};

// To hang up deliberately:
socket.close(1000, 'done');           // 1000 = normal closure

4 · When to use WebSocket vs. the alternatives

Don't default to WebSockets. They win for true two-way, low-latency traffic; lighter options win when you don't need both directions.

WebSocket
both ways		Chat, multiplayer, trading tickers, collaborative editing — both sides talk constantly. Cost: more infra; you handle reconnection. Service-to-service analog: grpc_cheatsheet.md.
SSE (Server-Sent Events)
server → client		Live feeds, notifications, dashboards where the client only receives. Simpler than WebSockets and auto-reconnects. Browser-native via EventSource. One-way only.
HTTP polling / long-polling
client asks		Fallback when WS/SSE are blocked. Long-polling holds the request open until news. Higher latency — the "ugly" baseline WebSockets replace. See http_cheatsheet.md.
Webhooks
server → server		A backend notifies another backend of an event (payment succeeded). Not for browsers; receiver needs a public URL.
gRPC streaming
service ↔ service		Internal microservices needing typed, high-performance streams. Not natively browser-friendly. See grpc_cheatsheet.md.
Rule of thumb: need the server to push but the client never replies on the same channel? Reach for SSE first — simpler and auto-reconnects. Need genuine back-and-forth? That's when WebSockets earn their cost.

5 · Practical notes — scaling, auth, libraries

Scaling & auth

Scalinga WebSocket is long-lived, pinned to one server. Many servers need sticky sessions (load balancer keeps a client on one server) or a pub/sub backplane — a message bus (e.g. Redis) relaying messages between instances so any can reach any client
Authhappens at the handshake — still an HTTP request, so cookies or a token (query param or Sec-WebSocket-Protocol header) are checked then. After upgrade there are no more HTTP requests to attach credentials to

Libraries

Socket.IO = the popular JS library wrapping WebSockets with automatic reconnection, "rooms" (broadcast groups), and a polling fallback. Convenient — but it's its own protocol: a Socket.IO client can't talk to a raw WebSocket server, and vice versa.

6 · The whole lifecycle in one picture

client                                   server
  │  GET /chat  Upgrade: websocket  ────► │   (handshake = an HTTP request)
  │  ◄──── 101 Switching Protocols        │   (agreed; same TCP socket reused)
  │  ──── "hi" (text frame) ────────────► │
  │  ◄──── "welcome" (text frame) ────────│   (server PUSHES, unprompted)
  │  ──── ping ────────────────────────►  │
  │  ◄──── pong ──────────────────────────│   (keep-alive heartbeat)
  │  ──── close (1000) ─────────────────► │
  │  ◄──── close (1000) ──────────────────│   (close handshake, both sides ack)

7 · Teardown — closing & reconnecting

Closing is its own handshake: one side sends a close frame with a status code, the other acks, then TCP ends.

Close codes

1000Normal closure — a clean, deliberate hang-up (socket.close())
1001going away — server shutting down or tab navigating off
1006Abnormal closure — connection dropped without a close frame (network died, server crashed). You never send 1006; the browser reports it when the line vanished

Reconnect with exponential backoff

Raw WebSockets don't auto-reconnect (a reason to use Socket.IO or SSE). On unexpected close: wait 1s, 2s, 4s, 8s… (capped, often with random "jitter") so a recovering server isn't hammered.

let delay = 1000;                     // start at 1s
function connect() {
  const s = new WebSocket('wss://example.com/chat');
  s.onopen  = () => { delay = 1000; };  // reset on success
  s.onclose = (e) => {
    if (e.code !== 1000) {              // not deliberate → retry
      setTimeout(connect, delay);
      delay = Math.min(delay * 2, 30000); // double, cap 30s
    }
  };
}
connect();
