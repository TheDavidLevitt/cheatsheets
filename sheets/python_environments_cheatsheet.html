Python Environments Cheat Sheet

venv · conda · uv · pixi — isolation, activation, and tool choice · started 2026-06-06 · twin of Tools/python_environments_cheatsheet.md

0 · The mental model (read once)

Environment (virtual environment) = an isolated Python setup: a pointer to (or copy of) the interpreter (the python executable that runs your code) plus its own private site-packages/ (the folder where installed libraries land). One toolbox per project — what you install in one is invisible to the others.
Why bother? ① Dependency conflicts: Project A needs pandas 1.5, Project B needs pandas 2.2 — one global Python can't hold both; two envs can. ② "Works on my machine" disease: your code only runs because of a forgotten install; an env + spec/lock file makes dependencies explicit & reproducible. ③ Protect system Python: the OS depends on it (see §1). 
Project A              Project B             System Python
.venv/                 .venv/                /usr/bin/python3
 ├─ bin/python          ├─ bin/python        (hands off — the
 └─ site-packages       └─ site-packages      OS depends on it)
    pandas 1.5             pandas 2.2
      ▲                      ▲
      └──── no conflict ─────┘
Activation demystified — it's just a tiny shell script, not magic: ① prepends the env's bin/ (Windows: Scripts\) to PATH — the shell variable listing dirs searched left→right for commands, so python and pip now resolve to the env's copies; ② sets VIRTUAL_ENV/CONDA_PREFIX; ③ adds the prompt prefix (.venv) as a visual cue. deactivate restores the old PATH. Corollary: activation is optional — .venv/bin/python script.py by full path works identically.

1 · System Python — never install into it

On macOS/Linux, /usr/bin/python3 belongs to the operating system. Since 2023, pip refuses to write into it:

error: externally-managed-environment
× This environment is externally managed
This is PEP 668 (Python Enhancement Proposal — Python's numbered design docs). The OS marks its Python as managed by apt/brew/the OS, not pip. The error is a feature: it pushes you into an environment. Never bypass with --break-system-packages.
See the error abovecreate a venv and install there
Need a CLI tool globallyuv tool install <name> or pipx install <name> — private env per tool, just the command exposed
Unsure which python runswhich python / where python — see §5

2 · The contenders

venv — "the toolbox in the trunk"

Standard-library module (python -m venv); creates a bare env, pair with pip (Python's package installer, pulls from PyPI — the public package index). Always available, zero install. pip installs, venv isolates — partners, not rivals.

conda / Anaconda — "the whole workshop"

Cross-language package + env manager. Anaconda = distribution bundling conda + 200+ data-science packages. Installs pre-compiled binary packages incl. non-Python deps (CUDA, MKL, C libs) from channels (package repos; conda-forge is the community one). Data-science standard; the usual corporate/Windows setup.

uv — "the electric screwdriver"

Astral's Rust all-in-one: replaces venv + pip + pip-tools + pyenv + pipx, 10–100× faster than pip. Project-centric: pyproject.toml + uv.lock, auto-creates .venv, even downloads Python versions. The modern default recommendation (v0.11.x, 2026 — pre-1.0 but hundreds of millions of downloads/month).

pixi — "conda's fast young cousin"

By prefix.dev, also Rust. Modern successor for the conda ecosystem: speaks conda channels and PyPI, project-based (pixi.toml/pyproject.toml) with a cross-platform lockfile (machine-written exact pins of the whole dependency tree). cargo/npm-style workflow (v0.59, 2026).

When to use what: work / corporate / data-sci → conda quick script → uv run script.py new project → uv reproduce a conda env → pixi (--import environment.yml) or conda minimal / teaching → venv + pip

4 · Spec files — declaring dependencies

requirements.txt (pip)flat list, one pkg per line (pandas==2.2.0); simple, but pip freeze captures everything installed, not just what you chose
environment.yml (conda)YAML (human-friendly config format): env name, channels, deps — incl. non-Python pkgs + a pip subsection
pyproject.toml (standard)TOML config; the standardized home for project metadata + deps (PEP 621) — read by uv, pixi, poetry, pip itself
uv.lock / pixi.lockmachine-generated exact pins + hashes of the whole tree. Spec = what you want; lockfile = exactly what you got. Commit both.

5 · Which Python is active right now?

Prompt prefix(.venv) / (base) = env active. conda's (base) is its default env — fine to see, bad to install into; make named envs.
which python / where pythonwhat the shell would run; under .venv/ or envs/myenv/ = you're in
python -c "import sys; print(sys.executable)"ground truth from the running interpreter — definitive when the shell lies (aliases, stale PATH)

6 · Editors & notebooks

VS Code interpreter picker

VS Code runs/debugs with whatever interpreter is selected, independent of your terminal. ⌘⇧P → "Python: Select Interpreter" — it auto-lists .venv/ and conda envs; the choice shows in the status bar. "Module not found" in VS Code but not terminal (or vice versa) = the two point at different interpreters.

Jupyter kernels

Kernel = the process Jupyter sends notebook code to. Jupyter = the front desk; kernels = the rooms — they can live in different envs. Register an env as a kernel via ipykernel:

# inside the activated env:
pip install ipykernel
python -m ipykernel install --user \
  --name myenv --display-name "Python (myenv)"

jupyter kernelspec list shows kernels; … uninstall myenv removes one.

7 · Teardown — leaving and deleting envs

deactivate / conda deactivateleave the env (venv/uv vs conda) — restores PATH; the env stays on disk
rm -rf .venvdelete a venv — it's just a folder; nothing else to unregister. uv recreates it from pyproject.toml + lockfile on the next uv run/uv sync
conda env remove -n myenvdelete a conda env (deactivate first; conda env list to see what exists)
pip uninstall x ≠ removing the envuninstall removes one package from the env. Broken env? Don't surgically uninstall — delete and recreate from the spec file
jupyter kernelspec uninstall myenvkernels register separately (§6) — deleting an env leaves a zombie kernel entry behind
which python + sys.executableverify you're back on system Python — should point outside any .venv/ or envs/ path (§5)
