Git Basics Cheat Sheet

Local workflow: init · add · stage · commit · started 2026-06-05 · twin of Tools/git_cheatsheet.md

0 · The mental model (read once)

Git = distributed version-control system (VCS) by Linus Torvalds, 2005. "Distributed" = every clone holds the full history; you can commit/branch/log with no server. VCS = software that records file changes over time so you can recall, compare, and merge versions.
Three places a change lives — the single most important idea: 
working tree   →   staging area   →   repository
(your files)       (the "index")      (.git, permanent)
     │                  │                    │
  git add ─────────────►│                    │
     │              git commit ─────────────►│
Working tree = files you edit on disk. Staging area / index = holding pen for the exact snapshot you'll commit next (editing does NOT auto-queue a file — you must add it). Repository = the .git/ folder; once you commit, it's permanent history with a unique ID.
Command grammar: git <verb> [options] [arguments]. Flags are short (-m) or long (--message) — Unix convention of one dash + letter vs. two dashes + word. Manual for any command: git help <verb>.

1 · git init — start a repository

init = initialize. Creates the hidden .git/ folder. Run once, in the project's top folder.

Create the repo

git initturn current folder into a repo
git init myprojectcreate folder + init in one step
git init -b maininit with initial branch named main (-b = branch)
git statusyour compass — run it constantly

One-time identity setup

Stamps every commit with author info. --global → ~/.gitconfig (all repos); drop it for just this repo.

git config --global user.name  "Your Name"
git config --global user.email "you@example.com"

2 · git add — stage changes

add copies a file's current content into the staging area. It stages new files AND modifications — think "add this version to the next snapshot."

Stage

git add file.txtstage one file
git add .stage everything in this folder & below
git add -Astage all changes in whole repo, incl. deletions (-A = all)
git add -ustage mods/deletes of tracked files only (-u = update)
git add -ppatch mode: choose chunks interactively

Unstage / remember

git restore --staged file.txtunstage but keep edits (modern)
git reset HEAD file.txtsame, older syntax

Distinction: . stages by location, -A by repo, -u by tracking status.

3 · Staging vs. committing

"Staging" isn't a command — it's the state after git add. Workflow: edit → add (stage) → commit. git status color-codes it: green = staged red = modified, not staged 
$ git status
Changes to be committed:        ← green: will be in next commit
        modified:   app.py
Changes not staged for commit:  ← red: edited but NOT staged yet
        modified:   notes.md

git diff shows red (unstaged); git diff --staged shows green (about to commit). Check both before committing.

4 · git commit — save a snapshot

commit writes the staged snapshot permanently, with a message and a unique SHA-1 hash (40-char fingerprint; you use the first 7). Borrowed from banking/databases: to commit = make final & durable.

Commit commands

git commit -m "message"commit staged changes, inline message
git commitopens editor for a longer message
git commit -am "msg"stage all tracked mods + commit (skips new files)
git commit --amendreplace last commit — don't amend pushed ones

Good message convention

Line 1 ≤ 50 chars, imperative mood: "Add login validation", not "Added". Test: "If applied, this commit will ___". Blank line, then a body explaining why.

$ git commit -m "Add input validation to login form"
[main a3f5e9c] Add input validation to login form
 1 file changed, 12 insertions(+), 2 deletions(-)

5 · The whole loop in one example

git init -b main                      # 1. start the repo
git config user.name "Your Name"   # 1b. (first time) identify yourself
echo "hello" > readme.txt             # 2. create/edit a file
git status                            # 3. see it in red (untracked)
git add readme.txt                    # 4. stage it (now green)
git status                            # 5. confirm it's staged
git commit -m "Add project readme"    # 6. save snapshot to history
git log --oneline                     # 7. see your commit

6 · Inspect & undo (handy neighbours)

Look

git statusworking tree + index state
git log --onelinecompact history
git log --oneline --graphwith ASCII branch graph
git diffunstaged changes (red)
git diff --stagedstaged preview (green)
git show HEADshow the latest commit

Undo (⚠ careful)

git restore file.txtdiscard unstaged edits (irreversible)
git restore --staged file.txtunstage, keep edits

HEAD = pointer to "where you are now" (tip of current branch). HEAD~1 = one commit back.

7 · Teardown / undo — getting out of Git

Undo & untrack

rm -rf .gitun-git a folder: deletes all history/branches/config; working files survive. ⚠ no undo
git reset --soft HEAD~1undo last commit, keep changes staged
git reset --hard⚠ discard ALL uncommitted work (tree + index)
git rm --cached file.txtuntrack a file but keep it on disk (then add to .gitignore)

Throwaway experiment pattern

git switch -c scratch    # disposable branch
#  ... risky edits, commits ...
git switch main
git branch -D scratch    # -D = force-delete, even unmerged

--soft vs --hard: both move HEAD back; --soft keeps your changes, --hard erases them. When unsure, start soft.
