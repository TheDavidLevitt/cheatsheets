BGP Cheat Sheet

Border Gateway Protocol — the glue between the internet's ~100k networks · started 2026-06-12 · twin of Tools/bgp_cheatsheet.md

0 · The mental model (read once)

BGP (Border Gateway Protocol) = the protocol by which the world's independently-run networks tell each other which IP address ranges they can reach. It's the internet's master routing table, built live by agreement. "Border" = it runs at the borders between networks, not inside them. Spec: RFC 4271 (BGP-4, 2006).
A postal service FOR postal services. The internet is a postal system for packets; BGP is the layer above — the post offices announcing to each other which ZIP codes they deliver to: 
"I, Network A, can reach 203.0.113.* — route it through me."
        │
   neighbours re-announce onward, adding themselves to the path:
        │
"To reach 203.0.113.*, go A → me — two hops."
The internet is not one network — it's ~100,000+ separate networks, each run by a different org, with no central route authority. BGP is how they gossip reachability until every router has a path to every range.
Policy, not shortest-path. BGP doesn't find the fastest route — it finds a route each network is willing to use, given who it pays and who it peers with (§2). One-liner: BGP is route gossip between countries.

1 · Core terms — the vocabulary

The nouns

AS
Autonomous System		one network under a single admin policy — an ISP, cloud, big university, CDN. The atomic unit BGP routes between (~100k+ on the net).
ASN
AS Number		an AS's identity, e.g. AS15169 (real) or AS64500 (example). A country code for networks.
IP prefixa CIDR block (Classless Inter-Domain Routing), e.g. 203.0.113.0/24 = 256 addresses. What an AS actually advertises — never a single IP.
Route advertisementa BGP message: "I can reach prefix X, here's the AS-path." The unit of gossip.

The two flavours + the meeting place

eBGP
external BGP		spoken between different ASes — the actual internet-stitching at the borders.
iBGP
internal BGP		spoken within one AS, to share externally-learned routes among that AS's own routers.
IXP
Internet Exchange Point		a physical building where many ASes plug into a shared switch to peer directly, instead of paying a third party to carry traffic between them.

Private/doc ASNs: just as 203.0.113.0/24 is reserved for docs, ASNs 64496–64511 (+ 64512–65534) are reserved for examples — this sheet uses AS64500–AS64511.

2 · Peering vs transit — the economics (who pays whom)

The part most explanations skip — and why BGP routes are policy, not distance. Two networks connect in two economically opposite ways.

Transit — buying access

Relationshipcustomer buys internet access from a provider
Who paysthe customer pays the provider (per Mbps / per port)
You getreachability to the whole internet
Analogypaying FedEx to deliver anywhere

Peering — swapping traffic

Relationshiptwo networks swap traffic directly
Who paysusually nobody (settlement-free), if traffic is roughly balanced
You getreachability to just that peer + its customers
Analogytwo firms swapping their own mail bags for free, often at an IXP
A Tier 1 network reaches the entire internet through peering alone — it buys transit from nobody. Everyone else buys some transit. Why this drives routing: a network prefers a free peering route or a paying-customer route over expensive transit, even if transit is shorter. BGP encodes that as policy.
Mental hook: money flows down (customers pay providers), traffic-swap is sideways (peers). BGP route selection quietly optimises the bill, not the mileage.

3 · How a route is chosen — the intuition

When a router learns several paths to one prefix it picks a "best" one. The full process has ~13 tie-breakers; you need the top two.

The two that matter

1. Local Preference"which exit does my own network prefer?" — set by policy (e.g. the cheap peering link). Highest wins. Local to one AS.
2. AS-path length"how many ASes does this route cross?" Fewer = preferred. Shortest wins. The closest thing to a globally-agreed tie-breaker.

Three intuitions

Not shortest-physical-path — AS-path counts networks crossed, not miles. A 2-AS slow path beats a 4-AS fibre path.
Local Pref beats AS-path — a network's own policy (who it pays) outranks global hop-count. That's how §2 economics override geography.
Loop prevention — a router seeing its own ASN already in a path discards the route (it'd be a circle).

4 · Why it's fragile and famous — BGP runs on trust

BGP's 1989 design assumed every operator was a known, cooperative academic or telco. So it largely believes whatever a neighbour advertises — there's no built-in check that an AS actually owns the prefix it announces.

Two failure modes

Route leakan AS re-advertises routes it shouldn't; traffic floods through a network too small to carry it. Accidental — a misconfig. Traffic takes a wrong, congested turn.
BGP hijackan AS advertises a prefix it does not own (often a more-specific prefix, which wins by longest-match), and the world routes that traffic to it. Malicious or fat-finger.

The memorable anchors

2008
Pakistan / YouTube		an ISP advertised a more-specific YouTube prefix to null-route it domestically — but leaked it globally. More-specific wins, so the world sent YouTube traffic to Pakistan, where it was dropped. YouTube dark for ~2h.
2021
Facebook		a maintenance change withdrew the BGP advertisements for Facebook/Instagram/WhatsApp prefixes. No route advertised → no path to reach them. Erased from the internet for ~6h — self-inflicted.
The teardown-equivalent: in Git, rm -rf .git un-gits a folder. In BGP, a route withdrawal un-internets a network — stop advertising your prefix and within minutes nobody on Earth can find you. Going offline isn't a crash; it's the absence of an advertisement.

5 · The fix — RPKI & ROAs (cryptographic origin validation)

The defence: cryptographically prove who may announce a prefix.

RPKI
Resource Public Key Infrastructure		a public-key system (run by the Regional Internet Registries — RIRs — that hand out IP blocks) letting a prefix owner sign which AS may originate it.
ROA
Route Origin Authorization		the signed statement: "AS64500 may originate 203.0.113.0/24."
ROV
Route Origin Validation		what a router does with ROAs: an advertisement whose origin doesn't match a valid ROA is marked Invalid and can be dropped.
RPKI/ROA stops the wrong-origin hijack (announcing a prefix you don't own). It does not yet fully protect the path — a liar can still forge an AS-path (BGPsec/ASPA target that, less deployed). Adoption is gradual but real: most large transit networks now drop RPKI-Invalid routes, shrinking the blast radius vs 2008.
Mental hook: RPKI is DNSSEC for routing — it doesn't change how the system works, it just adds a signature so neighbours can stop blindly believing every announcement.

6 · How you'd actually observe BGP — no config required

You almost never "configure BGP" yourself. It's run by ISPs, clouds, and multi-homed enterprises (connected to two+ providers for redundancy, running BGP to fail over). For everyone else, BGP is something you observe.

Tools

whois <IP>"which AS/org owns this address?" A terminal command (preinstalled on macOS/Linux). Returns prefix, org, often the ASN — the fastest "who is this?"
Looking-glass server"what does the internet see as the route to X, from elsewhere?" A public portal hosted on a provider's router — routing from their vantage, not yours.
traceroute <host>"which hops does my packet cross?" Indirect — the realised path, from which you infer AS borders.
bgp.tools / RIPEstatweb dashboards: an AS's prefixes, peers, upstreams, route history. Mention-level — verify specifics before quoting.

Try it

whois 203.0.113.42        # who owns this IP, the org, often the ASN
traceroute example.com    # the realised hop path (infer AS borders)

Reality check: unless you run a network or a multi-homed business, you'll read BGP far more than you touch it. The value is literacy — so "a BGP route leak" reads as traffic got mis-announced to the wrong network, not a hardware fault.

7 · Bridge to cloud (where you do meet BGP)

The one place a non-network-engineer meets BGP config is hybrid cloud. In GCP a Cloud Router speaks BGP to your on-prem network over Cloud VPN or Cloud Interconnect, so new subnets on either side are learned dynamically — no static routes to maintain. AWS (Direct Connect / VPN) and Azure (ExpressRoute) do the same. Details: GCP networking sheet.
