TLS Cheat Sheet

The layer behind HTTPS & the padlock: handshake · certificates · chain of trust · started 2026-06-12 · twin of Tools/tls_cheatsheet.md

0 · The mental model (read once)

TLS = Transport Layer Security: the standard crypto protocol that slips between TCP (Transmission Control Protocol, the reliable byte pipe) and an app protocol like HTTP (HyperText Transfer Protocol). HTTP inside a TLS tunnel = HTTPS (HTTP Secure) — the https:// URL and the padlock.
A sealed, verified envelope around your connection. Raw TCP is a postcard anyone on the wire can read and edit. TLS makes it a sealed, tamper-evident, signed envelope: 
application data (HTTP, SMTP, …)
┌──────────────────────────────┐
│   T L S  (the envelope)       │  encrypt · authenticate · integrity
└──────────────────────────────┘
┌──────────────────────────────┐
│   T C P  (the postal pipe)    │  reliable, ordered delivery
└──────────────────────────────┘
            IP  →  the wire
Three guarantees — the why of the whole protocol: 1 · Encryption (eavesdroppers see gibberish) 2 · Authentication (really example.com, not an impostor) 3 · Integrity (no byte tampered in transit)
Encryption alone is useless without authentication — encrypting a chat with an impostor just lets the impostor read you privately. The certificate is what makes the encryption worth having.

1 · Name history — why "SSL" still means TLS

TLS is a rename of an older protocol, so the names are used interchangeably (you "buy an SSL certificate" that is 100% TLS).

The lineage

1994 · SSL 2.0Netscape's Secure Sockets Layer; flawed, dead
1996 · SSL 3.0last SSL; broken (POODLE, 2014) — never use
1999 · TLS 1.0IETF takes over, renames SSL → TLS
2008 · TLS 1.2the workhorse for a decade
2018 · TLS 1.3current: faster handshake, dropped old broken crypto

What to remember

SSL = Secure Sockets Layer (Netscape, 1994). TLS = Transport Layer Security (IETF, 1999+). Same lineage; "SSL" is the sticky brand name.

Target today: TLS 1.3 (or 1.2). SSL 2/3 and TLS 1.0/1.1 are deprecated — browsers refuse them.

IETF = Internet Engineering Task Force (standardizes protocols via RFCs, Request for Comments docs).

2 · Symmetric vs. asymmetric — the key idea

From encryption_cheatsheet.md. This is why a handshake exists at all.

Two kinds of crypto

Asymmetric (public-key)a key pair: public (shareable) + private (secret). Solves "how do strangers agree on a secret?" but slow. RSA, ECDHE, Diffie-Hellman
Symmetric (shared-key)one shared secret both hold. Very fast on bulk data, but "how to share the key first?" AES (Advanced Encryption Standard)

The trick TLS plays

Use asymmetric crypto briefly to establish a shared secret over an open wire (the handshake), then switch to symmetric (AES) to transmit all data fast.

Memory anchor: asymmetric-to-establish, symmetric-to-transmit — the whole protocol in seven words. The shared secret it produces is the session key.

3 · The handshake, step by step (TLS 1.3)

The opening negotiation, done before real data: agree on crypto, prove the server's identity, secretly derive the session key. TLS 1.3 does it in one round trip (1-RTT).

CLIENT (browser)                          SERVER (example.com)
   │                                              │
   │ 1. ClientHello ────────────────────────────► │
   │    "I speak TLS 1.3. My cipher suites,        │
   │     my Diffie-Hellman public share, a random."│
   │                                              │
   │ ◄────────────────── 2. ServerHello + Cert     │
   │    "Use cipher X. My DH share + random +      │
   │     my CERTIFICATE (signed proof I'm          │
   │     example.com) + a Finished check."          │
   │                                              │
   │ 3. Verify + derive keys                       │
   │    cert chains to a trusted CA, name matches, │
   │    not expired; both sides combine the two DH │
   │    shares → the SAME session key              │
   │                                              │
   │ 4. Finished ───────────────────────────────► │
   │ ⇄ 5. ENCRYPTED application data (HTTP) ⇄       │

Step by step

1 · ClientHellobrowser lists TLS versions + cipher suites, sends its ephemeral DH share + a random
2 · ServerHello + Certserver picks a cipher, sends its DH share + its certificate (the authentication leg)
3 · Verify + deriveclient validates cert chain; both mix the two DH shares into the same session key — never sent on the wire
4 · Finisheda checksum proving the handshake wasn't tampered (integrity)
5 · App dataall further traffic AES-encrypted with the session key

Forward secrecy

DH / Diffie-Hellman = key-exchange letting two parties derive a shared secret over a public channel. TLS 1.3 mandates the ephemeral form (DHE / ECDHE) — a fresh keypair per session.

Forward secrecy (PFS, Perfect Forward Secrecy): because the session key came from throwaway keys, an attacker who later steals the server's long-term private key still can't decrypt past recorded sessions. Each conversation's secret died with it.

4 · Certificates & the chain of trust

A certificate = a digital ID card for a server: a public key + identity fields, signed by a Certificate Authority (CA) your device already trusts. That signature is what makes the padlock meaningful.

What a cert contains

Subject / CN / SANwho it's for: the domain(s), e.g. example.com, *.example.com. SAN = Subject Alternative Name (modern domain list)
Public keyserver's public key (private half never leaves the server)
Issuerwhich CA signed it
Validity"not before" / "not after" — certs expire (often 90 days now)
Signaturethe CA's signature over all of the above

Root → Intermediate → Leaf

Root CA       ← self-signed; in your TRUST STORE
  │ signs
Intermediate  ← CA's working key (roots kept offline)
  │ signs
Leaf cert     ← example.com's cert, served in handshake

Trust store / root store = the root CA certs shipped with your OS/browser (Apple, Microsoft, Mozilla, Google each curate one). Trust in the leaf is anchored by trust in the root.

Let's Encrypt + ACME — why HTTPS went universal. Let's Encrypt = a free, automated, non-profit CA (2015). ACME = Automatic Certificate Management Environment — the protocol that proves you control a domain and fetches/renews a cert with no human (Certbot runs it on a timer). Together they removed the cost/friction that kept small sites on plain HTTP — the main reason the web went mostly-HTTPS after ~2016.
Self-signed certs & the scary warning. A self-signed cert is signed by its own key, not a trusted CA, so it chains to nothing in your trust store. Encryption still works; authentication doesn't — nobody vouches it's really example.com. Browsers show a full-page NET::ERR_CERT_AUTHORITY_INVALID because that's exactly what a man-in-the-middle attack looks like. Fine for localhost; never for the public web.

5 · Key concepts you'll hear

The vocabulary

Cipher suitethe agreed bundle of algorithms — key exchange + bulk cipher + hash, e.g. TLS_AES_128_GCM_SHA256. TLS 1.3 trimmed it to a few safe ones
Forward secrecy (PFS)ephemeral keys → a stolen server key can't decrypt past sessions (§3)
SNI (Server Name Indication)client names the wanted domain at the start of the handshake, so one IP can host many HTTPS sites and the server knows which cert to present
HSTSHTTP Strict Transport Security: a header saying "only ever reach me over HTTPS" — blocks downgrade-to-HTTP attacks
mTLS (mutual TLS)normally only the server proves identity; mTLS makes the client present a cert too. Common in zero-trust internal systems, rare for public users
Session keythe symmetric (AES) key derived in the handshake that encrypts the data

6 · Practical / CLI — inspecting TLS

Run in a terminal (macOS/Linux). openssl ships with most systems.

Inspect a server's cert

openssl s_client -connect example.com:443open a raw TLS connection; dump the cert chain, protocol, cipher. The workhorse
openssl s_client -connect example.com:443 -servername example.comsame, but sends SNI — needed when one IP hosts many sites
… | openssl x509 -noout -dates -subject -issuerprint just expiry dates, subject, issuer of the leaf cert
curl -v https://example.comfull HTTP-over-TLS conversation; -v prints cert + handshake details

Reading the padlock & errors

Click the padlock → "Connection is secure" → view the chain, covered domains, validity dates. A closed padlock = TLS succeeded and the name matched — not that the site is honest, only that the channel is private.

Expired ERR_CERT_DATE_INVALID"not after" date passed — usually a failed renewal. 90-day certs make auto-renewal (ACME) essential
Name mismatch ERR_CERT_COMMON_NAME_INVALIDvalid cert, wrong domain (cert for example.com, visited as www.example.com with no SAN). Doesn't vouch for the name you typed

7 · Where TLS sits — the layering

TLS is protocol-agnostic: it wraps TCP and sits under any app protocol. The same envelope secures more than the web.

Plain → +TLS

HTTPHTTPS · port 443
SMTP (mail send)SMTPS / STARTTLS · 465 / 587
IMAP (mail read)IMAPS · 993
Generic TCP"TLS tunnel"

The full stack

HTTPS = HTTP over TLS over TCP over IP. The padlock is just the visible tip of that stack.

8 · Teardown — revoking & rotating certs

Kill a cert before expiry

Rotationroutinely replacing a cert (+ ideally a fresh keypair) before expiry — mostly automated by ACME on short-lived certs
Revocationdeclaring a still-unexpired cert invalid (e.g. leaked private key)
CRLCertificate Revocation List — a CA-published blocklist of serial numbers. Bulky; clients must download it
OCSPOnline Certificate Status Protocol — ask "is this cert still good?" live. OCSP stapling lets the server attach that proof in the handshake

Why short-lived certs won

Revocation checking is famously unreliable (clients often skip it). So the industry moved to short-lived certs: a 90-day cert that leaks is dangerous for at most 90 days — no revocation needed.
