Claude API Cheat Sheet

Messages API: requests · tool use · caching · errors · models · started 2026-06-06 · twin of Tools/claude_api_cheatsheet.md

0 · The mental model (read once)

API (Application Programming Interface) = an HTTP contract for talking to software. The Claude API = POST https://api.anthropic.com/v1/messages — JSON in, JSON Message out. The SDK (Software Development Kit) = the anthropic client library that wraps the HTTP plumbing.
The API is stateless — the single most important idea. The server remembers nothing between calls; every request carries the entire conversation history: 
call 1: [user:"Hi"]                                  → asst:"Hello!"
call 2: [user:"Hi", asst:"Hello!", user:"2+2?"]      → asst:"4"
call 3: [user:"Hi", asst:"Hello!", user:"2+2?",
         asst:"4", user:"Double it"]                  → asst:"8"
You are the memory: keep a messages list, append every turn, resend it whole. You pay input tokens for the full history each call (→ prompt caching, §6). Memory hook: it's the Messages API, plural — you always send messages, never a message.
Request grammar: three required fields — model, max_tokens, messages — everything else optional. Roles in messages are only user / assistant, alternating; the system prompt is its own top-level field, not a role (that's the OpenAI convention, not Anthropic's). max_tokens is a hard ceiling, not a target — replies truncate at it.

1 · Setup: auth + SDK

Raw HTTP headers (SDK sets these)

x-api-key: sk-ant-...        ← key (placeholder!)
anthropic-version: 2023-06-01 ← API version pin
content-type: application/json

Never hard-code the key. export ANTHROPIC_API_KEY=sk-ant-... or a .env file. Hook: sk-ant = secret key, anthropic.

Python SDK — canonical call

# pip install anthropic
import anthropic
client = anthropic.Anthropic()  # reads ANTHROPIC_API_KEY

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system="You are a concise assistant.",
    messages=[{"role":"user","content":"Hello"}],
)
print(response.content[0].text)

2 · The response object (Message)

Fields

response.contentalways a list of typed content blocks
response.stop_reasonwhy generation ended (table →)
response.usage.input_tokens / output_tokensbilling counts
usage.cache_creation_input_tokens / cache_read_input_tokenscache diagnostics (§6)

Stop reasons

end_turnnatural end
max_tokenshit the ceiling — truncated
stop_sequencea custom stop string fired
tool_usemodel wants a tool — your move (§4)
pause_turnlong-running server tool needs time
Filter blocks by type, never index by position. content[0].text breaks the moment tool use / thinking / vision appears — and block ordering is model-dependent (Sonnet often [text, tool_use], Haiku often [tool_use] only). 
text = next((b.text for b in response.content if b.type == "text"), None)
tool = next((b for b in response.content if b.type == "tool_use"), None)

3 · Key parameters

Knobs

temperaturerandomness 0–1 (default 1.0). 0 ≈ deterministic — extraction/classification; 1.0 — creative. Rarely touch top_p/top_k.
stop_sequencesstrings that halt generation; the stop string is NOT included in output
systempersona/standing instructions; must be a list of blocks when caching (§6)

Streaming

Same content, delivered incrementally — show tokens as they generate.

with client.messages.stream(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        messages=messages) as stream:
    for text in stream.text_stream:
        print(text, end="", flush=True)

4 · Tool use — the agentic loop

Claude can't run code — it can only ask you to. Five steps: 
1. you   → prompt + tool schemas
2. model → stop_reason="tool_use" + tool_use block(s)
3. you   → execute the tool locally
4. you   → user message with tool_result block(s)
5. model → final answer (or more tool calls — loop to 2)

Tool schema (JSON Schema subset)

{
  "name": "get_weather",
  "description": "Get current weather for a city.
     Use when the user asks about weather.",
  "input_schema": {
    "type": "object",
    "properties": {
      "city":  {"type":"string",
                "description":"City name"},
      "units": {"type":"string",
                "enum":["celsius","fahrenheit"]}
    },
    "required": ["city"]
  }
}

⭐ description is the most important field — it drives when Claude calls the tool. Param descriptions second. "strict": True guarantees schema-exact calls — the modern structured-output route (prefill is dead on 4.6+).

Result back + dispatch

{"role": "user", "content": [{
  "type": "tool_result",
  "tool_use_id": tool_use.id,  # pairs req & result
  "content": json.dumps(result),
  "is_error": False  # True → model recovers
}]}
TOOLS = {"get_weather": get_weather,
         "get_news": get_news}
func = TOOLS[tool_use.name]
result = func(**tool_use.input)
# ** unpacks dict as kwargs — schema params
# must match function param names

Built-in server tools need no dispatcher: web_search_*, code execution, text editor (current name: str_replace_based_edit_tool — versioned, must match exactly).

5 · Vision, PDFs, extended thinking

Images & PDFs (content blocks)

{"type": "image",
 "source": {
   "type": "base64",       # or "url"
   "media_type": "image/png",
   "data": base64.standard_b64encode(
             img_bytes).decode("utf-8")}}

PDFs: same pattern with "type": "document" — Claude reads text AND layout/figures natively.

Extended thinking

Internal reasoning before answering. Modern: thinking={"type": "adaptive"} (model picks depth). Legacy "enabled" needs budget_tokens. Responses gain thinking blocks; redacted_thinking blocks must round-trip unchanged with tools.

6 · Prompt caching

Stop re-paying for the stable prefix (big system prompt, tool schemas, long docs) on every stateless call. Mark a block with "cache_control": {"type": "ephemeral"}. Cache covers everything up to and including the last marked block — order content stable-first, volatile-last. TTL (Time To Live) = 5 min, refreshed on each hit. Verify in usage: cache_creation_input_tokens first call, cache_read_input_tokens (~10% of input price) after. 
system=[{                        # ⚠ system must be a LIST when caching —
    "type": "text",              #   bare string/dict throws
    "text": big_system_prompt,   #   400: system: Input should be a valid array
    "cache_control": {"type": "ephemeral"},
}]

7 · Model family (verified 2026-06-06)

Opus 4.8 · claude-opus-4-8most capable, most expensive — hard reasoning, long-horizon agents
Sonnet 4.6 · claude-sonnet-4-6the default — balanced capability/cost, most production work
Haiku 4.5 · claude-haiku-4-5fastest, cheapest — classification, translation, summarization, high volume
Memory hook: the names are poem lengths — Haiku (shortest) → Sonnet → Opus (a grand work). Every ID is a pinned snapshot; since the 4.6 generation IDs are dateless (claude-sonnet-4-6) but still pinned; older ones carry a date (claude-sonnet-4-5-20250929). All current models: 200K input context, default max output 4096. Strategy: default Sonnet → up to Opus when it fails → down to Haiku when overkill.

8 · Errors + retry conventions

Status codes

400 invalid_requestmalformed request — fix it, don't retry
401 authenticationbad/missing key — check ANTHROPIC_API_KEY
403 permissionkey lacks access
404 not_foundbad model name / endpoint
413 request_too_largetrim the payload
429 rate_limitback off & retry; honor retry-after
500 api_errorAnthropic-side — retry with backoff
529 overloadedtemporarily overloaded — retry with backoff

Retry rules of thumb

4xx (except 429) = your bug, don't retry. 429/5xx = transient: retry with exponential backoff (1s, 2s, 4s… + jitter). The SDK auto-retries these twice (max_retries).

RemoteProtocolError (connection drops, no clean 400) often = wrong endpoint routing — beta feature on the non-beta endpoint (client.beta.messages.create(betas=[...])). And read SDK warning lines in tracebacks — they often diagnose before the headline exception.

9 · Prompt engineering quick hits

Six techniques (priority order)

1 · Clear & directexplicit goals, no hedging
2 · Specificconcrete examples beat abstract instructions
3 · XML tagsseparate data from instructions; cuts injection risk
4 · Few-shot examplesshow the output shape, don't describe it
5 · Chain of thought"think step-by-step"; superseded by extended thinking for hard tasks
6 · Role assignmentsystem prompt anchors persona

XML conventions + prefill caveat

Common tags: <context> <task> <input> <instructions> <examples> <format> <thinking> — names arbitrary, consistency matters. Pairs with f-strings: f"<input>\n{user_data}\n</input>".

Prefilling (pre-writing the assistant turn to steer continuation) is a classic technique but disabled on Sonnet 4.6+. Modern structured output: output_config.format with a JSON schema, or strict tool use.

10 · Teardown / hygiene

There is no session to close — the API is stateless (§0). "Ending a conversation" = discarding your local messages list; the server already forgot. What actually needs cleaning up:
API keysrevoke or rotate in the Console (Settings → API keys) — especially any key that may have leaked
Uploaded filesdelete via the Files API — client.beta.files.delete(file_id); uploads persist until you remove them
Cached promptsnothing to do — TTL-based (5 min, §6), they expire on their own
Spend on a stale keyset the key's/workspace's spend limit to zero in the Console — a kill switch without deleting the key
Local hygienenever leave keys in code, notebooks, or shell history — env vars or a secrets manager; .env in .gitignore
Memory hook: stateless server = nothing to tear down there; the teardown surface is all on your side — keys, files, local copies.

APIApplication Programming Interface — the HTTP contract at api.anthropic.com
HTTPHyperText Transfer Protocol — the request/response protocol the web and this API run on
RESTRepresentational State Transfer — the web-API style: resources at URLs + HTTP verbs
SDKSoftware Development Kit — the anthropic client library
JSONJavaScript Object Notation — the {"key": value} data format requests/responses use
Statelessserver keeps no memory; client resends full history each call
Content blocktyped unit inside content (text, tool_use, thinking, image, …)
System promptstanding instructions; top-level field, not a message role
Tool useClaude requests a function call; you execute & return a tool_result
JSON Schemavocabulary that tool input_schemas are written in
XMLeXtensible Markup Language — the <tag>…</tag> syntax used as prompt delimiters
Prompt cachingreuse of a stable prompt prefix (cache_control, 5-min TTL)
TTLTime To Live — how long a cache entry survives without a hit
Stop reasonwhy generation ended (end_turn, max_tokens, tool_use, …)
Exponential backoffretry waits that double each attempt — for 429/5xx
Prefilllegacy: pre-writing the assistant turn; disabled on 4.6+
