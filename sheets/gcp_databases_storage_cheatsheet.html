0 · The decision spine (memorize this first)

Eight questions, eight services. Walk top to bottom and stop at the first yes: 
Structured + transactional (an app's tables)?  → Cloud SQL
   …but needs to be GLOBAL + strongly consistent? → Spanner
   …but just needs FASTER regional Postgres?      → AlloyDB
Flexible app/web/mobile data, JSON-ish docs?    → Firestore
Massive throughput, key-lookup, time-series?    → Bigtable
Just caching hot values for sub-ms reads?       → Memorystore
Analyzing huge datasets with SQL (dashboards)?  → BigQuery
Storing whole files (images, backups, logs)?    → Cloud Storage
The exam's favorite split — OLTP vs OLAP. OLTP (Online Transaction Processing) = many small, fast reads/writes — what a live app does (place order, update profile). Cloud SQL, Spanner, AlloyDB, Firestore, Bigtable live here. OLAP (Online Analytical Processing) = few huge queries scanning millions of rows to answer "what happened?" — BigQuery's whole job. Same language (SQL) as Cloud SQL, opposite workload.

The eight services at a glance

Mnemonic — "SQL SAFe B-M-B-C": SQL family (Cloud SQL, AlloyDB, Spanner) for transactions, Firestore for flexible docs, Bigtable for throughput, Memorystore for cache, BigQuery for analysis, Cloud Storage for files. Or just remember the spine's question order: shape → scale → speed → analysis → files.

1 · Data shapes — relational vs document vs wide-column vs key-value

Every database question is secretly a question about the shape of your data. Four shapes cover the exam.

ShapeMetaphorLooks likeGCP service(s)Strength / weakness
Relational a spreadsheet workbook — every row has the same columns tables, rows, fixed schema, foreign keys Cloud SQL, AlloyDB, Spanner joins + ACID transactions / schema changes are ceremony
Document a folder of forms — each form can have different fields collections of nested JSON-like documents Firestore flexible schema, maps to app objects / weak at cross-document joins
Wide-column a giant filing wall — billions of labeled drawers, most empty one big sparse table: row key → column families → cells Bigtable huge scale, fast key lookups / no SQL, no joins, design lives in the row key
Key-value a coat check — hand over a ticket, get exactly one item back key → opaque value, nothing else Memorystore fastest possible lookup / can't query by content at all
ACID = Atomicity, Consistency, Isolation, Durability — the transaction guarantees relational databases make (a bank transfer either fully happens or fully doesn't). NoSQL systems relax some of these to win scale; how much they relax is exactly what differentiates Firestore (per-document and multi-document transactions) from Bigtable (single-row atomicity only).

2 · Per-service drill-ins (click to expand)

Cloud SQL — managed MySQL / PostgreSQL / SQL Server · the default app database

A managed VM running the database engine you already know. Google handles patches, backups, replication, failover; you get a normal connection string. Example: a chat app's relational store — users, messages, channels — lives happily in a Cloud SQL Postgres instance called my-app-db.

EnginesMySQL, PostgreSQL, SQL Server — pick what your code speaks
Scale modelvertical (bigger machine) + read replicas; writes go through one primary
HA (High Availability)optional standby in another zone, automatic failover — regional, not global
Backupsautomated daily + point-in-time recovery (PITR) via write-ahead logs
Connectivityprivate IP, public IP + authorized networks, or the Cloud SQL Auth Proxy (IAM-authenticated tunnel — the recommended way)
Ceiling: regional scale only. When one primary can't take the write load, or users span continents, you've outgrown it — that's the Spanner/AlloyDB fork.

Docs: cloud.google.com/sql/docs

AlloyDB — PostgreSQL-compatible, rebuilt for speed · HTAP in one box

Google took the Postgres front end and swapped the storage engine underneath for its own disaggregated, log-structured one. Your app sees ordinary Postgres; Google claims ~4× transactional throughput vs stock Postgres.

Killer featurea built-in columnar engine: hot rows are also kept in column format in memory, so analytical queries run up to ~100× faster on the same database
HTAPHybrid Transactional/Analytical Processing — one system serving both OLTP and OLAP, instead of exporting to a warehouse
Pick it whenCloud SQL Postgres is too slow, you want dashboards over live data, or you're migrating off Oracle/commercial Postgres
Ceiling: still a single primary writer — read pool scales out, writes don't. Global write scale remains Spanner's territory. Lightly tested on the ACE exam; know what it is and its niche.

Docs: cloud.google.com/alloydb/docs

Spanner — globally distributed relational · the "impossible" database

Relational tables + SQL + ACID transactions, but sharded across machines and continents with horizontal scale-out — add nodes, get more read and write capacity. The trick: TrueTime, Google's GPS-and-atomic-clock API that timestamps transactions globally.

External consistencythe strongest guarantee: globally, transactions appear in the real-time order they committed — like one giant single machine
SLAup to 99.999% availability for multi-region configs (~5 min downtime/year)
Pick it whenrelational and global and consistent — financial ledgers, global inventory, multiplayer game state
Watch out: expensive (priced per node/processing unit + storage); individual queries can be slower than a local Postgres because consensus spans regions. Overkill unless you genuinely need global writes. Exam rule: see "global + relational + strong consistency" → answer Spanner.

Docs: cloud.google.com/spanner/docs

Firestore — serverless NoSQL document DB · MongoDB family · real-time

Stores documents (JSON-like maps of fields) grouped into collections; documents can nest subcollections. Same family as MongoDB — if you'd reach for Mongo, on GCP you reach for Firestore. Example: a recipe-sharing app stores each recipe as one document — title, ingredients array, nested ratings — no schema migration when you add a field.

Serverlessno instances to size; scales automatically; pay per operation + storage (generous free tier)
Real-time listenersclients subscribe to a query and get pushed updates — chat and collaborative apps for free
Offline syncmobile/web SDKs cache locally and reconcile when back online
ModesNative mode (the one to pick) vs Datastore mode (backward-compatible with the older Datastore API; server-side only, no real-time)
Watch out: queries need pre-built indexes; weak at heavy analytics and multi-document joins. For analysis, stream/export to BigQuery (see BigQuery card).

Docs: cloud.google.com/firestore/docs

Bigtable — wide-column NoSQL · the petabyte firehose

The system behind Search, Maps and Gmail, exposed as a service (HBase-compatible API). One enormous sparse table: a row key maps to column families full of timestamped cells. Rows are sorted by key and split into contiguous key ranges (tablets) spread across nodes — so throughput scales linearly: more nodes, more tablets served.

Pick it whenmillions of writes/sec at single-digit-ms latency: sensor/IoT (Internet of Things) telemetry, time-series metrics, ad-tech events, financial ticks
Row-key designthe only index — queries are key lookups and key-range scans. Design keys to spread load: sensor42#2026-06-06T10:00 good; a plain timestamp prefix sends all writes to one node (hotspotting)
Atomicitysingle-row only — no multi-row transactions, no SQL, no joins
Watch out: minimum ~1 node always on — there's a meaningful monthly floor even at zero traffic. Bigtable is for big workloads; small flexible data wants Firestore.

Docs: cloud.google.com/bigtable/docs

Memorystore — managed Redis / Memcached · a cache, not a store

In-memory key→value service for sub-millisecond reads. Sits in front of a real database: check the cache first, fall back to the DB, write the answer back into the cache (cache-aside pattern).

Use forcaching query results, session state, leaderboards (Redis sorted sets), rate limiting, pub/sub fan-out
Redis vs MemcachedRedis = rich data types, persistence options, replication; Memcached = simpler, multi-node pure cache. Default to Redis unless migrating existing Memcached code
TiersRedis Basic (single node, cache only) vs Standard (replicated, HA with automatic failover)
The rule that matters: RAM is not durable. A restart or eviction can empty it. Anything you can't recompute or refetch does not belong only in Memorystore. Exam phrasing: "store session data with lowest latency" → Memorystore; "store it durably" → a database.

Docs: cloud.google.com/memorystore/docs

BigQuery — serverless columnar OLAP warehouse · priced per scan

The analytics roof over everything else. Fully serverless: no clusters to size — load (or stream) data, write SQL, and Google fans the query out over thousands of workers. Scans terabytes in seconds.

Columnar storagedata stored per column, not per row — a query touching 3 of 200 columns reads only those 3. Great for analytics, wrong for single-row OLTP lookups
Pricing modelon-demand = pay per bytes scanned by each query (plus cheap storage); flat-rate/editions = reserved slots for heavy shops
External tablesquery data in place without loading it — federation over Cloud Storage files (CSV/JSON/Parquet) and Bigtable (GA — Generally Available)
Firestore routeno direct federation — use the Stream Firestore to BigQuery extension / export pipeline to land documents as rows
Native JSON typea JSON column type stores semi-structured data and queries it with dot notation — half-schemaless inside a warehouse
ExtrasBigQuery ML (train models in SQL), scheduled queries, free 10 GiB storage + 1 TiB scan/month tier
Not an app database. Per-query latency is seconds, inserts are batch/stream-oriented, and pricing punishes "SELECT one row" patterns. OLTP belongs in the spine's other branches.

Docs: cloud.google.com/bigquery/docs

Cloud Storage — object storage · buckets, classes, lifecycle, signed URLs

Object storage = whole files retrieved by name over HTTP, not a filesystem and not a disk. A bucket (globally-unique name, chosen location) holds objects (the files + metadata). Flat namespace — photos/2026/cat.jpg is one object whose name contains slashes; "folders" are an illusion the console draws.

ClassMin storage durationUse when accessed…
Standardnonefrequently — live website assets, active data
Nearline30 days≈ once a month — recent backups
Coldline90 days≈ once a quarter — DR (disaster recovery) copies
Archive365 days≈ once a year — compliance retention

Colder = cheaper storage but pricier retrieval + early-deletion fees. All classes have the same ms-level access latency — "cold" means billing temperature, not tape robots.

Lifecycle rulesper-bucket automation: "after 30 days → Nearline; after 365 → Archive; after 7 years → delete." Set once, forget forever
Signed URLstime-limited tokenized links granting anyone (no Google account) GET/PUT on one object — how apps let users download private files or upload directly to the bucket
Also knowuniform bucket-level access (IAM only, recommended) vs fine-grained ACLs; object versioning; gsutil/gcloud storage CLI; static-website hosting
ACLs — the legacy permission system. An ACL (Access Control List) is Cloud Storage's pre-IAM permission model (2010, modeled on Amazon S3): a grant list attached to each bucket and each object, giving entities READER/WRITER/OWNER. Why it's not recommended: ACLs run in parallel with IAM — access is granted if either system allows it — so answering "who can read this object?" means checking every object's ACL on top of the IAM policy. Per-object grants sprawl over time, and an allUsers READER ACL is the classic accidental-public-exposure vector.
The fix & exam answer: uniform bucket-level access. A bucket setting that disables ACLs entirely, making IAM the single source of truth. Recommended default for new buckets, enforceable org-wide by org policy, and irreversible 90 days after enabling. Exam triggers: "centralize / simplify bucket permissions" → uniform bucket-level access. "Per-object access for individual users" → fine-grained access (ACLs) is the legacy answer, but signed URLs or redesigning the bucket layout is usually better.

Docs: cloud.google.com/storage/docs

3 · Block storage — the disks under your VMs

Block vs object: block storage = a raw virtual disk attached to one VM (Virtual Machine), formatted with a filesystem, read/written in small blocks — what a VM boots from and what a self-managed database writes to. Object storage = whole files by name over HTTP, attached to nothing. Disk : bucket :: internal drive : shared file box.
The substrate: every network-attached disk (all Persistent Disk and Hyperdisk types) is virtual — blocks live replicated on Colossus, Google's distributed file system, and travel to your VM over the datacenter network. That's why a durable disk survives its VM and can grow live. Only Local SSD is a physical drive bolted to the host.

Persistent Disk (PD) — performance scales with size

TierMedia · rough perf vs StandardBest for
Standard PD pd-standardHDD · ×1 baselinecheap bulk/sequential data; non-boot drives
Balanced PD pd-balancedSSD · ≈×18the sensible default, incl. boot disks; best price/GB of the SSDs
SSD PD pd-ssdSSD · ≈×90performance-sensitive databases; best price/IOPS
Extreme PD pd-extremeSSD, provisionable IOPS · ≈×310high-end DBs (e.g. SAP HANA)

PD's quirk: IOPS (Input/Output Operations Per Second — random read/write rate) and throughput (MiB/s — bulk sequential rate) grow with disk size. Need more speed? Buy more gigabytes, even empty ones.

Hyperdisk — performance decoupled from capacity

The newer generation fixes PD's quirk: provision size, IOPS and throughput as three independent dials, tunable live without detaching.

Balancedgeneral default for new machine series
Balanced HAsynchronously replicated across two zones
Extremehighest IOPS — mission-critical databases
MLhuge read throughput, attach read-only to many VMs — model/dataset loading
Throughputcheap tunable MiB/s for analytics/scratch

Local SSD & boot disks

Local SSDphysical NVMe on the host: highest IOPS, lowest latency — and ephemeral: data is gone when the VM stops or migrates. Scratch space, caches, temp shuffle only
Boot diskevery VM has exactly one — holds the OS image; must be durable (so never Local SSD). Balanced PD / Hyperdisk Balanced is the normal choice
Memory hook: durable disks are network fictions on Colossus; only Local SSD is real — and the real one forgets.

Docs: cloud.google.com/compute/docs/disks

4 · Cost gotchas — where beginners get billed

BigQuery: you pay per scan, not per answer

SELECT * on a 2 TB table costs the same whether it returns one row or all of them — on-demand pricing bills bytes scanned. Defenses: never SELECT *; partition tables (by date) and cluster them so queries prune; check the dry-run estimate in the console before running; set per-user byte quotas.

Bigtable: the always-on floor

A minimum ~1-node cluster bills 24/7 whether or not a single request arrives. There is no scale-to-zero. Prototyping a small dataset? Use Firestore (free tier, serverless) and graduate to Bigtable when throughput demands it.

Stopped VM ≠ free VM

Stopping a VM stops compute billing — but its Persistent Disks, snapshots, and reserved static IPs keep billing. The classic surprise invoice is a fleet of stopped VMs with fat SSD disks attached. Delete the disk (or snapshot-then-delete) if you're done with it.

Egress: data checks in cheap, checks out expensive

Ingress (data in) is free; egress (data out to the internet or across regions/continents) is billed per GB. Keep the VM, the database and the bucket in the same region; cold storage classes add per-GB retrieval fees on top of egress.

Habit: set a budget alert on day one (Billing → Budgets & alerts) — it doesn't cap spending, but it emails you before a surprise becomes a story.

5 · Acronym & jargon decoder

OLTPOnline Transaction Processing — many small fast reads/writes (a live app)
OLAPOnline Analytical Processing — few huge scans for analysis (a warehouse)
HTAPHybrid Transactional/Analytical Processing — one engine doing both (AlloyDB)
ACIDAtomicity, Consistency, Isolation, Durability — transaction guarantees (term coined by Härder & Reuter, 1983)
SQL / NoSQLStructured Query Language (relational tables) vs "Not only SQL" — non-relational stores (document, wide-column, key-value)
IOPSInput/Output Operations Per Second — random-access rate; databases care
HDD / SSDHard Disk Drive (spinning, cheap, slow) vs Solid-State Drive (flash, fast) — the two disk media tiers
ThroughputMiB/s of bulk sequential data; analytics and big files care
Serverlessno servers to manage, autoscaling, pay-per-use — still servers underneath, like "wireless" still has wires
HA / SLAHigh Availability / Service Level Agreement (the uptime promise, e.g. 99.999%)
PITRPoint-In-Time Recovery — restore a DB to any moment, via replayed logs
PDPersistent Disk — network-attached durable block storage for VMs; performance scales with size
GCSGoogle Cloud Storage — the object-storage service (buckets of files)
ACLAccess Control List — Cloud Storage's legacy per-object/bucket permission model; prefer uniform bucket-level access (IAM only)
DRDisaster Recovery — backup copies kept for restoring after a regional failure (Coldline-tier use case)
External consistencySpanner's guarantee: global transactions appear in true commit order
Hotspottinga key design that funnels traffic to one node (e.g. timestamp-prefixed Bigtable row keys)
Federation / external tablequerying data where it lives instead of loading it (BigQuery over GCS/Bigtable)
Egress / ingressdata leaving / entering — leaving is the one that costs
ColossusGoogle's internal distributed file system — the substrate under PD, Hyperdisk, GCS, and the databases
