PowerShell Cheat Sheet

Objects · Verb-Noun · pipeline · discovery — for bash speakers on corporate Windows · started 2026-06-06 · twin of Tools/powershell_cheatsheet.md

0 · The mental model (read once)

PowerShell = Microsoft's shell + scripting language built on .NET (Microsoft's application framework/runtime), by Jeffrey Snover, 2006. Two flavors: Windows PowerShell 5.1 (ships with Windows, legacy) and PowerShell 7+ (pwsh, cross-platform, open source). A cmdlet ("command-let") = a native PowerShell command, always named Verb-Noun.
The big idea: objects flow through the pipe, not text. bash pipes pass bytes; you re-parse columns forever. PowerShell pipes pass typed .NET objects with named properties (data) and methods (actions) — you refer to fields by name, never by column position. 
bash:        ls -l | awk '{print $5, $9}' | sort -n
             (text → re-parse → hope the format holds)

PowerShell:  Get-ChildItem | Sort-Object Length | Select-Object Length, Name
             (objects → properties BY NAME → format-proof)
What you see on screen is just default formatting, applied at the very end. The data underneath is always the full object — pipe to Get-Member to see it.
Command grammar: Verb-Noun -Parameter Value. Verbs come from an approved list (Get-Verb shows ~100); nouns are singular; parameters are one dash + whole word (-Recurse). Case-insensitive. Memory hook: know the noun, derive the family — Get-Service, Start-Service, Stop-Service, Restart-Service. bash makes you memorize names; PowerShell lets you guess them.

The discovery trio — never memorize, always ask

Get-Command -Noun *process*"what commands exist for X?" (search by noun/verb)
Get-Help Get-Process -Examples"how do I use it?" (-Online opens web docs — works without admin)
something | Get-Member"what properties/methods does this object have?"

1 · bash → PowerShell translation

Built-in aliases keep bash muscle memory working at the prompt (Get-Alias ls shows what an alias points to). Fine interactively — use real cmdlet names in scripts.

Files & navigation

ls / dir → Get-ChildItem-Recurse ≈ -R; -Force ≈ -a (hidden)
cd → Set-Locationcd ~ works; Push-/Pop-Location ≈ pushd/popd
pwd → Get-Locationalias pwd
cat → Get-Content-Tail 20 -Wait ≈ tail -f
cp → Copy-Item-Recurse for dirs
mv → Move-Itemalso renames (or Rename-Item)
rm → Remove-Item-Recurse -Force ≈ rm -rf ⚠ same danger
mkdir → New-Item -ItemType Directoryalias mkdir takes just a path
touch → New-Item file.txtno direct touch; creates the file
find → gci -Recurse -Filter *.pyfilter while walking the tree

Text, processes, system

grep → Select-Stringalias sls; sls -Pattern "err" *.log
which → Get-Commandshows path AND type
ps → Get-Processobjects! (Get-Process chrome).CPU
kill → Stop-ProcessStop-Process -Name notepad
echo → Write-Outputjust typing "hello" also outputs
man → Get-Helpalias help pages the output
history → Get-HistoryCtrl+R reverse-search works too
wc -l → Measure-Object -Linegc f.txt | Measure-Object -Line
head/tail → Select-Object -First/-Last nor Get-Content -TotalCount / -Tail
env → Get-ChildItem env:see the $env: drive below
Trap: on 5.1, curl and wget are aliases for Invoke-WebRequest, NOT the real tools — flags like -O fail. PowerShell 7 removed those aliases.

2 · Pipeline workhorses: Where · Sort · Select

PowerShell's grep/sort/cut — but on object properties, not text columns.

The trio

Where-Object (where, ?)filter objects by condition ≈ grep on properties
Sort-Object (sort)sort by any property; -Descending
Select-Object (select)pick properties / first-last N ≈ cut + head

$_ = the current object in the pipe (no bash equivalent — bash pipes have no "current object"). Comparison operators are words: -eq -ne -gt -lt -like -match — because > means redirection in a shell. 10MB, 1GB are built-in literals.

Examples

# Top 5 memory-hungry processes
Get-Process | Sort-Object WS -Descending |
  Select-Object -First 5 Name, Id, WS

# Files over 10 MB, biggest first
Get-ChildItem -Recurse |
  Where-Object { $_.Length -gt 10MB } |
  Sort-Object Length -Descending |
  Select-Object Name, Length

# Stopped services mentioning 'update'
Get-Service | Where-Object {
  $_.Status -eq 'Stopped' -and $_.Name -like '*update*' }

3 · Variables & the $env: drive

bash vs PowerShell

x=5 → $x = 5always $, even when setting; spaces fine
"value is $x"interpolation in double quotes — same rule as bash
$HOME → $env:USERPROFILEenv vars live on the env: drive
export X=v → $env:X = "v"this session only
env → Get-ChildItem env:list all env vars

Variables hold typed objects: $x = Get-Date then $x.Year just works.

Drives & providers

Providers expose many data stores as navigable drives: filesystem (C:), env vars (env:), Windows registry (HKCU:), certificates (Cert:), variables (Variable:). Same cmdlets everywhere — Get-ChildItem env: lists env vars exactly like Get-ChildItem C:\ lists files. One grammar, many stores.

4 · Execution policy — why scripts won't run at work

Execution policy = a gate on running .ps1 script files. Microsoft's own words: NOT a security boundary — a seatbelt against accidental script execution. Interactive commands are never blocked.

The policies

Restrictedno scripts at all — 5.1 default on Windows clients, the usual culprit
RemoteSignedlocal scripts run; downloaded ones need a signature (PS 7 default, sane)
AllSignedeverything must be signed
Bypassnothing blocked, no warnings

Safe fixes (no admin needed)

Get-ExecutionPolicy -List   # policy at every scope

# your user account only:
Set-ExecutionPolicy RemoteSigned -Scope CurrentUser

# this window only, vanishes on close:
Set-ExecutionPolicy Bypass -Scope Process

# one-off, no policy change:
powershell -ExecutionPolicy Bypass -File .\script.ps1
Group Policy override: if IT sets policy via Group Policy (centralized Windows config management), the MachinePolicy/UserPolicy scopes win and Set-ExecutionPolicy errors out. -Scope Process or the -ExecutionPolicy Bypass launch flag is the remaining legitimate, logged route. If that's blocked too, it's a deliberate IT decision — ask, don't fight.

5 · $PROFILE — your .bashrc

Setup & reload

$PROFILE              # show the path
Test-Path $PROFILE    # exists?
New-Item $PROFILE -Force   # create (+ parent dirs)
notepad $PROFILE      # edit
. $PROFILE            # reload — dot-source, like bash 'source'

Gotchas

5.1 path: Documents\WindowsPowerShell\…_profile.ps1; PS 7: Documents\PowerShell\… — separate profiles, they don't share. The profile is itself a script → subject to execution policy: under Restricted it silently won't load.

6 · 5.1 vs 7, and the corporate cage

Which PowerShell am I in? ($PSVersionTable)

Windows PowerShell 5.1powershell.exe · ships with Windows · Windows-only (.NET Framework) · maintenance only · your corporate baseline
PowerShell 7+pwsh.exe · separate install · Windows/macOS/Linux · actively developed, open source

Side-by-side install, no conflict. Nearly everything on this sheet works in both.

Locked-down box: what still works

Constrained Language Mode (applied via AppLocker / WDAC, Windows application-control systems) blocks arbitrary .NET method calls, Add-Type, COM. Check: $ExecutionContext.SessionState.LanguageMode → FullLanguage or ConstrainedLanguage.

Still works: all built-in cmdlets, the pipeline, Where/Sort/Select, variables, $env:, file ops in your folders, CurrentUser/Process scopes, the full discovery trio. The cage blocks .NET escape hatches, not the shell — 90% of learning happens fine inside it.

7 · The whole loop in one example

$PSVersionTable.PSVersion                    # 1. which PowerShell am I in?
Set-Location ~\Documents                     # 2. cd
Get-ChildItem -Filter *.log                  # 3. ls *.log
Get-Content .\app.log -Tail 20               # 4. tail -20 app.log
Select-String -Pattern "ERROR" *.log |       # 5. grep ERROR *.log
  Select-Object Filename, LineNumber, Line   #    ...pick properties, not columns
Get-Process | Sort-Object CPU -Descending |  # 6. busiest processes
  Select-Object -First 5 Name, Id, CPU
Get-Process firefox | Get-Member             # 7. "what IS this object?" — build the habit

8 · Teardown — getting out

exitleave the session (bash: exit / ctrl d)
Ctrl Ckill the foreground command — same as bash
Stop-Process -Name notepadkill by name (bash: pkill notepad)
Stop-Process -Id 1234 -Forcekill by PID, forcefully (bash: kill -9 1234)
Remove-Item Env:MY_VARremove an env var — session only (bash: unset MY_VAR)
Remove-Variable myVarremove a PowerShell variable (no $ in the name here)
Set-ExecutionPolicy Restricted -Scope CurrentUserundo an execution-policy change (or Undefined to fall back to the wider scope)
Scope matters on the policy undo: revert in the same -Scope you changed (CurrentUser vs Process). -Scope Process changes die with the window anyway.
