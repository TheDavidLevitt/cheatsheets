0 · The layer template — every architecture is the same sandwich

Almost every system on GCP (and on the exam) is assembled from the same five layers, top to bottom. When a scenario question hits, place each named requirement on a layer and the answer usually falls out.

LayerJobGCP parts
1 · Namea human-friendly, permanent addressCloud DNS (A record → static IP or load balancer)
2 · Front doorone stable entry point; spread traffic, terminate TLS, cacheApplication LB (L7, HTTP/S, global) · Passthrough NLB (L4, raw TCP/UDP) · Cloud CDN
3 · Computerun the codeCloud Run (serverless containers) · GKE (Kubernetes) · MIG of VMs · App Engine
4 · Statestore the data (see §3)Cloud SQL · Firestore · Bigtable · GCS · BigQuery · Memorystore (cache)
5 · Async gluedecouple parts that run at different speedsPub/Sub (message bus) · Cloud Tasks (targeted task queue) · Eventarc (event routing)
Reflex: requirements like "users visit a URL" = layers 1–2; "process/serve" = layer 3; "store/query" = layer 4; "spiky, bursty, background, decouple" = layer 5. The wrong answers in a question are usually parts from the right layer family but the wrong member, or a layer-4 storage product offered for a layer-5 messaging job.

1 · Eight canonical architectures

① Static website / single-page app

Cloud DNS ─► Application LB (+ Cloud CDN)
                  └─► GCS bucket (HTML/JS/CSS files)

No servers at all: the site is just files in a Cloud Storage bucket; the load balancer gives it HTTPS + a stable address; the CDN (Content Delivery Network — copies cached at Google's edge locations worldwide) makes it fast everywhere.

Trigger words: "static content", "no backend logic", "cheapest way to host a website".

② Classic three-tier web app

Cloud DNS ─► Application LB
                  └─► MIG of VMs  (or Cloud Run)
                        ├─► Memorystore (Redis cache)
                        └─► Cloud SQL (relational DB)

The "online store" shape: stateless web/app servers in a MIG (Managed Instance Group — a pool of identical autoscaling VMs from one template) so any instance can die and be replaced; all state lives one layer down in Cloud SQL; Memorystore caches hot reads.

Trigger words: "web app + relational database", "session data in a cache", "autoscale web tier".

③ Containerized microservices

Cloud Build ─► Artifact Registry (images)
                       └─► GKE Autopilot
                            ├─ Service "frontend" ─► pods
                            └─ Service "backend"  ─► pods

Many small services, each in its own container. Images are stored in Artifact Registry; pods are ephemeral, so each microservice hides behind a Kubernetes Service (stable virtual IP + DNS name). Choose Autopilot when the question stresses minimal ops / pay-per-pod; choose Cloud Run instead when they don't actually need Kubernetes.

Trigger words: "microservices", "containers", "teams deploy independently".

④ IoT / streaming telemetry

devices ─► Pub/Sub ─► Dataflow ─► BigQuery
            (buffer)   (transform   (store +
                        in flight)   SQL analysis)

The single most-tested pipeline. Millions of sensors publish readings; Pub/Sub absorbs the bursts (nothing dropped); Dataflow cleans/deduplicates/windows the stream in flight; BigQuery both stores the result and answers SQL over it. Swap the tail for Bigtable when the need is millisecond lookups of time-series rows rather than analytics.

Trigger words: "sensor/event stream", "ingest spikes", "transform then analyze".

⑤ Batch ETL / big-data crunching

GCS (raw files) ─► Dataflow ─► BigQuery   (new pipeline)
GCS (raw files) ─► Dataproc ─► GCS/BQ     (existing Spark/Hadoop jobs)

ETL = Extract, Transform, Load: pull raw data in, reshape it, land it in a warehouse. New pipelines → Dataflow. A company migrating jobs it already runs on a Spark/Hadoop cluster → Dataproc (see §2 for the full distinction).

Trigger words: "nightly job", "process files in a bucket", "migrate our cluster".

⑥ Event-driven file processing

upload ─► GCS bucket ─(event)─► Cloud Run function
                                     └─► thumbnail/result
                                          back to GCS / Firestore

Nothing runs until something happens: a file landing in a bucket fires an event that invokes a small serverless function (resize the image, parse the invoice, scan the upload). Costs nothing while idle.

Trigger words: "whenever a file is uploaded", "react to events", "no idle cost".

⑦ CI/CD pipeline

git push ─► Cloud Build (test + build image)
                 └─► Artifact Registry
                       └─► deploy to Cloud Run / GKE

CI/CD = Continuous Integration / Continuous Delivery: every code push is automatically tested, built into a container image, stored, and rolled out. Cloud Build is the trigger-driven build engine; Artifact Registry is the image shelf; Cloud Run/GKE traffic-splitting gives canary releases (send 5% of traffic to the new version first).

Trigger words: "on every commit", "automate build and deploy", "canary/rollback".

⑧ Hybrid: on-prem + cloud

on-prem network ═ VPN / Interconnect ═ VPC
        (Cloud Router exchanges routes via BGP)
   private VMs ─► Private Google Access ─► GCS/APIs

A company keeps its datacenter and extends into GCP. Cloud VPN (encrypted over internet) or Dedicated/Partner Interconnect (private physical link) joins the networks; Cloud Router uses BGP (Border Gateway Protocol — routers advertising "these addresses live behind me") so routes update automatically; Private Google Access lets internal-only VMs reach Google services without touching the public internet.

Trigger words: "connect our datacenter", "no public IPs", "dynamic routing".

2 · Dataproc vs Dataflow — anchored, no Hadoop background assumed

The anchor: rented factory vs conveyor-belt service.
Dataproc = Google rents you a factory you already know how to run. Hadoop and Spark are 2010s-era open-source frameworks for crunching huge datasets on a cluster (a fleet of machines working together). Thousands of companies wrote jobs for them. Dataproc spins that same cluster up in ~90 seconds so those existing jobs run unchanged — but it is still a cluster: you choose its size and turn it off.

Dataflow = Google runs a conveyor belt for you. You only describe the processing steps (dedupe → average over 5-minute windows → write out); Google builds, staffs, and auto-scales the machinery. No cluster exists for you to manage. It handles both streaming (items processed as they arrive) and batch (truckloads at a time). The pipeline language is Apache Beam — trivia, not the point.
Question says…Answer
"existing Hadoop/Spark jobs", "migrate our cluster", "minimal code changes"Dataproc — lift-and-shift the factory
"streaming", "real-time", "windowed/in-flight transforms", "serverless data processing", "no cluster management"Dataflow — the conveyor belt
"ad-hoc SQL on huge data", "analysts query it"BigQuery — neither; that's the warehouse at the end of the pipe
Trap: Dataflow is not "managed Hadoop" — that's Dataproc. If the scenario never mentions an existing cluster or Spark/Hadoop by name, the data-processing answer is almost always Dataflow.

3 · Where data lives — one line each

Half the wrong answers in architecture questions are the right kind of component but the wrong store. One sentence per store, with its tell.

StoreWhat it isTell
Cloud Storage (GCS)objects/files of any size in buckets"files, images, backups, archives"
Cloud SQLmanaged MySQL/Postgres/SQL Server — transactional rows, one region"relational, existing app, ACID"
Spannerrelational like Cloud SQL but globally distributed and horizontally scalable"global + relational + 99.999%"
Firestoreserverless document DB (JSON-like) for app/mobile state, live sync"mobile/web app data, offline sync"
Bigtablewide-column NoSQL for huge key-addressed datasets at millisecond latency"time-series/IoT at TB–PB scale, low-latency reads" — storage, never the answer to "distribute work" (that's Pub/Sub)
BigQuerystores the data and runs SQL analytics on it — a warehouse, not just a query engine"analyze with SQL, dashboards, petabytes"
Memorystoremanaged Redis/Memcached, in-RAM cache"sub-millisecond, session/cache"
