Data Formats Cheat Sheet

Data serialization: JSON · YAML · TOML · CSV · XML · JSONL · Parquet · started 2026-06-06 · twin of Tools/formats_cheatsheet.md

Serialization = turning a program's in-memory data (a dict, a list, an object) into a flat sequence of text or bytes that can be saved or sent — and deserialization = reading it back. A data format is the agreed grammar for that flat form. JSON (JavaScript Object Notation), the lingua franca of data exchange, is the one to learn first; Douglas Crockford "discovered" it (his word) in 2001 by lifting JavaScript's object-literal syntax — which is why it looks like a Python dict but isn't quite one.

0 · The mental model: writing data down

A data format answers exactly one question: how do you write structured data down as text or bytes so a program can read it back? Formats are about representation, not transport.

  in memory                  serialize                 on disk / on the wire
  ─────────                  ─────────►                ─────────────────────
  {"name": "Ada",                                      {"name": "Ada", "age": 36}
   "age": 36}                ◄─────────                (a JSON file, a YAML file,
   (a Python dict)           deserialize                a row in a CSV, …)
  • Text formats (JSON, YAML, TOML, CSV, XML) are human-readable — open them in any editor. Easy to debug, larger, slower to parse.
  • Binary formats (Parquet, Pickle) are bytes, not characters — unreadable by eye but compact and fast. Use them for scale.
Sending these over the web? The format is just the body; the protocol that carries it (methods, status codes, REST, auth) lives in http_cheatsheet.md. This sheet is only about the data itself.

1 · JSON — the whole syntax in one example

JSON has exactly six types: object, array, string, number, boolean (true/false), and null. That's the entire language.

{
  "name": "Ada",
  "age": 36,
  "height_m": 1.78,
  "is_active": true,
  "nickname": null,
  "languages": ["en", "fr"],
  "address": {
    "city": "Lisbon",
    "country": "PT"
  }
}

Annotations

  • { } = object — unordered key/value pairs. Keys MUST be double-quoted strings (single quotes are invalid — this trips up Python users constantly).
  • [ ] = array — ordered list, any mix of types.
  • Numbers: no distinction between int and float; no NaN/Infinity; no leading zeros (042 invalid).
  • No trailing commas — ["a", "b",] is a syntax error (unlike Python).
  • No comments — by design; Crockford removed them to stop people abusing them for parser directives. (Workaround people use: a "_comment" key.)
  • Encoding is UTF-8; strings use \", \n, \uXXXX escapes.

2 · YAML — JSON's indentation-based cousin

YAML = "YAML Ain't Markup Language" — a recursive acronym (the name contains itself), a Unix in-joke in the style of GNU ("GNU's Not Unix"). Originally it stood for "Yet Another Markup Language" (2001); renamed to stress it's for data, not document markup.
Key fact: YAML is a superset of JSON — every valid JSON file is valid YAML. YAML adds indentation-based structure, comments, and quote-less strings on top. 
# Comments work! (unlike JSON)
name: Ada
languages:        # a list — each item gets "- "
  - en
  - fr
address:          # nesting via indentation (2 spaces is the convention)
  city: Lisbon
inline_also_works: {city: Lisbon}   # JSON syntax is legal YAML

YAML gotchas (it's too clever)

Tabs are forbiddenindentation must be spaces; a tab is a hard syntax error
The Norway problemunquoted no parses as boolean false (so a country list [GB, FR, NO] corrupts Norway). Same for yes/on/off; YAML 1.2 fixed it but many parsers are 1.1. Quote it: "NO"
Unquoted 3.10parses as the float 3.1 — Python version numbers in CI must be quoted: "3.10"
09:30sexagesimal/time parsing surprises in YAML 1.1
Rule of thumbwhen in doubt, quote your strings

3 · TOML, CSV/TSV — and XML in two lines

TOML (Tom's Obvious Minimal Language — Tom Preston-Werner, GitHub co-founder, 2013) = INI files with a real spec. Deliberately flat and unambiguous where YAML is clever. You meet it as pyproject.toml, Python's standard project config: 
[project]
name = "myapp"
requires-python = ">=3.11"
dependencies = ["requests", "pandas"]

[tool.ruff]          # [section] headers = "tables"
line-length = 100    # strings always quoted, so no Norway problem
CSV (Comma-Separated Values) = the oldest trick in the book: one record per line, fields split on commas. TSV is the same with tabs (no comma-vs-decimal ambiguity). No real standard (RFC 4180 is a sketch) — quoting, encodings, and delimiters (Excel in French locales uses ;!) vary. Read it with pandas.read_csv() or stdlib csv.DictReader, never with .split(",") — quoted fields can contain commas.
XML (eXtensible Markup Language, 1998) = the verbose tagged format (<user><name>Ada</name></user>) that dominated the SOAP-era of web services (SOAP = Simple Object Access Protocol, XML-based RPC). You'll meet it in legacy enterprise feeds, RSS, and SVG — but new formats are JSON. Largely legacy.

4 · The bigger formats you'll meet

JSONL (JSON Lines)one complete JSON object per line — streamable, appendable, the standard for LLM training data and batch APIs. The whole file is not valid JSON; each line is.
Parquetbinary columnar format (stores column-by-column, not row-by-row) → great compression + fast "give me just these 2 columns" reads. The data-engineering default; pandas.read_parquet().
Markdownwhat the source file is written in — plain-text formatting conventions (# heading, **bold**, [link](url)) by John Gruber, 2004; "markdown" is a pun on "markup".
PicklePython-only binary serialization — never load one from an untrusted source (arbitrary code execution).
Which to reach for:
  • Config a human edits → TOML or YAML (comments, readable).
  • Data exchanged between programs / APIs → JSON (universal).
  • A long stream of records you append to → JSONL.
  • A spreadsheet export / quick tabular dump → CSV/TSV.
  • Millions of rows for analytics → Parquet (columnar, compressed, fast).

5 · Sending these over the web

A data format is the payload; it says nothing about how the bytes get from A to B. That's a separate concern:

QuestionLives in
How is the data written down?this sheet (JSON, YAML, …)
How does the request/response travel?http_cheatsheet.md (HTTP, status codes)
How should I name URLs and verbs?http_cheatsheet.md (REST)
How does the server know it's me?http_cheatsheet.md / authentication_cheatsheet.md
In practice: a JSON object becomes the body of an HTTP request, sent with a Content-Type: application/json header. Format and protocol are orthogonal — you could send the same JSON over HTTP, a file, or a message queue.

6 · Teardown / hygiene

A large parsed file in memoryfor big CSV/JSON, stream it (csv.reader, ijson, or pandas chunks) rather than loading the whole thing — formats don't free memory for you.
A Pickle from elsewheredon't pickle.load() untrusted data — it can execute arbitrary code on load. Prefer JSON for anything crossing a trust boundary.
A YAML config you editedrun it through a linter / yaml.safe_load (never plain yaml.load, which can instantiate arbitrary Python objects).
Stale *.parquet / cache filesdelete derived columnar caches when the source changes — they won't notice and you'll read stale data.
