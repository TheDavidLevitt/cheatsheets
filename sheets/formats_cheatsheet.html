Data Formats & API Concepts Cheat Sheet

JSON · YAML · HTTP · REST · auth · webhooks · started 2026-06-06 · twin of Tools/formats_cheatsheet.md

JSON (JavaScript Object Notation) = the lingua franca of data exchange. Douglas Crockford "discovered" it (his word) in 2001 by lifting the object-literal syntax straight out of JavaScript — which is why it looks like a Python dict but isn't quite one. API (Application Programming Interface) = a contract for how one program talks to another; on the web that contract is usually HTTP requests carrying JSON. HTTP (HyperText Transfer Protocol) = the request/response protocol the web runs on (Tim Berners-Lee, 1991). REST (REpresentational State Transfer) = an architectural style for APIs, named in Roy Fielding's 2000 PhD thesis (he co-authored the HTTP spec — REST is HTTP used the way it was designed).

1 · JSON — the whole syntax in one example

JSON has exactly six types: object, array, string, number, boolean (true/false), and null. That's the entire language.

{
  "name": "Ada",
  "age": 36,
  "height_m": 1.78,
  "is_active": true,
  "nickname": null,
  "languages": ["en", "fr"],
  "address": {
    "city": "Lisbon",
    "country": "QA"
  }
}

Annotations

  • { } = object — unordered key/value pairs. Keys MUST be double-quoted strings (single quotes are invalid — this trips up Python users constantly).
  • [ ] = array — ordered list, any mix of types.
  • Numbers: no distinction between int and float; no NaN/Infinity; no leading zeros (042 invalid).
  • No trailing commas — ["a", "b",] is a syntax error (unlike Python).
  • No comments — by design; Crockford removed them to stop people abusing them for parser directives. (Workaround people use: a "_comment" key.)
  • Encoding is UTF-8; strings use \", \n, \uXXXX escapes.

2 · YAML — JSON's indentation-based cousin

YAML = "YAML Ain't Markup Language" — a recursive acronym (the name contains itself), a Unix in-joke in the style of GNU ("GNU's Not Unix"). Originally it stood for "Yet Another Markup Language" (2001); renamed to stress it's for data, not document markup.
Key fact: YAML is a superset of JSON — every valid JSON file is valid YAML. YAML adds indentation-based structure, comments, and quote-less strings on top. 
# Comments work! (unlike JSON)
name: Ada
languages:        # a list — each item gets "- "
  - en
  - fr
address:          # nesting via indentation (2 spaces is the convention)
  city: Lisbon
inline_also_works: {city: Lisbon}   # JSON syntax is legal YAML

Where you meet it

GitHub Actions.github/workflows/*.yml — CI pipelines
condaenvironment.yml — env name + dependency list
Docker Composedocker-compose.yml
Kubernetesbasically everything
MkDocs / many toolsmkdocs.yml, config files generally

YAML gotchas (it's too clever)

Tabs are forbiddenindentation must be spaces; a tab is a hard syntax error
The Norway problemunquoted no parses as boolean false (so a country list [GB, FR, NO] corrupts Norway). Same for yes/on/off; YAML 1.2 fixed it but many parsers are 1.1. Quote it: "NO"
Unquoted 3.10parses as the float 3.1 — Python version numbers in CI must be quoted: "3.10"
09:30sexagesimal/time parsing surprises in YAML 1.1
Rule of thumbwhen in doubt, quote your strings

3 · TOML, CSV — and XML in two lines

TOML (Tom's Obvious Minimal Language — Tom Preston-Werner, GitHub co-founder, 2013) = INI files with a real spec. Deliberately flat and unambiguous where YAML is clever. You meet it as pyproject.toml, Python's standard project config: 
[project]
name = "myapp"
requires-python = ">=3.11"
dependencies = ["requests", "pandas"]

[tool.ruff]          # [section] headers = "tables"
line-length = 100    # strings always quoted, so no Norway problem
CSV (Comma-Separated Values) = the oldest trick in the book: one record per line, fields split on commas. No real standard (RFC 4180 is a sketch) — quoting, encodings, and delimiters (Excel in French locales uses ;!) vary. Read it with pandas.read_csv() or stdlib csv.DictReader, never with .split(",") — quoted fields can contain commas.
XML (eXtensible Markup Language, 1998) = the verbose tagged format (<user><name>Ada</name></user>) that dominated the SOAP-era of web services (SOAP = Simple Object Access Protocol, XML-based RPC). You'll meet it in legacy enterprise APIs, RSS feeds, and SVG — but new APIs are JSON.

4 · HTTP fundamentals

Anatomy of a request — the response mirrors it: status line (HTTP/1.1 201 Created), headers, blank line, body.

POST /v1/users HTTP/1.1            ← method + path + protocol version
Host: api.example.com              ← headers: key: value lines
Authorization: Bearer eyJhbGc...
Content-Type: application/json
                                   ← blank line separates headers from body
{"name": "Ada"}                  ← body (only for POST/PUT/PATCH)

The verb table

VerbMeaningBody?Idempotent?Analogy
GETread a resourcenoyesSELECT
POSTcreate / trigger an actionyesno (repeat = duplicate!)INSERT
PUTreplace a resource entirelyyesyesoverwrite the file
PATCHmodify part of a resourceyesusuallyedit one field
DELETEremove a resourcenoyesrm
Idempotent = repeating the request has the same effect as doing it once — safe to retry. That's why retry logic retries GET freely but must be careful with POST.

Status code families — first digit tells the story

FamilyMeaningFamous members
2xxsuccess200 OK · 201 Created (after a POST) · 204 No Content
3xxredirection301 Moved Permanently (the URL changed — update your bookmark) · 302 Found (temporary)
4xxyour fault (client error)400 Bad Request · 401 Unauthorized (really means unauthenticated — who are you?) · 403 Forbidden (I know who you are, and no) · 404 Not Found · 429 Too Many Requests (rate-limited — back off)
5xxtheir fault (server error)500 Internal Server Error · 502 Bad Gateway · 503 Service Unavailable (overloaded/down — retry later)
Memory hook for 401 vs 403: 401 = "log in first" (missing/bad credentials), 403 = "logged in, still no" (insufficient permissions).

5 · REST — what "RESTful" actually means

REST is not a protocol or library — it's a set of conventions (from Fielding's thesis) for designing HTTP APIs:

  1. Resources are nouns in URLs — /users/42/orders, never /getUserOrders?id=42. The thing, not the action.
  2. Verbs come from HTTP — the method says what you're doing to the noun (GET reads it, DELETE removes it).
  3. Stateless — every request carries everything the server needs (auth included); the server keeps no session memory between requests. That's what makes REST APIs scale — any server replica can answer any request.
  4. Responses are representations (usually JSON) of resource state — hence the name: you transfer a representation of the resource's state.

URL anatomy

https://api.example.com:443/v1/users/42?fields=name,email&limit=10#section
└─┬─┘   └───────┬───────┘└┬┘└────┬─────┘└──────────┬───────────┘ └──┬──┘
scheme       host        port   path          query params       fragment
(protocol)  (which server)     (which        (?key=value pairs   (client-only,
                                resource)     joined by &)        never sent)

Worked example — same call, curl and Python

# GET with a query param and auth header
curl -s "https://api.example.com/v1/users?limit=2" \
     -H "Authorization: Bearer $TOKEN" | jq .

# POST: create a resource (note 201 Created on success)
curl -s -X POST "https://api.example.com/v1/users" \
     -H "Authorization: Bearer $TOKEN" \
     -H "Content-Type: application/json" \
     -d '{"name": "Ada", "city": "Lisbon"}'
import requests

headers = {"Authorization": f"Bearer {token}"}

r = requests.get("https://api.example.com/v1/users",
                 params={"limit": 2}, headers=headers)   # params → ?limit=2
r.raise_for_status()          # raises on 4xx/5xx — always do this
users = r.json()              # parsed JSON → list/dict

r = requests.post("https://api.example.com/v1/users",
                  json={"name": "Ada", "city": "Lisbon"},  # json= sets body
                  headers=headers)                         # AND Content-Type
print(r.status_code)          # 201
json= (auto-serializes + sets the header) vs data= (raw form/bytes) is the classic requests distinction.

6 · Auth patterns in one table

PatternLooks likeNotes
API key in headerx-api-key: sk-abc123…simplest; a static secret identifying the app. (Anthropic's API uses exactly this.)
Bearer tokenAuthorization: Bearer eyJhbGc…"whoever bears this token is authorized." Often a JWT (JSON Web Token — signed JSON payload); usually short-lived
Basic authAuthorization: Basic dXNlcjpwYXNzbase64 of user:password — old, only over HTTPS, rare in modern APIs
OAuth 2.0a flow, not a headerin one sentence: a choreography where you grant App A limited access to your data in Service B without giving A your B password — it ends with A holding a Bearer token
Universal rule: secrets live in environment variables (os.environ["API_KEY"]), never in code or Git.

7 · Webhooks vs polling

PollingWebhook
Directionyou ask repeatedly: "anything new?"they call you when something happens
Mechanicsloop of GET requests on a scheduleyou register a URL; the service POSTs JSON to it on each event
Latency vs costlatency = poll interval; wasteful (most polls return nothing)near-instant, no wasted calls
Catchrate limits (hello, 429)you must run a publicly reachable server, and verify the payload signature (anyone can POST to a URL)
Memory hook: polling = checking the mailbox every five minutes; webhook = the doorbell.

8 · Other formats you'll meet

JSONL (JSON Lines)one complete JSON object per line — streamable, appendable, the standard for LLM training data and batch APIs. The whole file is not valid JSON; each line is.
Parquetbinary columnar format (stores column-by-column, not row-by-row) → great compression + fast "give me just these 2 columns" reads. The data-engineering default; pandas.read_parquet().
Markdownwhat the source file is written in — plain-text formatting conventions (# heading, **bold**, [link](url)) by John Gruber, 2004; "markdown" is a pun on "markup".
PicklePython-only binary serialization — never load one from an untrusted source (arbitrary code execution).

9 · Teardown / hygiene

Done with an API keyrevoke it in the provider's console — deleting it from your code is not revocation; a leaked key works until the server side kills it
Done with a Bearer token / session"log out" properly = the server invalidates the session/token; merely deleting the cookie only makes your browser forget it — the credential may still be valid if stolen
Done with a webhook endpointunsubscribe/disable it in the sending service — otherwise it keeps POSTing events to a URL you no longer control (a re-registered domain could receive your data)
Done with a streaming connectionclose it explicitly — with requests.get(url, stream=True) as r: (or r.close()) — abandoned streams hold sockets and server resources open
