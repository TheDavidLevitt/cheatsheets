HTML Cheat Sheet

Reading & writing web pages: structure · tags · forms · DevTools · started 2026-06-06 · twin of Tools/html_cheatsheet.md

0 · The mental model (read once)

HTML = HyperText Markup Language: describes the structure and content of a page. "HyperText" = text containing links (Tim Berners-Lee, 1989). "Markup" = annotations wrapped around content — HTML descends from SGML (Standard Generalized Markup Language, 1986; from IBM's GML, 1969), which is why tags look the way they do: publishers "marked up" manuscripts with marginal instructions; HTML does it with <tags>.
Three languages, one house: 
HTML            CSS              JavaScript
structure       style            behavior
(walls, rooms,  (paint,          (electricity,
 doors)          furniture,       plumbing —
                 layout)          things that DO stuff)
HTML = what's on the page and what it means. CSS (Cascading Style Sheets) = what it looks like — lives in <style> blocks or .css files. JavaScript (JS) = what it does — lives in <script> blocks or .js files. Hunting for content? Skip everything inside <style> and <script>.
Anatomy of an element: 
 opening tag                          closing tag
 ┌────┴─────────────────────┐         ┌──┴──┐
 <a href="https://mdn.io" >  Read me  </a>
  │  └──────┬────────────┘   └──┬──┘    │
 tag     attribute            content   note the slash
 name    (name="value")
Attributes live only in the opening tag. A few elements are void (no content, no closing tag): <br> <hr> <img> <input> <meta>. Elements nest like Russian dolls — last opened, first closed. Convention: lowercase tag names.

1 · Minimal valid document, annotated

<!DOCTYPE html>                       <!-- 1: "modern HTML5, standards mode" -->
<html lang="en">                      <!-- 2: root element; lang helps a11y/SEO -->
<head>                                <!-- 3: metadata ABOUT the page, never rendered -->
  <meta charset="utf-8">              <!-- 4: encoding; first, or é/ü garble -->
  <meta name="viewport"
        content="width=device-width, initial-scale=1">  <!-- 5: don't look tiny on phones -->
  <title>Page title</title>           <!-- 6: tab text + search-result headline -->
</head>
<body>                                <!-- 7: everything the user actually sees -->
  <h1>Hello</h1>
</body>
</html>

Historic note: old doctypes were long SGML document-type references; HTML5 (2014) shrank the incantation to <!DOCTYPE html>.

2 · Text tags

Headings & paragraphs

<h1>…<h6>headings, level 1–6. One h1 per page; it's an outline, not a font-size picker
<p>paragraph — the default unit of body text
<br>line break (void) — for poems/addresses, not spacing
<hr>horizontal rule (void) — thematic break
<code> / <pre>inline code / preformatted block — the only place whitespace survives (elsewhere runs of spaces collapse to one)

Bold & italic — semantic vs cosmetic

<strong>strong importance (renders bold) — says "this matters"
<em>emphasis (renders italic) — says "stress this word"
<b> / <i>appearance only, no meaning (ship names, Latin terms). Prefer strong/em when you mean importance

3 · Lists, links, images

Lists & images

<ul><li>…</li></ul>unordered list (bullets) of list items
<ol><li>…</li></ol>ordered list (numbered)
<img src="cat.png" alt="A sleeping cat">image (void). alt shows on failure + read by screen readers — write a description, not a filename

Links

<a href="url">text</a>a = anchor, href = hypertext reference (1990s coinages that stuck)
<a href="#section-id">jump to element with that id on the same page
target="_blank" rel="noopener"open in new tab — always pair them; without rel the new page gets a handle back to yours

4 · Tables

For tabular data only — table-based page layout is a dead 1990s hack; CSS does layout now.

Structure

<table>
  <tr> <th>Cmd</th>     <th>Meaning</th> </tr>  <!-- header row -->
  <tr> <td>git add</td> <td>stage</td>   </tr>  <!-- data row -->
</table>

Tags

<tr>table row
<th>table header cell (bold, centered, announced as header)
<td>table data cell
colspan / rowspancell attribute: span across columns / rows

5 · Semantic layout tags (the page skeleton)

HTML5 named the regions every page already had, replacing soups of anonymous <div>s. They render identically — the value is meaning (for code readers, screen readers, search engines).

Regions

<header>top-of-page/section intro: logo, title
<nav>navigation links block
<main>the unique main content — one per page
<section>thematic grouping, usually with its own heading
<article>self-contained piece (post, card, comment)
<aside>tangential: sidebar, pull-quote
<footer>copyright, links, sources

Last resorts

<div>generic block with no meaning — use only when nothing semantic fits
<span>inline equivalent: meaningless wrapper for styling a few words

Reading tip: in unfamiliar HTML, scan for <main>, <nav>, <article> first — they're the file's table of contents.

6 · Forms basics

Pattern

<form action="/subscribe" method="post">
  <label for="email">Email</label>
  <input type="email" id="email" name="email" required>
  <button type="submit">Sign up</button>
</form>

Pieces

<form action method>where to send data; get (in URL) vs post (in body)
<input type="…">the workhorse (void). type: text, email, password, number, checkbox, radio, date, file, range…
<label for="x">caption tied to input with id="x" — clicking it focuses the input. Always pair
name="…"the submission key — no name, no data sent
<textarea>, <select><option>multi-line box; dropdown

7 · class vs id · inline vs block

The CSS/JS hooks

class="card warn"reusable, multiple per element, shared by many elements. CSS: .card
id="pricing"unique, one per page. Anchors (#pricing), JS lookups. CSS: #pricing

Rule of thumb: style with classes, navigate/script with ids.

Inline vs block

Blocknew line, full width: h1–h6 p div ul table + section tags
Inlineflows within text, content-width: a strong em span code img

width on an inline element does nothing — CSS display overrides either default; that's where layout work begins.

8 · Entities, comments, collapsibles

Entities (escapes)

&lt; &gt;literal < > — bare angle brackets start tags (less/greater than)
&amp;literal & — it starts entities itself
&nbsp;non-breaking space — won't wrap or collapse (10&nbsp;km)
&copy; &eacute;© é — with UTF-8 just type the real char; entities remain for the three escape cases above

Comments & native collapsibles

<!-- comment -->invisible in page, visible in source. No nesting
<details><summary>Title</summary>…</details>click-to-expand, zero JS; add open to start expanded. Our own cheat sheets use this

9 · Reading the DevTools inspector

Right-click any page → Inspect (or Cmd+Opt+I). The Elements panel shows the live DOM (Document Object Model) tree — what the browser built, not the raw source (Cmd+Opt+U shows that). Differences are normal: JS adds nodes, and the browser repairs malformed HTML.
Click the arrow-in-box icon (top-left of DevTools), hover the page — it highlights elements and jumps the tree to them: the fastest answer to "what tag is this thing?" The Styles pane shows every CSS rule hitting the selected element (struck-through = lost the cascade). Double-click anything in the tree to edit it live — vanishes on reload, perfect for safe experiments.

10 · Teardown / removing things

Comment out vs delete<!-- … --> is reversible and great while experimenting — but commented markup still ships and is visible in View Source. Delete for real removals; Git remembers anyway
Remove an element in DevToolsselect it in Elements ▸ press Delete (or right-click ▸ Delete element). Temporary — you edited the DOM, not the file; refresh restores everything
Unlink a stylesheet / scriptdelete (or comment out) the <link rel="stylesheet" href="…"> or <script src="…"> tag in <head> — the page loads without that layer
Graceful degradationremove CSS → plain but perfectly readable document. Remove JS → links, forms, <details> still work if the HTML is semantic; a div-soup page built by JS shows nothing. That resilience is the payoff of semantic HTML
