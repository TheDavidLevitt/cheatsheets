4 · Jupyter — reading cell execution state

The [N] at the left of each cell is the execution counter for the current kernel session.

Indicators

[ ]never run in this kernel session
[*]currently running or queued
[N]last ran as execution #N this session
dim [N] + gutter dotedited since it last ran — output is stale

Does a cell need rerunning?

1. Read execution numbers as replay order — if cell 3 is [7] and cell 5 is [3], you ran them out of order. 2. Watch for dim-number + gutter-dot (modified after running). 3. When in doubt: Restart Kernel and Run All — the notebook's clean make.

Concept — notebook reproducibility: a kernel is a long-running process; each cell mutates its namespace, and saved outputs reflect whatever state the kernel was in when each cell ran — not necessarily top-to-bottom. "Works on my machine" in notebooks almost always traces to this. Restart-and-run-all is the cure; run it before committing so counts go [1],[2],[3]….

Defaults vary by extension/version — check the extension's own keybindings.

Customizing shortcuts

Open the Keyboard Shortcuts editor: ⌘K ⌘S / Ctrl+K Ctrl+S, or Palette → "Preferences: Open Keyboard Shortcuts." Searchable by command name + binding; re-bind, add an alternate, or remove.

Advanced: bindings live in keybindings.json (Palette → "Open Keyboard Shortcuts (JSON)") — same config pattern as settings.json.
