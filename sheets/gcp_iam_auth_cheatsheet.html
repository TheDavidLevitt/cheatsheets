0 · The IAM spine (read once)

IAM (Identity and Access Management) answers exactly one question: who can do what on which resource. Every concept in this sheet is a piece of that sentence: who = principal, can do what = role (a bundle of permissions), on which resource = the resource the policy is attached to.
The chain, bottom-up: 
permission            compute.instances.get        (atomic; you can never grant one directly)
   └─ bundled into ROLE      roles/compute.viewer    (smallest grantable unit)
        └─ tied to PRINCIPAL by a BINDING            (role + members [+ optional condition])
             └─ bindings collected in an ALLOW POLICY
                  └─ policy ATTACHED to a resource
                       └─ INHERITED down:  org → folder → project → resource

Permission grammar: service.resource.verb

Permissions read like a sentence in reverse — service, then the resource type within it, then the action:

compute.instances.getCompute Engine · VM instances · read one
storage.objects.listCloud Storage · objects · enumerate
iam.serviceAccounts.actAsIAM · service accounts · act as (the attach gate — §3)
resourcemanager.projects.deleteResource Manager · projects · delete (scary verbs exist)

Permissions usually map 1:1 to REST API methods — compute.instances.get guards the instances.get API call. That's the etymology: permissions are API methods wearing IAM badges.

Inheritance: the policy waterfall

Effective access at a resource = the policy on it union every policy above it. Inheritance is additive only — a child policy can add access, never subtract what a parent granted. (Subtracting is the job of deny policies and org policies, beyond ACE basics.)

Organization        ← grant here = everywhere
  └─ Folder         ← grant here = whole department
       └─ Project   ← the everyday grant level
            └─ Resource (bucket, VM, topic…)

Mnemonic: IAM flows like water — downhill only, and you can't un-pour it from below.

One binding, in the wild (what gcloud projects get-iam-policy shows): 
bindings:
- role: roles/storage.objectViewer
  members:
  - group:data-readers@example.com
  condition:                              # optional
    title: business-hours
    expression: request.time.getHours("Europe/Paris") < 18
A condition is a CEL (Common Expression Language) test that must also pass — time windows, resource-name prefixes. The policy holding the bindings is officially the allow policy (older docs just say "IAM policy").

1 · Principals — the "who"

A principal (older term: member) is any identity a binding can name. Each has a typed prefix in the policy:

PrincipalExample & notes
user:user:alex@example.com — one human Google account
group:group:devs@example.com — a Google Group. Best practice: grant to groups, not users — membership changes become an admin click instead of a policy edit on every project.
serviceAccount:serviceAccount:app@example-project-123.iam.gserviceaccount.com — a non-human identity (§3)
domain:domain:example.com — everyone in a Cloud Identity / Workspace domain
allAuthenticatedUsersanyone on Earth with any Google account — barely better than public
allUsersliterally everyone, no sign-in — this is how you make a bucket public (deliberately or catastrophically)
Exam + real-life trap: allUsers and allAuthenticatedUsers look harmless in a list of principal types. They are the two ways data leaks. Org policy iam.allowedPolicyMemberDomains (domain-restricted sharing) exists specifically to ban them.

2 · Roles — the "can do what"

Iron rule: you can never grant a single permission directly. The smallest grantable unit is a role. Want exactly one permission? Build a custom role containing exactly that one.

Basic roles — the legacy three

roles/viewerread everything in the project
roles/editorviewer + modify almost everything
roles/ownereditor + manage IAM & billing link

Pre-2017 "primitive roles", renamed basic. They span thousands of permissions across every service — the opposite of least privilege. Fine for a solo sandbox; an audit finding everywhere else.

Predefined roles — the everyday answer

roles/compute.instanceAdmin.v1manage VMs, nothing else
roles/storage.objectViewerread bucket objects only
roles/pubsub.publisherpublish to topics only

Curated per service by Google (1,000+ exist), updated as services grow. Naming grammar: roles/SERVICE.persona. Default exam answer for "least privilege".

Custom roles — your exact list

A permission list you assemble, defined at project or org level (not folder). Use when no predefined role is tight enough. Maintenance is on you: when a service adds permissions, your custom role doesn't grow itself.

gcloud iam roles create logReader \
  --project=example-project-123 \
  --title="Log Reader" \
  --permissions=logging.logEntries.list
Memory hook: basic = blunt, predefined = precise, custom = crafted. The exam's least-privilege ladder runs blunt → precise → crafted; climb only as far as you must.
Cloud Storage ACLs (Access Control Lists): a legacy, storage-only permission system that predates IAM and runs in parallel with it — access is granted if either allows. Prefer uniform bucket-level access, which disables ACLs so IAM is the single source of truth — see the storage sheet.

3 · Service accounts — non-human identities

A service account (SA) is an identity for code — a VM, a Cloud Run service, a CI (Continuous Integration) pipeline. Email-shaped (name@project-id.iam.gserviceaccount.com) but no password, no inbox, no human. Twist to remember: an SA is both a principal and a resource — it can be granted roles, and roles can be granted on it (controlling who may use it).

The default Compute Engine SA: PROJECT_NUMBER-compute@developer.gserviceaccount.com (note: project number, e.g. 123456789012, not the project ID). Historically it was auto-granted Editor on the whole project — so any process on any default VM could modify nearly anything. Modern guidance (and newer org policies) kill that grant. Fix: create a dedicated SA per workload with only the predefined roles it needs, and attach that instead.

ATTACH — the resource wears the identity

You attach an SA to a resource (VM, Cloud Run, Cloud Function) at creation/update. Code on that resource then fetches tokens from the metadata server (metadata.google.internal, a link-local endpoint only the VM can reach) — no credential files anywhere.

gcloud compute instances create vm1 \
  --service-account=app-sa@example-project-123.iam.gserviceaccount.com \
  --scopes=cloud-platform

Gate: to attach an SA you need iam.serviceAccounts.actAs on it — carried by roles/iam.serviceAccountUser. This stops a low-privilege dev from deploying code as a high-privilege SA (privilege escalation by deputy).

IMPERSONATE — a principal borrows the identity

Your own credentials ask Google's IAM Credentials API to mint short-lived tokens (default 1 h) for the SA. Nothing attached, nothing downloaded — access is auditable and expires by itself.

gcloud storage ls \
  --impersonate-service-account=\
app-sa@example-project-123.iam.gserviceaccount.com

Gate: roles/iam.serviceAccountTokenCreator on the SA. Standard way to test "what can my app's SA actually see?" from your laptop.

Mnemonic: attach = wear it (the VM puts the identity on like a uniform, via the metadata server) vs impersonate = borrow it (you check out a temporary badge). Gates rhyme with intent: User role to make a resource use it, TokenCreator role to create tokens as it.
Why downloaded SA key files are the anti-pattern: gcloud iam service-accounts keys create key.json hands you a private key that never expires, works from anywhere, and can't tell Google who is actually using it. Leaked keys (committed to Git, baked into images, pasted in chat) are a top cloud-breach vector. Every alternative above — attached SA, impersonation, and Workload Identity Federation for external systems like GitHub Actions — gives short-lived, auto-rotated, attributable credentials. Treat key creation as a last resort; org policy iam.disableServiceAccountKeyCreation can ban it outright.

4 · ADC — how code finds credentials

ADC (Application Default Credentials) is the search strategy baked into all Google client libraries: your code never names a credential, the library looks for one. Three steps, first hit wins:

1. $GOOGLE_APPLICATION_CREDENTIALS          env var → path to a credential JSON
        ↓ not set?
2. ~/.config/gcloud/application_default_credentials.json
        ("well-known file", written by
         gcloud auth application-default login)
        ↓ not there?
3. metadata server → the ATTACHED service account
        (you're on GCP: GCE, Cloud Run, GKE…)
        ↓ none?
ERROR: could not find default credentials
The elegance: the same code uses your credentials on your laptop (step 2) and the attached SA in production (step 3) — zero code changes, zero key files.

Two logins, two credential stores

CommandFeeds
gcloud auth loginthe gcloud CLI itself — every later gcloud/gsutil/bq command
gcloud auth application-default loginADC — your code & client libraries (and Terraform), via the well-known file

Separate stores: logging into one does nothing for the other. The classic confusion: gcloud works fine but your Python script throws "could not find default credentials" — you did login #1 and skipped login #2.

Refresh token vs access token

Refresh tokenlong-lived, stored on disk at login; never sent to APIs — its only job is to mint access tokens
Access tokenshort-lived (~1 h) bearer token actually sent in Authorization: Bearer … headers

OAuth 2.0 pattern: the refresh token is the key to the mint, access tokens are the coins. gcloud auth print-access-token shows you a coin.

Managed-account reauth (the invalid_rapt error): Workspace / Cloud Identity admins can enforce session control — cloud sessions expire (e.g. every 16 h) even though the refresh token exists. Symptom: commands that worked yesterday fail with invalid_rapt (ReAuth Proof Token) or "reauthentication required". Fix: just log in again — gcloud auth login and/or gcloud auth application-default login, matching whichever store expired. Personal gmail.com accounts never see this.

5 · How GCP auth maps to OAuth

Everything in §4 is plain OAuth 2.0 (the open delegation standard — actors, flows, and token mechanics in the companion Authentication & OAuth sheet) wearing Google branding. Decoder ring:

GCP thingWhat it is in OAuth terms
gcloud auth logina textbook authorization-code flow: gcloud opens your browser, you log in and consent at accounts.google.com (the authorization server), gcloud receives the code on a localhost redirect and swaps it for tokens
cached credential on diskthe refresh token from that flow — long-lived minter, which is why you log in once, not hourly
what every API call sendsan access token — the short-lived (~1 h) bearer token in Authorization: Bearer …; gcloud auth print-access-token shows you one
ADCnot a credential type at all — just a search convention for finding OAuth credentials (env var → well-known file → metadata server)
reauth / invalid_raptorg session policy freezing your refresh token until you re-authenticate — the token exists but the authorization server refuses to mint from it
service-account authsigned-JWT-for-token exchange (client-credentials style, RFC 7523): the SA signs a JWT with its private key, the token endpoint returns an access token — no human, no consent screen
Workload Identity FederationOIDC federation: an external workload (GitHub Actions, AWS, another cloud's CI) presents its own platform's OIDC token as proof of identity, and Google exchanges it for GCP credentials — identity proven by federation, no key file ever downloaded
Legacy wrinkle — access scopes vs IAM: VM access scopes (--scopes=cloud-platform) are OAuth scopes baked into the attached SA's tokens — a pre-IAM coarse filter on what the token may touch. Effective access = scopes ∩ IAM roles: a VM scoped to storage-read-only can't write to buckets even if its SA has Editor. Modern guidance: set cloud-platform (widest scope) and do all real access control with IAM roles — one precise system instead of two overlapping ones.
Why this mapping is worth owning: once you see that gcloud login = auth-code flow, SA auth = signed-JWT exchange, and ADC = a lookup order, every "weird" GCP auth behavior becomes predictable from OAuth first principles — including why tokens expire hourly, why refresh tokens never hit APIs, and why key files are the outlier that breaks the model.

6 · API keys vs IAM auth vs API enablement — three different things

Three mechanisms that beginners blur into one "make the API work" blob. Disambiguate:

API keyIAM authenticationAPI enablement
What it is an opaque string identifying the calling project (for billing/quota) OAuth tokens proving who the caller is, checked against IAM policy a per-project on/off switch for each service's API
Answers "whose quota do I bill?" "who are you, and may you?" "is this service even switched on here?"
Carries identity? no — anyone holding the string can use it yes — principal, expiry, audit trail n/a — not a credential at all
Used for a few public-data APIs (Maps, Translate); restrict by referrer/IP/API everything that touches your resources — the normal path prerequisite for any call: disabled API → 403 SERVICE_DISABLED no matter how mighty your role
Command gcloud services api-keys create gcloud auth … + IAM bindings gcloud services enable compute.googleapis.com
Debug order when an API call fails: ① is the API enabled in the project? ② are you authenticated as who you think (gcloud auth list)? ③ does that principal hold a role with the needed permission? Enablement → identity → permission; most "IAM problems" turn out to be ① or ②.

CommandDoes
gcloud auth listshow all logged-in accounts; * marks the active one
gcloud config set account alex@example.comswitch the active account (must already be in the list)

Grammar hook: grants live under gcloud projects (the policy belongs to the resource), role definitions under gcloud iam roles, identities under gcloud iam service-accounts, your own logins under gcloud auth. Ask "whose object is this?" and the command group follows.

8 · Glossary recap

IAMIdentity and Access Management — who can do what on which resource
Permissionatomic capability, service.resource.verb; never directly grantable
Rolenamed bundle of permissions; basic (blunt) / predefined (precise) / custom (crafted)
Principalan identity a binding can name: user, group, serviceAccount, domain, allUsers, allAuthenticatedUsers
Bindingone role ↔ list of principals (+ optional CEL condition)
CELCommon Expression Language — the mini-language for IAM condition tests (time windows, name prefixes)
Allow policythe set of bindings attached to one resource; inherited down org → folder → project → resource
Service account (SA)non-human identity for code; both a principal and a resource
Attach vs impersonateresource wears the SA (metadata server; gate actAs/ServiceAccountUser) vs principal borrows it via short-lived tokens (gate TokenCreator)
Metadata serverlink-local endpoint on GCP compute that hands the attached SA's tokens to local code
ADCApplication Default Credentials — client-library search order: env var → well-known file → metadata server
Refresh / access tokenlong-lived minter on disk vs short-lived bearer token sent to APIs
OAuth 2.0the open delegation standard underlying all GCP auth — auth-code flow, refresh/access tokens, scopes
OIDCOpenID Connect — the identity layer on top of OAuth; Workload Identity Federation trusts an external platform's OIDC tokens

WIFWorkload Identity Federation — exchanges an external workload's OIDC token for GCP credentials; no key file downloaded
APIApplication Programming Interface — the request endpoints IAM permissions guard (one permission ≈ one API method)
invalid_raptReAuth Proof Token expired under managed-account session control; fix = log in again
API keyproject-identifying string for quota/billing on a few public APIs; carries no identity
API enablementper-project on/off switch for a service; off → 403 regardless of IAM
