HTTP & REST Cheat Sheet

The web's request/response protocol · methods · status codes · REST · started 2026-06-12 · twin of Tools/http_cheatsheet.md

HTTP (HyperText Transfer Protocol) = the request/response protocol the web runs on. Tim Berners-Lee sketched the first version at CERN in 1991; HTTP/1.1 (1997) is still the mental model everyone carries. REST (REpresentational State Transfer) = an architectural style for APIs, named in Roy Fielding's 2000 PhD thesis. Fielding co-authored the HTTP spec, so REST is really "HTTP used the way it was designed."

0 · The mental model: one round trip

HTTP is strict turn-taking. The client (browser, your code, curl) speaks first with a request; the server answers once with a response. Then it's over.

  client (you)                                   server
  ────────────                                   ──────
  HTTP request  ──────────────────────────────►  matches URL to a resource
   method  (GET / POST / …)                       does the work
   URL     https://api.example.com/users/42       │
   headers (auth, content-type, …)                │
   body    (JSON, if sending data)                ▼
                ◄──────────────────────────────  HTTP response
                                                  status line  (200 / 404 / …)
                                                  headers
                                                  body  (usually JSON)
  • Request/response, one-shot. One request → exactly one response. The client always initiates (the server never spontaneously calls back — see webhooks, §10, for the inversion).
  • Stateless. The server keeps no memory between requests. Every request carries everything needed to serve it (auth included) — which lets a load balancer route each request to any of a hundred identical replicas.
HTTP rides on layers you don't see here: TCP (reliable byte stream), TLS (encryption — the S in HTTPS), DNS (hostname → IP). See networking_cheatsheet.md for that plumbing.

0b · HTTP versions in one line each

VersionYearThe change that matters
HTTP/0.91991one line: GET /page — no headers, no status codes
HTTP/1.01996added headers, status codes, content types
HTTP/1.11997keep-alive (reuse one TCP connection for many requests); still the canonical mental model
HTTP/22015multiplexing — many requests in flight on one connection; binary framing; header compression
HTTP/32022runs over QUIC (UDP-based, TLS built in) instead of TCP → faster setup, survives Wi-Fi→cellular switches

The semantics (methods, codes, headers) are identical across versions — only the wire transport changed. Learn HTTP/1.1's model and you know them all.

1 · Anatomy of a request  ·  2 · Anatomy of a response

Request

POST /v1/users HTTP/1.1    ← method + path + version
Host: api.example.com      ← headers: "Key: value"
Authorization: Bearer eyJ…
Content-Type: application/json
Accept: application/json
                           ← blank line ends headers
{"name": "Ada"}          ← body (POST/PUT/PATCH)
  1. Method (verb) — what you want done (§3)
  2. URL / path — which resource (§5)
  3. Headers — Key: value metadata (§7)
  4. Body — payload, usually JSON

Response

HTTP/1.1 201 Created       ← version + code + reason
Content-Type: application/json
Location: /v1/users/42     ← points at the new thing
                           ← blank line
{"id": 42, "name": "Ada"}  ← body

Same shape, mirrored: status line (the code is the headline — §4), headers, blank line, body. The reason phrase ("Created") is human sugar; your program checks the code (201).

Safe = read-only; observes but never changes server state (GET, HEAD, OPTIONS) — always cacheable. Idempotent = once or ten times leaves the same end state, so retry-safe (GET/PUT/DELETE). POST is not idempotent — retry and you may create two users; that's why retry logic treats POST with care. Safe ⊂ idempotent (DELETE changes state yet deleting twice still ends "gone").

4 · Status codes — the first digit tells the story

FamilyMeaningFamous members
1xxinformational (rare)100 Continue · 101 Switching Protocols (WebSocket upgrade)
2xxsuccess200 OK · 201 Created (after POST) · 204 No Content (success, nothing to return — e.g. after DELETE)
3xxredirection301 Moved Permanently (update bookmark) · 302 Found (temporary) · 304 Not Modified (your cache is still good)
4xxyour fault (client)400 Bad Request · 401 Unauthorized · 403 Forbidden · 404 Not Found · 409 Conflict (dup / edit collision) · 418 I'm a teapot (1998 joke, still in spec) · 429 Too Many Requests (rate-limited)
5xxtheir fault (server)500 Internal Server Error · 502 Bad Gateway (upstream broke) · 503 Service Unavailable (overloaded/down — retry later)
401 vs 403: 401 = "log in first" (missing/bad credentials — server doesn't know you; its name "Unauthorized" is a misnomer for unauthenticated). 403 = "logged in, still no" (it knows you; you lack permission).
502 vs 503: 502 = a proxy/gateway got a broken answer from behind it; 503 = the server is up but refusing work.
Rule: 4xx → fix your request; 5xx → retry (maybe), or it's not your problem.

                                resource)     joined by &)        never sent)

The fragment (after #) is handled by the browser only and never sent to the server. Port defaults to 443 (https) / 80 (http) and is usually omitted.

6 · REST — what "RESTful" actually means

Not a protocol or library — a set of conventions for designing predictable HTTP APIs:

  1. Resources are nouns in URLs — /users/42/orders, never /getUserOrders?id=42. Name the thing, not the action.
  2. Verbs come from HTTP — the method says what you're doing to the noun. You never invent verbs.
  3. Stateless — every request carries everything (auth included); no session memory. This is what makes REST scale.
  4. Representations — responses are a representation (usually JSON) of the resource's state. That's literally where the name comes from.

CRUD over HTTP — the canonical mapping

CRUD = Create, Read, Update, Delete. REST maps them onto a noun (/users):

IntentMethod + URLTypical success
Create a userPOST /users (body)201 Created + Location
Read all usersGET /users200 OK + array
Read one userGET /users/42200 OK + object (or 404)
Update (replace)PUT /users/42 (full body)200 OK
Update (partial)PATCH /users/42 (changed fields)200 OK
Delete a userDELETE /users/42204 No Content

Collection (/users) vs item (/users/42): POST to the collection to create; act on the item by id for read/update/delete.

7 · Key headers

HeaderOnWhat it does
Content-Typereq & respwhat the body is — application/json, text/html, multipart/form-data. The receiver parses accordingly.
Acceptrequestwhat the client wants back — Accept: application/json. Drives content negotiation (§8).
Authorizationrequestthe credential — Bearer <token>, Basic <base64> (§8).
User-Agentrequestidentifies the client software (curl/8.4, a browser string).
Cache-Controlreq & respcaching rules — no-store, max-age=3600. Pairs with ETag + 304 to skip re-downloading unchanged data.
Set-Cookie / Cookieresp / reqserver stores a cookie in the browser; browser sends it back automatically (§8).
Locationresponsewhere the new/moved resource lives (with 201 or 3xx).
CORS headersresponseAccess-Control-Allow-Origin: … — the server's opt-in list of which other sites' JS may read its responses. Browsers block cross-site reads by default; a missing CORS header is the classic "works in curl, fails in the browser" bug.

8 · Auth · content negotiation · cookies & sessions

Auth patterns (full depth in authentication_cheatsheet.md)

PatternLooks likeNotes
API key in headerx-api-key: sk-abc123…simplest; a static secret identifying the app. (Anthropic's API uses exactly this.)
Bearer tokenAuthorization: Bearer eyJhbGc…"whoever bears this token is authorized." Often a JWT (JSON Web Token — signed JSON payload); usually short-lived.
Basic authAuthorization: Basic dXNlcjpwYXNzbase64 of user:password — old, only safe over HTTPS, rare in modern APIs.
OAuth 2.0a flow, not a headergrant App A limited access to your data in Service B without handing A your B password — ends with A holding a Bearer token.
Universal rule: secrets live in environment variables (os.environ["API_KEY"]), never in code or Git.

Content negotiation

The client says what it can accept; the server picks. Accept: application/json + Accept-Language: fr ask for "JSON, in French if you have it"; the server replies with matching Content-Type / Content-Language. Same URL, different representations.

Cookies & sessions (brief)

A stateless protocol still needs to remember "you're logged in." The server sends Set-Cookie: session=abc123 once; the browser automatically attaches Cookie: session=abc123 on every later request to that domain, and the server looks up abc123 in its session store. (Token APIs skip cookies and send Authorization each call — the more RESTfully stateless approach.)

9 · Worked example — same call in curl and Python

CLI (terminal) — curl

# GET with a query param + auth header.
# -v = verbose: prints request/response lines + headers
curl -v "https://api.example.com/v1/users?limit=2" \
     -H "Authorization: Bearer $TOKEN"

# POST: create a resource (expect 201 Created)
curl -s -X POST "https://api.example.com/v1/users" \
     -H "Authorization: Bearer $TOKEN" \
     -H "Content-Type: application/json" \
     -d '{"name": "Ada", "city": "Lisbon"}'

Python — requests

import requests
headers = {"Authorization": f"Bearer {token}"}

r = requests.get("https://api.example.com/v1/users",
                 params={"limit": 2}, headers=headers)
r.raise_for_status()   # raises on any 4xx/5xx
users = r.json()       # parse JSON body → list/dict

r = requests.post("https://api.example.com/v1/users",
                  json={"name": "Ada", "city": "Lisbon"},
                  headers=headers)   # json= sets body + header
print(r.status_code)   # 201
json= (auto-serializes the dict and sets Content-Type: application/json) vs data= (raw form/bytes) is the classic requests distinction.

10 · Webhooks vs polling

So far you call the server. Sometimes you want the server to tell you when something happens:

PollingWebhook
Directionyou ask repeatedly: "anything new?"they call you when something happens
Mechanicsa loop of GET requests on a scheduleyou register a URL; the service POSTs JSON to it on each event
Latency vs costlatency = poll interval; wasteful (most polls return nothing)near-instant, no wasted calls
Catchrate limits (hello, 429)you must run a publicly reachable server and verify the payload signature (anyone can POST to a URL)
Memory hook: polling = checking the mailbox every five minutes; webhook = the doorbell. A webhook inverts the roles — their server becomes the client of your endpoint.

11 · Teardown / connection hygiene

A TCP connectionHTTP/1.1 keeps connections alive (Connection: keep-alive) for reuse; they close on Connection: close or idle timeout. Let your client library pool & close them — don't open a fresh connection per request.
An API keyrevoke it in the provider's console — deleting it from your code is not revocation; a leaked key works until the server side kills it.
A Bearer token / session"log out" properly = the server invalidates the token/session; merely deleting the cookie only makes your browser forget it — a stolen copy stays valid.
A webhook endpointunsubscribe/disable it in the sending service — otherwise it keeps POSTing events to a URL you no longer control (a re-registered domain could receive your data).
A streaming connectionclose it explicitly — with requests.get(url, stream=True) as r: (or r.close()) — abandoned streams hold sockets and server resources open.
