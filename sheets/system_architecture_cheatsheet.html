System Architecture Cheat Sheet

From silicon to your script: languages · kernels · containers · started 2026-06-06 · twin of Tools/system_architecture_cheatsheet.md

0 · The mental model: the layer cake (read once)

One picture for the whole sheet. Software is layers; each layer talks only to its neighbours and pretends the ones below don't exist: 
┌─────────────────────────────────────────────────────┐
│ YOUR CODE        app.py, script.sh, main.go         │ ← what you write
├─────────────────────────────────────────────────────┤
│ RUNTIME/LIBS     Python interpreter, libc, numpy,   │ ← what runs it
│                  JVM, Node.js, shared libraries     │
├─────────────────────────────────────────────────────┤
│ OS USER SPACE    shell (zsh), ls/cp, daemons, GUI   │ ← the OS you see
├──────────────── syscall boundary ───────────────────┤
│ KERNEL           scheduler, memory mgr, drivers,    │ ← the OS you don't
│                  filesystems, network stack         │
├─────────────────────────────────────────────────────┤
│ HARDWARE         CPU, RAM, disk, NIC, GPU           │ ← the silicon
└─────────────────────────────────────────────────────┘
Going down = asking the layer below for a service. Going up = hiding ugly details. That hiding is abstraction — the most load-bearing idea in computing.
Why it matters: every tool lives at exactly one layer. zsh = user space. venv = runtime. Docker = a wall inside user space. A VM = a copy of everything from the kernel up. The layer tells you what a tool can and cannot isolate.

Vocabulary anchors

CPU (Central Processing Unit)the chip executing instructions, billions/sec
OS (Operating System)kernel + user-space programs shipped together
Binary / executablefile of machine code the CPU runs directly

Etymology hooks

Kernelthe core of a nut/seed — the essential part once you strip the shell (yes, bash/zsh is named as its complement)
Daemonbackground service (sshd, dockerd — trailing d is the tell); from Maxwell's demon, adopted at MIT in the 1960s. Pronounced "demon".

1 · Languages — how code becomes execution

The CPU understands one language: machine code — raw numeric instructions. Assembly = machine code with human mnemonics (mov, add, jmp), translated 1:1 by an assembler. Everything compiles down to it eventually.

Three strategies, three trade-offs

Compiled
C, C++, Rust, Go		compiler translates whole source → native binary ahead of time. Fastest execution; slowest iteration (edit→recompile→run); binary is per-OS/CPU.
Interpreted
Python, Ruby, shell		interpreter executes at runtime. Python: source → bytecode (instructions for a fake portable CPU, the .pyc files) → its virtual machine loop runs it. Instant iteration; 10–100× slower for tight loops.
JIT (Just-In-Time)
JavaScript (V8), Java/JVM, C#		starts interpreting, watches for hot code, compiles those parts to machine code mid-flight. Near-compiled speed after warm-up; slower startup.
"Python is slow but numpy is fast" decoded: numpy's array math is compiled C (and Fortran) in a thin Python skin. a + b on two numpy arrays = one Python call dispatching into a C loop crunching a million elements at native speed. Same trick in pandas, PyTorch, sqlite3 — Python as glue over compiled cores. Rule: Python loops are slow; Python calls to compiled code are fast.

Source code = text you write → bytecode = instructions for a VM → machine code = the only thing the CPU runs.

2 · The kernel — the part of the OS you never see

The kernel's four jobs

Schedulingwhich process gets the CPU next, switching 1000s of times/sec — how 4 cores "run" 400 processes
Memory managementvirtual memory: each process gets a private fake address space mapped to real RAM; A can't read B's memory
Driverstranslate "write these bytes" into each device's specific incantations
Filesystemsthe illusion of named files/folders over raw disk blocks (APFS, ext4)

Processes vs threads

Processa running program; own private virtual memory; created by fork/exec; a crash stays contained
Threada worker inside a process; shares memory with sibling threads; one bad thread can corrupt the whole process
User mode vs kernel mode & syscalls. The CPU enforces two privilege levels: your code runs in user mode (no hardware access), the kernel in kernel mode (anything). The only doorway between them is the syscall (system call): open, read, write, fork, exec, exit… a few hundred total. 
$ strace -c python3 -c 'print("hi")'   # Linux: count every syscall
  openat ~90×   read ~60×   mmap ~70×   write 1×  ← the print!
strace (macOS: dtruss) shows a program's entire conversation with the kernel.
Kernel family tree — why your Mac terminal feels like Linux: 
Unix (Bell Labs, 1969)
 ├─► BSD (Berkeley, 1977) ──► Darwin/XNU ──► macOS, iOS
 └─► (ideas, POSIX std) ────► Linux (Torvalds, 1991) ─► Android, servers, cloud
Windows NT (1993) — separate lineage entirely
POSIX (Portable Operating System Interface) = the IEEE standard freezing the Unix API — why ls/grep work across the family. Linux shares no Unix code, only the interface. XNU ("X is Not Unix") = macOS's kernel (Mach + BSD); macOS is a certified Unix — your Mac commands transfer almost unchanged to a Linux server. Windows NT descends from VMS, not Unix; WSL bolts a real Linux kernel alongside.

3 · Virtualization — VMs vs containers

Two ways to slice one machine into many isolated environments. The difference: which layer gets duplicated.

VM vs container

VM (Virtual Machine)duplicates everything: fake hardware + full guest kernel + user space, via a hypervisor (VMware, KVM). Boots in minutes, weighs GBs, strongest isolation, can run a different OS (Linux VM on a Mac).
Containerduplicates user space only — all containers share the host kernel. Boots in ms, weighs MBs; a kernel exploit escapes all containers; Linux containers need a Linux kernel (Docker on macOS quietly runs a Linux VM underneath!).
The two kernel features behind containers: namespaces = scoping what a process can see (own process list, network, hostname, filesystem root — inside one, a container believes it's the whole machine); cgroups (control groups) = limits on what it can use (CPU share, RAM, I/O) so one container can't starve the rest.
Etymology hook: named after the 1956 standardized steel shipping container that made cargo machine-handleable anywhere. Same promise for software: pack the app with its entire user space so it runs identically on any compatible kernel. "Works on my machine" → ship the machine (minus the kernel).

Docker vocabulary

Imageread-only template: frozen user-space filesystem + metadata — the "class"
Layerimages are stacked diffs; layers cached & shared (10 Python images ≈ one base layer on disk)
Containerrunning instance of an image + thin writable layer — the "object"
Registryserver hosting images; Docker Hub = the GitHub of images
Dockerfilethe recipe (FROM, COPY, RUN) that builds an image

Docker commands

docker run -it ubuntu bashpull if needed, start a container, interactive shell inside
docker ps / ps -alist running / all containers
docker exec -it <name> bashshell into an already-running container
docker build -t myapp .build image myapp from the Dockerfile here
docker imageslist local images

When to use what

Bare metalevery drop of performance or direct hardware access (GPU training, high-frequency anything)
VMdifferent OS than host; hard multi-tenant security boundaries; legacy systems
Containerthe default for deploying services: fast, light, reproducible, the unit every cloud speaks
Kubernetes in one callout: Kubernetes (K8s, Greek for "helmsman") = container orchestration — given hundreds of containers across many machines, it decides placement, restarts crashes, scales replicas, routes traffic. Its atom is the pod: one or more containers deployed and scheduled as a single unit.
Serverless isn't serverless — and you're soaking in it: cloud "serverless" (AWS Lambda, Cloud Run) = your code dropped into a container or microVM (a stripped-down, ms-boot VM like Firecracker) created and destroyed on demand. Live example: the Cowork/Claude sandbox that ran shell commands to build this very file is an isolated Linux environment — your Mac folders mounted into a contained Linux user space. The layer cake isn't theory; it's where this sentence was built.

4 · Tying it together — python app.py, keystroke to CPU

$ python app.py
1. Shell parses        zsh (user space) splits command + arg, finds python via $PATH
2. fork + exec         shell's two syscalls: clone itself, replace clone with the
                       python binary — first kernel involvement
3. Kernel loads        maps the binary into fresh VIRTUAL MEMORY, wires libc,
                       sets instruction pointer to Python's entry point
4. Interpreter boots   the python binary (compiled C!) inits its runtime:
                       built-in types, sys.path, import machinery
5. Source → bytecode   reads app.py (openat + read syscalls), compiles to
                       bytecode, caches in __pycache__/
6. VM loop runs        interpreter executes bytecode instruction by instruction —
                       each one a chunk of the interpreter's machine code on the CPU
7. I/O = syscalls      every print() → a write syscall: CPU flips to kernel mode,
                       kernel drives the device, flips back (scheduler slicing
                       your process in and out the whole time)
8. Exit                exit syscall → kernel frees memory, hands the waiting
                       shell the exit status (echo $? — 0 = success). Prompt returns.

Every layer appeared once: your code (5–6), runtime (4–6), user space (1), syscall boundary (2, 7, 8), kernel (3, 7, 8), hardware (everywhere, invisibly).

5 · Teardown (containers & processes)

How to stop things at each layer — gently first, forcefully if needed.

docker stop <name>stop politely — sends SIGTERM (please clean up), waits ~10 s, then SIGKILL
docker kill <name>stop instantly — straight to SIGKILL, no grace period; no chance to flush or clean up
docker rm <name>delete a stopped container (the "object"); the image stays
docker rmi <image>delete an image (the "class"); fails if containers still use it
docker system prunebulk cleanup — deletes ⚠ all stopped containers, dangling images, unused networks, build cache; -a adds unused tagged images too. Read the prompt.
kill <pid> vs kill -9 <pid>SIGTERM (catchable — graceful) vs SIGKILL (uncatchable — last resort; no cleanup runs)
Shut down vs power offa clean shutdown lets the kernel flush buffers and unmount filesystems; yanking power risks data in RAM never reaching disk — SIGTERM vs SIGKILL, one layer down
Memory hook: the pattern repeats at every layer — ask first (SIGTERM / docker stop / shutdown), force only when asking fails (SIGKILL / docker kill / power cut).
