GCP Stable Endpoints & Messaging Cheat Sheet

One meta-pattern, four services: Pub/Sub decoupling, Kubernetes Services, Cloud DNS, and static vs ephemeral IPs — the decision triggers and distractor traps at GCP Associate Cloud Engineer depth.

0 · The meta-pattern that unifies this whole sheet

Never pin a fixed address to an ephemeral thing — put a stable abstraction in front. Every section here is the same idea wearing a different hat. The thing that changes (a pod, a replaceable VM, a pool of workers) hides behind a thing that stays put (a Service, a static IP + DNS name, a load balancer). The exam loves the wrong answer where someone hardcodes the ephemeral address directly.
The ephemeral thingThe stable abstraction in frontSection
Pods (new IP on every restart)a Kubernetes Service (stable virtual IP + DNS name)§2
A replaceable VM (new ephemeral IP on restart)a static IP + a Cloud DNS A-record§3, §4
Many backends scaling up/down (a MIG)a load balancer (one stable frontend)§1 backends, cross-ref
A fast/slow producer vs consumera Pub/Sub topic (a buffer that decouples them)§1

MIG = Managed Instance Group: a pool of identical, autoscaling VMs created from one template (covered in the Compute sheet). It is the canonical "changing set of backends" that both load balancers and Pub/Sub worker-pools sit in front of.

1 · Pub/Sub — managed async messaging that decouples

Pub/Sub = "Publish / Subscribe": a fully managed, global message bus. Its whole reason to exist is decoupling — the thing producing messages never has to know who (or how many things) will consume them, and the two sides can run at different speeds.

The model — four nouns

Publisher ──► Topic ──► Subscription ──► Subscriber
 (sends)     (named    (a queue of      (pull or
            channel)  unread msgs)      push)
Publisherany code that sends a message to a topic
Topica named channel messages are published to
Subscriptiona named stream attached to a topic; holds unacked messages until a subscriber acks them
Subscriberconsumes from a subscription — pull (subscriber asks) or push (Pub/Sub POSTs to an HTTPS endpoint)

Fan-out = one topic, many subscriptions: each subscription gets its own independent copy of every message. Work distribution = one subscription, many subscriber instances: messages are spread across the workers (each message goes to one worker).

Two jobs it does on the exam

1 · Buffer spikes (ingest). A flood of events arrives faster than workers can process. Pub/Sub absorbs the burst into the topic; workers drain it at their own pace. Nothing is dropped, nothing falls over.

2 · Fan work out to a worker pool. "Spread these transactions across a MIG of workers" → publish each transaction; the MIG's instances pull and process in parallel; autoscale the MIG on the subscription's backlog.

Distractor trap: "spread/distribute transactions across a MIG" is a messaging problem → Pub/Sub. Bigtable is a wrong answer here — Bigtable is a database (storage), not a way to distribute work. Storage ≠ delivery.

Delivery semantics — the part people get wrong

PropertyReality
Delivery guaranteeAt-least-once by default — a message may be delivered more than once, so subscribers must be idempotent (safe to process the same message twice). (Exactly-once is an opt-in mode on pull subscriptions.)
OrderingNot FIFO by default — messages are generally unordered. It is not a strict first-in-first-out queue out of the box.
Ordering keysattach an ordering key to messages → Pub/Sub delivers messages with the same key in publish order. FIFO within a key, not globally. Use when per-entity order matters (e.g. all events for one user-id).
Trap: "guaranteed strict ordering of all messages" is not plain Pub/Sub. If a question demands global FIFO, the honest answer is ordering keys (per-key only) or a different design — don't assume the queue is ordered.
Memory hook — "Pub/Sub is a post office, not a phone call." You drop a letter (publish) and walk away; you never wait for the recipient. Letters may arrive out of order, and the post office might even deliver a duplicate. Decoupled, async, at-least-once. (A phone call would be synchronous and coupled — that's a direct API request, not Pub/Sub.)

The canonical streaming pipeline

Pub/Sub  ──►  Dataflow  ──►  BigQuery
(ingest)    (transform,    (analyze,
            stream proc)    SQL warehouse)

Memorize this trio. Pub/Sub ingests/buffers the event stream; Dataflow (managed Apache Beam) does the streaming transform/enrichment; BigQuery is the analytics warehouse it lands in. "Ingest streaming events for real-time analytics" almost always wants this shape.

Pub/Sub vs Cloud Tasks (one line)

Pub/Sub = high-throughput event broadcast/fan-out, many subscribers, you don't control per-message timing. Cloud Tasks = a managed task queue where you dispatch individual tasks to a specific handler with explicit per-task control (rate limits, scheduling, retries). Rule of thumb: events to many listeners → Pub/Sub; commands to one worker, with fine control → Cloud Tasks.

docs: cloud.google.com/pubsub

Minimal Pub/Sub, end to end

# a topic (the named channel)
gcloud pubsub topics create orders-topic

# a subscription on it (pull by default)
gcloud pubsub subscriptions create orders-sub --topic=orders-topic

# publish a message
gcloud pubsub topics publish orders-topic --message="order-42"

# pull and acknowledge it
gcloud pubsub subscriptions pull orders-sub --auto-ack

# enable per-key ordering when you create the subscription:
gcloud pubsub subscriptions create ordered-sub \
  --topic=orders-topic --enable-message-ordering

2 · Kubernetes Service — a stable front for ephemeral pods

A pod is the smallest deployable unit in Kubernetes (K8s — the container orchestrator GKE runs; GKE = Google Kubernetes Engine). Pods are ephemeral: when one dies, restarts, scales, or gets rescheduled, it comes back with a brand-new IP address.

The classic wrong answer: "hardcode the pod's IP so other pods can reach it." Never do this — the IP changes the moment the pod restarts, and the connection breaks. Pod IPs are not addresses you pin anything to. This is the §0 meta-pattern in its purest form.

What a Service is

A Service is a stable abstraction in front of a changing set of pods. It gives you:

A stable virtual IPthe "ClusterIP" — fixed for the Service's lifetime, regardless of pod churn
A stable DNS namein-cluster name like my-app.default.svc.cluster.local — call it by name, never by pod IP
Label selectionpicks its target pods by label (e.g. app=my-app), not by IP — as pods come and go, the live set updates automatically
Load balancingspreads requests across the currently-healthy matching pods

Trigger: "uninterrupted pod-to-pod communication despite scaling and rolling updates" → put a Service in front. Callers hit the Service's stable name; the Service tracks whichever pods are alive.

Service types (recognition)

ClusterIPdefault. Internal-only virtual IP, reachable from inside the cluster. The answer for pod-to-pod / internal traffic.
NodePortopens the same port on every node's IP — a low-level building block, rarely the direct answer.
LoadBalancerprovisions an external (cloud) load balancer with a public IP — the answer for exposing a Service to the internet.
The in-cluster twin of a load balancer. A Service is "stable frontend → changing backends" living inside the cluster — exactly what an external load balancer does for VMs, scaled down to pods.

Scope note — this is a Kubernetes-only concept

PlatformHow you get a stable endpoint
GKE / Kubernetescreate a Service (this section) — the stable IP/DNS abstraction
Cloud Runyou get an automatic stable HTTPS URL per service out of the box — no Service object to manage
Plain Docker on a VMneither — no Service, no auto-URL; you wire up your own IP/DNS/LB by hand
Trap: "Kubernetes Service" as an answer only makes sense when the workload is on GKE/K8s. If the scenario is Cloud Run, the stable endpoint is its built-in URL; if it's a lone VM, it's a static IP. Don't reach for "Service" on a non-Kubernetes scenario. 
# expose a Deployment's pods behind a stable internal Service
kubectl expose deployment my-app --port=80 --target-port=8080 \
  --name=my-app-svc            # ClusterIP by default

# or declaratively (service.yaml)
apiVersion: v1
kind: Service
metadata: { name: my-app-svc }
spec:
  selector: { app: my-app }    # picks pods by LABEL, not IP
  ports: [{ port: 80, targetPort: 8080 }]
  type: ClusterIP              # ClusterIP | NodePort | LoadBalancer

3 · Cloud DNS — names instead of numbers

DNS = Domain Name System: the internet's phone book, translating human names (www.example.com) into IP addresses. Cloud DNS is GCP's managed, highly-available DNS service. You organize records into zones (a zone = the records for one domain you control).

Public vs private zones — the key split

Zone typeResolvable from where / used for
Public zoneresolvable from the whole internet. Use to publish a public website or API under your domain — point names at internet-reachable IPs.
Private zoneresolvable only inside your VPC (Virtual Private Cloud — your private GCP network). Use to give internal services friendly names without exposing them publicly.
Private zones are useful, not cosmetic. They let you name internal services (db.internal.example.com → a private IP) so apps reference stable names, while nothing is reachable from the internet. The stable-name benefit of DNS without the exposure.

Record types you must recognize

Aname → IPv4 address (the workhorse: point a hostname at a server's IP)
AAAAname → IPv6 address ("quad-A" — four times the A, for the longer IPv6)
CNAMEname → another name (an alias; Canonical Name)
MXname → mail server (Mail eXchange; where email for the domain goes)
To expose VMs by name to the public: create a public zone + A records pointing at the VMs' stable (static) IPs (§4). Name + stable address = the meta-pattern again. Point the A record at a static IP so replacing the VM never breaks the name.
Distractor — internal DNS can't serve external users. Compute Engine gives every VM an automatic internal DNS name (e.g. my-vm.zone.c.project.internal), but it resolves only inside the VPC. If a question asks how external users reach your VMs by name, internal DNS is the wrong answer — you need a public Cloud DNS zone with A records. 
# a public managed zone for your domain
gcloud dns managed-zones create my-zone \
  --dns-name="example.com." --description="public" --visibility=public

# add an A record pointing the name at a static IP
gcloud dns record-sets create app.example.com. \
  --zone=my-zone --type=A --ttl=300 --rrdatas=203.0.113.10

# a PRIVATE zone, bound to one VPC, resolvable only inside it
gcloud dns managed-zones create internal-zone \
  --dns-name="internal.example.com." --visibility=private \
  --networks=my-vpc

4 · Static vs ephemeral external IPs

An IP address attached to a VM can be ephemeral (temporary) or static (reserved). The difference is what happens when the VM stops — and it's the crux of "why did DNS break after we rebooted the server?"

TypeBehaviourWhen it's the answer
Ephemeral external IP assigned automatically; released when the VM stops; the VM gets a different IP on next start throwaway / dev VMs where nothing outside depends on a fixed address
Static (reserved) external IP you reserve it; it persists across stops, restarts, and even VM replacement (you re-attach the same IP) anything external must reliably find you: a DNS A-record target, a webhook URL, an allowlisted source
Internal static IP a reserved private address inside the VPC; stable across restarts stable private endpoint for an internal service other VMs depend on
The whole point of static IPs: point a DNS A record at a static external IP, and replacing or restarting the server never breaks the name — the address stays put behind the name. Pin the name to the reserved IP, not to the box. (§0 again: stable abstraction in front of a replaceable thing.)

Cost & security — the counter-intuitive billing

External IPv4, in usebills hourly — external IPv4 is a charged, scarce resource
Reserved but unattachedbills MORE than an attached one — a deliberate anti-hoarding charge so people release IPs they aren't using
Cheapest and safestno external IP at all: a private VM + Cloud NAT for outbound egress + IAP for admin access

Cloud NAT (Network Address Translation) lets no-external-IP VMs reach the internet outbound (updates, API calls) without being reachable inbound. IAP (Identity-Aware Proxy) gives you SSH/RDP admin access to those VMs through an identity-checked tunnel — no public port 22. Together they remove the external IP entirely.

BYOIP trap. BYOIP = Bring Your Own IP: import your own public IP block into GCP. It's for large orgs with IP reputation/allowlist commitments — not the cost-effective answer for "we need a stable IP." If the scenario is just "keep a fixed address," reserve a static IP; BYOIP is a distractor unless the question explicitly involves owning/migrating an existing IP range.
Anycast note. Global resources (e.g. global external load balancers) use anycast: one IP advertised from many locations, so users hit the nearest edge. You don't manage anycast directly on a VM — recognize the word as "one IP, many places," the global-LB flavor of a stable frontend. 
# reserve a static external IP (regional)
gcloud compute addresses create app-ip \
  --region=europe-west1

# attach it when creating the VM
gcloud compute instances create my-app \
  --zone=europe-west1-b --address=app-ip

# create a VM with NO external IP (private + cheapest/safest)
gcloud compute instances create my-private-app \
  --zone=europe-west1-b --no-address

5 · Decision reflex — what the trigger words mean

Messaging / decoupling

"decouple producers from consumers"Pub/Sub
"buffer spikes", "ingest events"Pub/Sub
"fan-out", "distribute work to many workers / a MIG"Pub/Sub (not Bigtable)
"stream events for analytics"Pub/Sub → Dataflow → BigQuery
"per-task control, scheduled dispatch to one handler"Cloud Tasks
"strict FIFO of all messages"NOT plain Pub/Sub (ordering keys = per-key only)

Stable endpoints

"uninterrupted pod-to-pod comms despite scaling"Kubernetes Service (ClusterIP)
"expose a Service to the internet"Service type LoadBalancer
"hardcode the pod IP"always WRONG
"external users reach VMs by name"public Cloud DNS zone + A records
"name internal services, no public exposure"private Cloud DNS zone
"something OUTSIDE must reliably find you (DNS/webhook/allowlist)"static external IP
"nothing inbound needed", "cheapest + safest"no external IP (private VM + Cloud NAT + IAP)
"keep our existing public IP block"BYOIP (else it's a distractor)
