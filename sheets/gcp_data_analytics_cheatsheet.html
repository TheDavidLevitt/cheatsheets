0 · The lineage story (read once — it decodes every name below)

Google had a web-scale data problem before anyone else, solved it internally, published the papers, watched the open-source world rebuild the ideas as Hadoop, quietly moved on to better internal systems — and now sells managed open-source back to you (Dataproc) next to its own next-generation engines (BigQuery, Dataflow). Every product name in this sheet is a chapter of that story.
2003Google · paper

GFS paper — the Google File System: chunk petabytes across thousands of cheap machines, replicate for failure tolerance.

2004Google · paper

MapReduce paper — a processing model: split data, map in parallel, shuffle, reduce. Programmers write two functions; the framework handles distribution and failures.

2006Open source · Yahoo / Apache

Hadoop is born — open-source clones of both papers: HDFS (Hadoop Distributed File System ≈ GFS) + MapReduce. Named after creator Doug Cutting's son's toy elephant — hence the elephant logos all over the ecosystem.

~2010Google · internal

Google moves on: GFS → Colossus (its successor file system); MapReduce → newer engines (Dremel → BigQuery; FlumeJava/Millwheel → Dataflow). The papers keep coming (Dremel 2010, Spanner 2012…).

2012–14Open source

Hadoop 2 splits scheduling out as YARN; Apache Spark arrives — in-memory processing, ~10–100× faster than disk-based MapReduce, and largely replaces it as the ecosystem's engine.

2014–16Google Cloud

Google sells it all as services: BigQuery (Dremel, GA 2011), Dataproc (managed Hadoop/Spark, 2016), Dataflow (2015; its programming model donated to Apache as Beam).

Memory hook: green dots = Google's internal world, blue dots = the open-source echo. Dataproc is Google hosting the echo; BigQuery and Dataflow are Google selling what it built after moving on. When an exam question says "existing Hadoop/Spark jobs" → echo world → Dataproc. "New pipeline / serverless" → Google's own next-gen → Dataflow or BigQuery.

1 · Colossus — the floor under (almost) everything

Colossus is Google's cluster-level file system, the successor to GFS. You never touch it directly and it has no console page — but it sits underneath nearly every GCP storage product: Cloud Storage, BigQuery storage, Firestore, Bigtable, Spanner, Persistent Disk snapshots…

Why you should care #1 — durability for free

Colossus encodes and replicates data across many disks and machines automatically. That's why Cloud Storage can promise 11 nines (99.999999999%) annual durability on every class, and why you never run fsck or worry about a dead disk in GCP. The hard distributed-systems work is amortized across all of Google.

Why you should care #2 — storage is separated from compute

This is the architectural idea of modern GCP analytics. In classic Hadoop, data lives on the disks of the same machines that process it (HDFS) — to get more compute you buy more disks too, and vice versa. On GCP, data sits in Colossus-backed storage and compute is rented separately, by the second. BigQuery can throw thousands of CPUs at data it doesn't store locally, because Google's internal network (Jupiter) is fast enough to make remote reads feel local.

Consequences of separation:
  • Scale storage and compute independently — petabytes stored, zero CPUs idle.
  • Clusters become disposable: a Dataproc cluster reading from Cloud Storage can be deleted after the job with no data loss. Pay for compute only while it runs.
  • Many engines can read the same data simultaneously (BigQuery, Dataproc, Dataflow all reading one Cloud Storage bucket).
Memory hook: Colossus is the foundation slab — invisible, shared, load-bearing. Products are rooms built on it. You rent rooms (compute); the slab (durable storage) is part of the building.

2 · Hadoop anatomy — three parts + an ecosystem

Core Hadoop = storage + scheduler + processing model. Know which layer each name belongs to.

LayerComponentWhat it does
StorageHDFSHadoop Distributed File System — chunks files into blocks, replicates them (default 3×) across cluster nodes' local disks. The GFS clone. This is the part Cloud Storage replaces on GCP.
SchedulerYARNYet Another Resource Negotiator — decides which jobs get which CPUs/memory on the cluster. Added in Hadoop 2 so engines other than MapReduce (notably Spark) could share a cluster.
ProcessingMapReducethe original programming model (below). Mostly superseded by Spark, but the model still underlies everything.

MapReduce in one worked example — word count

Count word frequencies across a huge pile of text files:

SPLIT    input is chunked; each chunk goes to a worker

MAP      each worker emits (word, 1) for every word it sees
         worker A: ("data",1) ("cloud",1) ("data",1)
         worker B: ("cloud",1) ("data",1)

SHUFFLE  framework groups by key across workers
         "data"  → [1,1,1]      "cloud" → [1,1]

REDUCE   one worker per key sums its list
         ("data", 3)            ("cloud", 2)

You write only map() and reduce(); the framework handles splitting, shuffling, retries, and dead workers. That was the 2004 breakthrough: parallelism without writing distributed code.

"Hadoop" colloquially = the whole zoo

In job ads and exam questions, "a Hadoop cluster" almost never means just HDFS — it means the ecosystem that grew on top:

Sparkfast in-memory engine; the modern default for processing
HiveSQL layer that compiles queries to MapReduce/Spark jobs
HBasewide-column NoSQL database on HDFS (clone of Google's Bigtable paper)
Pig, Flink, Presto…more engines and query layers; all elephant-adjacent logos

Mapping: Hive → BigQuery, HBase → Bigtable, Spark → Dataproc/Dataflow. Migration questions are mostly this table.

Memory hook: HDFS = the warehouse, YARN = the foreman handing out shifts, MapReduce/Spark = the workers. "Hadoop" = the whole industrial park, elephant statue at the gate.

Three ways to process big data on GCP. The exam tests whether you can pick the right one from a one-sentence scenario.

Dataproc detail that matters: because of storage/compute separation (§1), the modern pattern is ephemeral clusters — store data in Cloud Storage, spin up a cluster per job via the GCS connector, delete it when done. A long-lived, mostly idle cluster on HDFS is the anti-pattern (and a wrong answer on the exam).
BigQuery as the endgame: many migrations skip the cluster entirely — workloads that ran Hive-on-Hadoop "analytics clusters" are often replaced outright by BigQuery: same SQL idea, zero cluster management, scales further.

Exam phrasings → answers

"analyze with standard SQL", "data warehouse", "ad-hoc analytics, no infrastructure"BigQuery

Memory hook: Dataproc = procure a cluster (you own machines for a while). Dataflow = data flows, you don't manage (serverless pipelines). BigQuery = the query is the interface (SQL, nothing else).

4 · Pub/Sub — the messaging glue

Pub/Sub (publish/subscribe) is GCP's global, serverless messaging service — the shock absorber between systems that produce events and systems that consume them, at any rate, without either knowing about the other.

 publisher ──► TOPIC ──► subscription A ──► subscriber (e.g. Dataflow)
 (app, IoT,         └──► subscription B ──► subscriber (e.g. archiver)
  service)
Publisheranything that sends a message (a web app, a sensor, a Cloud Function)
Topicthe named channel messages are published to — publishers know only the topic
Subscriptiona named attachment to a topic; each subscription gets its own full copy of every message (fan-out). Messages wait here until acknowledged (up to 7 days by default).
Subscriberconsumes from a subscription — pull (subscriber asks) or push (Pub/Sub HTTP-POSTs to an endpoint)
The canonical streaming pattern (memorize this pipeline): 
  events ──► Pub/Sub ──► Dataflow ──► BigQuery
            (ingest,     (transform,   (store +
             buffer)      window,       analyze)
                          aggregate)
Pub/Sub absorbs bursts and guarantees at-least-once delivery; Dataflow reads the subscription, transforms/aggregates in streaming mode; BigQuery receives clean rows for analysis. Any exam question describing "ingest streaming events → process → analyze" is sketching this exact diagram.
Memory hook: Pub/Sub is a post office with mailboxes — publishers drop letters at the topic; every subscription is a separate mailbox that gets a copy; letters wait until the subscriber signs for them (acknowledges).

5 · BigQuery analytics depth

One layer below "serverless SQL warehouse": how it stores, how it charges, and how it reaches data it doesn't own.

Columnar storage — why scans are fast and billed per column

Row stores (Cloud SQL) keep each record together — great for "fetch order #42". BigQuery stores each column together (Capacitor format, on Colossus). SELECT revenue FROM sales reads only the revenue column off disk — maybe 2% of the table's bytes. Analytics queries touch few columns of many rows, so columnar wins by orders of magnitude. Corollary: SELECT * reads every column and bills you for all of it.

Slots — the hidden compute units

A slot is BigQuery's unit of query compute (roughly a virtual CPU + memory). When you run a query, BigQuery silently grabs hundreds or thousands of slots from a shared pool, fans the work out, and releases them — separation of storage and compute (§1) in its purest form. On-demand projects get a default pool (~2,000 slots); heavy users buy reservations (committed slots) for predictable cost.

Pricing + cost control

On-demand queriespay per TB of data scanned (~$6.25/TB, first 1 TB/month free). The query's cost is set by bytes read, not rows returned.
Storagecheap, per GB-month; drops ~50% automatically for tables untouched 90 days (long-term storage)
Partitioningsplit a table by a date/integer column; a query filtered to WHERE day = '2026-06-01' scans one partition, not the whole table. Cost-control tool #1.
Clusteringsort data within partitions by up to 4 columns; filters on those columns skip blocks. Cost-control tool #2.
Dry run / previewthe console shows estimated bytes before you run; bq query --dry_run from the CLI. Table preview is free — don't SELECT * LIMIT 10 to peek (LIMIT does not reduce bytes scanned).
Quotasset maximum bytes billed per query or per user/day to cap accidents

External tables & federated queries — query data BigQuery doesn't store

Cloud Storageexternal tables over CSV/JSON/Parquet/Avro files in a bucket — query files in place, no load step
Bigtableexternal table over a Bigtable instance — SQL over NoSQL for analytics
Cloud SQL / Spannerfederated queries: EXTERNAL_QUERY() pushes SQL to the live database and joins results with warehouse tables

Trade-off: convenience over speed — external reads are slower than native Capacitor storage and skip some optimizations. Load the data natively when it's queried often.

The canonical pattern: export billing & logs into BigQuery

GCP's own answer to "analyze my GCP usage" is BigQuery:

Billing exportenable in the Billing console → detailed cost rows land in a BigQuery dataset → SQL/dashboards over spend per project, service, label. The ACE answer to "analyze costs in detail".
Log sinksCloud Logging sink with a BigQuery destination → audit/app logs become queryable tables. The answer to "analyze logs with SQL" or "retain logs beyond Logging's retention".

Pattern to recognize: when any GCP service offers an "export for analysis" feature, the destination is BigQuery.

Memory hook: BigQuery bills like a photocopier — you pay per page scanned, not per page you keep. Partitioning/clustering = telling it which chapter to copy. SELECT * = photocopying the whole library.
