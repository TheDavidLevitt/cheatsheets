1 · VPC — your software-defined private network

A VPC (Virtual Private Cloud) is your own private network drawn in software on top of Google's physical one: a named container in which your VMs get private IP (Internet Protocol) addresses, talk to each other, and are walled off from every other customer. Every VM lives in exactly one VPC; firewall rules and routes are properties of the VPC, not of individual machines.

THE GCP quirk to hammer: VPCs are GLOBAL, subnets are REGIONAL. One VPC spans every Google Cloud region on Earth; you carve regional subnets out of it where you need them. Two VMs in Tokyo and Belgium on the same VPC reach each other over private IPs on Google's backbone — no VPN (Virtual Private Network), no public internet. This differs from AWS and Azure, where a VPC/VNet is locked to one region and cross-region needs explicit peering. Exam questions love this asymmetry.

Three subnet modes

ModeWhat you get
Auto modeGCP pre-creates one subnet in every region (out of 10.128.0.0/9), and adds more as new regions launch. Convenient, wasteful, and its fixed ranges can collide with networks you later want to connect.
Custom modestarts empty; you create only the subnets you need, with ranges you choose. Best practice and the exam answer for production. Auto → custom conversion is one-way (you can convert auto to custom, never back).
The default networkevery new project ships with an auto-mode VPC named default, plus permissive starter firewall rules (allow SSH/RDP/ICMP from anywhere). Fine for labs; delete or replace it for anything real.

Memory hook: the VPC is the country (one set of laws = firewall rules everywhere); subnets are cities (each has a street plan = an IP range, and a location = a region). You pass laws once; you build streets city by city.

2 · Subnets & CIDR — slicing the address space

A subnet (subnetwork) is a regional slice of private IP space, defined by a CIDR (Classless Inter-Domain Routing) range. A VM's internal IP comes from the subnet it's attached to.

Decoding CIDR: 10.0.1.0/24

An IPv4 address is 32 bits, written as four 8-bit decimal numbers (octets). The /N suffix says: the first N bits are fixed (the network part); the remaining 32−N bits vary (the host part).

10.0.1.0/24
└──────┘ └┘
network  24 bits fixed → 32−24 = 8 bits free
         → 2⁸ = 256 addresses: 10.0.1.0 … 10.0.1.255

smaller /N  = bigger block   (/16 → 65,536 addresses)
bigger  /N  = smaller block  (/29 → 8 addresses, GCP's minimum)

GCP reserves 4 addresses per subnet: the first two (network address + gateway) and the last two (second-to-last reserved, last = broadcast). A /24 really gives you 252 usable VM addresses. History hook: "classless" replaced the rigid 1980s class A/B/C system (/8, /16, /24 only) in 1993 — CIDR lets you cut blocks at any bit boundary.

docs: subnets & reserved addresses

RFC 1918 — the private ranges

RFC 1918 (Request for Comments 1918, the 1996 internet standard) sets aside three IPv4 blocks that are never routed on the public internet — everyone reuses them privately, which is why your home router and your VPC can both be 192.168.x.x without conflict:

10.0.0.0/8~16.7M addresses — the usual choice for cloud VPCs
172.16.0.0/12~1M addresses
192.168.0.0/16~65k addresses — home-router territory

Subnet ranges in one VPC must not overlap each other — and plan ahead: they also must not overlap any network you'll later peer or VPN to. Renumbering a live subnet is painful; a spreadsheet of allocated ranges is cheap.

Why multiple subnets at all?

Per-region presencea VM must sit in a subnet in its own region — serving Europe and Asia means a subnet in each
Tier segmentationseparate web / app / database subnets so firewall rules can say "only the web subnet's range may reach the db subnet on tcp:5432"
IP-range planningdeliberate, non-overlapping blocks keep future peering, VPN, and hybrid connectivity possible

Subnets can be expanded in place (/24 → /23) without downtime — but never shrunk. Start modest.

3 · IP addresses — internal, external, ephemeral, static

Internal vs external

Internal IPExternal IP
private (RFC 1918) address from the subnet's range; every VM has one; how VMs talk to each other inside the VPC; free public, internet-routable address; optional; how the internet reaches the VM (and, without NAT, how the VM reaches the internet); billed

The VM itself only ever sees its internal IP (check ip addr inside — no public address there). The external IP is a 1-to-1 NAT (Network Address Translation) mapping that Google's fabric performs at the edge.

Ephemeral vs static external IPs

Ephemeral (default)borrowed from Google's pool; released when the VM stops — restart and you get a different address. Greek ephēmeros, "lasting a day."
Static (reserved)an address you reserve by name; it stays yours across stops, restarts, even reattachment to a different VM — until you explicitly release it

When static matters: a DNS A record pointing at the VM, a third-party webhook configured to call your endpoint, a partner's firewall allowlist with your address in it. Anything where someone else wrote your IP down. You can also promote a VM's current ephemeral IP to static in place (addresses create --addresses=<current-ip>).

docs: IP addresses

Wallet hooks

External IPv4, in usenow billed (~$0.005/hr ≈ $3.6/month per address) — external IPs stopped being free in 2024
Static IP, reserved but unattachedbills more than one in use — the idle-reservation penalty. Release what you don't use.
Internal IPsfree

The cheapest external IP is no external IP: private VM + Cloud NAT for outbound + IAP (Identity-Aware Proxy) for admin SSH — the pattern diagrammed in §6.

4 · Single-stack vs dual-stack — IPv4 and IPv6

Why two protocols exist: IPv4 addresses are 32-bit → only ~4.3 billion of them, and the pool is exhausted (the last free blocks were handed out in 2011–2019). NAT everywhere is the workaround: thousands of private devices hide behind one public IPv4. IPv6 uses 128-bit addresses (written hex, e.g. 2600:1900:4000::1) → ~3.4×10³⁸ of them, effectively unlimited — every device can have a globally unique address and NAT becomes unnecessary. Adoption is the slow part, hence the transition strategy: dual-stack, where a machine speaks both protocols at once.

Single-stack (GCP default)IPv4 only — the default for every subnet and VM
Dual-stackIPv4 + IPv6, opted into per subnet (--stack-type=IPV4_IPV6) and then per VM; the VM gets a /96 IPv6 range alongside its IPv4 address

ACE depth: recognize that the option exists, know the default is single-stack IPv4, and know dual-stack is enabled at the subnet level. Don't memorize IPv6 internals for this exam.

5 · Firewall rules — the VPC's bouncer

GCP firewall rules are defined at the VPC level and enforced at every VM — there's no separate appliance to install. They are stateful: if a connection is allowed in, its reply traffic is automatically allowed back out (and vice versa). You never write a "reply" rule.

Anatomy of a rule

DirectionINGRESS (incoming, the default) or EGRESS (outgoing)
Actionallow or deny
Targetswhich VMs the rule applies to: all instances, network tags (labels like web-server you stick on VMs), or service accounts (sturdier — tags can be edited by anyone with VM access; SA targeting follows identity)
Source / destinationCIDR ranges, tags, or service accounts (e.g. ingress from 203.0.113.7/32)
Protocol : porttcp:22, tcp:80,443, icmp, all
Priorityinteger 0–65535; lower number = stronger (evaluated first; first match wins). Default 1000.

The two implied rules (priority 65535)

Implied deny all ingressnothing gets in unless a rule allows it — a brand-new custom VPC is closed to the world
Implied allow all egresseverything may leave unless a rule denies it

Memory hook: the default posture is a one-way mirror — VMs can look out, nobody can look in. Sitting at the weakest possible priority (65535), the implied rules lose to any rule you write.

"My VM is unreachable" debugging order: is there an allow ingress rule for that port, with a target (tag/SA) the VM actually has, from a source range that includes the caller?

The three classic rules

# SSH, but only from your own address
gcloud compute firewall-rules create allow-ssh-admin \
  --network=prod-vpc --direction=INGRESS --action=ALLOW \
  --rules=tcp:22 --source-ranges=203.0.113.7/32 \
  --target-tags=ssh-allowed

# web traffic from anywhere
gcloud compute firewall-rules create allow-http \
  --network=prod-vpc --allow=tcp:80,tcp:443 \
  --source-ranges=0.0.0.0/0 --target-tags=web-server

# IAP's fixed source range (the no-external-IP pattern, §6)
gcloud compute firewall-rules create allow-iap-ssh \
  --network=prod-vpc --allow=tcp:22 \
  --source-ranges=35.235.240.0/20

0.0.0.0/0 = "every IPv4 address" (0 fixed bits); /32 = exactly one address. 35.235.240.0/20 is Google's fixed IAP range — the one CIDR worth memorizing.

Exam trap: firewall rules don't conflict by order written — only by priority number. A deny tcp:22 at priority 900 beats an allow tcp:22 at priority 1000, regardless of creation order. Equal priority: deny wins.

6 · Routes & Cloud NAT — the no-external-IP pattern

Routes tell packets leaving a VM where to go next. Every VPC auto-creates a route per subnet (keeps internal traffic internal) plus a default route to 0.0.0.0/0 via the internet gateway. But the internet gateway only carries traffic for VMs that have an external IP — a private-only VM hits the default route and goes nowhere. Enter Cloud NAT.

Cloud NAT — outbound-only internet

Cloud NAT is a regional, fully managed NAT service: it lets private VMs initiate outbound connections (pull OS updates, call external APIs) by translating their private source addresses to a pool of public IPs it manages. Crucially it is outbound-only — it never accepts inbound connections, so the VMs remain unreachable from the internet. It rides on a Cloud Router (the regional control-plane resource that hosts it; no VM to manage, no single point of failure).

Outbound (VM → internet)✔ allowed, source-translated by NAT
Inbound (internet → VM)✘ impossible — no listening address exists
Admin SSHvia IAP tunnel — Google's proxy checks your identity, then forwards to the internal IP

Why this is the answer: zero internet-facing attack surface (no port 22 exposed, no port anything), zero external-IP cost, full outbound connectivity, and SSH gated by IAM (Identity and Access Management) instead of by whoever finds the address. "Most secure way to give private VMs internet access / SSH" on the exam → Cloud NAT + IAP.

7 · Connecting VPCs — peering & Shared VPC (recognition level)

VPC Network Peering

Connects two VPCs (same or different projects, even different organizations) so they exchange traffic over private internal IPs — no public internet, no VPN, no gateway hardware. Both sides must configure the peering; each keeps its own firewall rules and administration. Not transitive: A↔B and B↔C does not give A↔C. And it only works if the CIDR ranges don't overlap — the payoff of the range planning in §2.

docs: VPC peering

Shared VPC

One host project in an organization owns the VPC; multiple service projects attach to it and create their resources (VMs, etc.) directly in the host's subnets. One network team administers subnets, firewall rules, and routes centrally; application teams keep separate projects (and separate billing/IAM) without each building a network. Exam cue: "central network administration across many projects in an org" → Shared VPC; "connect two existing networks privately" → peering.

docs: Shared VPC

8 · Command quick reference

Grammar reminder: gcloud compute <resource> <verb> [name] [--flags]. Networks are global (no flag); subnets, addresses, routers, and NAT take --region.

gcloud compute networks create prod-vpc --subnet-mode=customnew empty custom-mode VPC (vs --subnet-mode=auto)
gcloud compute networks list / describe prod-vpcinventory / details of VPCs
gcloud compute networks subnets create web-subnet --network=prod-vpc --region=europe-west1 --range=10.0.1.0/24carve a regional subnet out of the VPC
gcloud compute networks subnets expand-ip-range web-subnet --region=europe-west1 --prefix-length=23grow a subnet in place (/24 → /23); never shrinks
gcloud compute addresses create web-static-ip --region=europe-west1reserve a regional static external IP (add --addresses=<ip> to promote a VM's current ephemeral one)
gcloud compute addresses listall reserved IPs + status — RESERVED means unattached and billing extra
gcloud compute addresses delete web-static-ip --region=europe-west1release a static IP (stops the idle billing)
gcloud compute firewall-rules create allow-http --network=prod-vpc --allow=tcp:80 --source-ranges=0.0.0.0/0 --target-tags=web-serverallow ingress web traffic to tagged VMs
gcloud compute firewall-rules list --filter="network=prod-vpc"audit a VPC's rules
gcloud compute routers create prod-router --network=prod-vpc --region=europe-west1the Cloud Router that will host NAT
gcloud compute routers nats create prod-nat --router=prod-router --region=europe-west1 --auto-allocate-nat-external-ips --nat-all-subnet-ip-rangesCloud NAT for every subnet in the region — outbound internet for private VMs
gcloud compute instances create my-vm-01 --subnet=web-subnet --no-address --zone=europe-west1-b--no-address = no external IP — the §6 pattern's VM
gcloud compute ssh my-vm-01 --zone=europe-west1-b --tunnel-through-iapSSH into that private VM via IAP
