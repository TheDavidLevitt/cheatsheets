Authentication & OAuth — how login actually works

0 · Authentication vs authorization — two different questions

Authentication (AuthN) answers "who are you?" — proving an identity. Authorization (AuthZ) answers "what may you do?" — granting capabilities to that identity. Airport version: showing your passport at the gate is authentication; whether your ticket says economy or cockpit is authorization.

Why the distinction matters

The two are implemented by different systems that are easy to blur. A login form, a fingerprint reader, and a one-time code are all AuthN. An IAM (Identity and Access Management) policy, a file's permission bits, and an OAuth scope are all AuthZ. Confusing them produces classic bugs: a perfectly authenticated user reaching admin pages (missing AuthZ check), or AuthZ rules keyed to a username nobody verified (missing AuthN).

The twist that powers this whole sheet

OAuth 2.0 is an authorization protocol — its job is delegating access ("this app may read my calendar"), not proving identity. The identity layer was bolted on later as OIDC (OpenID Connect, §5). Etymology check: the name "OAuth" comes from Open Authorization — the AuthZ half is right there in the name. Keep that in mind and half the confusion in this domain evaporates.

1 · The pre-OAuth dark age — the password anti-pattern

Circa 2006, a new app that wanted to read your Gmail contacts or post to your Twitter had exactly one option: ask for your actual password and log in as you.

Why password-sharing is catastrophic, point by point:
Total accessthe app didn't get "read contacts" — it got everything: read mail, send mail, delete the account. No way to scope down.
No revocationthe only way to cut one app off was to change your password — which broke every app at once.
Storage liabilitythe app had to store your password (often plaintext) to keep working. Every such app was a breach waiting to happen.
No attributionthe provider couldn't tell the app's requests from yours — no per-app audit trail, no per-app rate limits.
History hook: OAuth was born in 2006–07 out of exactly this pain — Twitter (Blaine Cook, working on OpenID) and Ma.gnolia needed a way for third-party apps to act on a user's behalf without the password. OAuth 1.0 shipped in 2007 (RFC 5849); the simpler, TLS-reliant OAuth 2.0 (RFC 6749, 2012) replaced it and is what everyone means by "OAuth" today. The problem statement was never "how do users log in" — it was "how do we delegate limited access without handing over the keys".

Docs: RFC 6749 — The OAuth 2.0 Authorization Framework

2 · OAuth 2.0 — four actors, one valet key

OAuth 2.0 is a delegation framework: a way for you to grant an app limited, revocable access to your stuff held by a third party. Four roles appear in every flow:

ActorRFC nameWho it is
YouResource ownerthe human who owns the data and can consent to sharing it
The appClientthe application that wants access — a web app, mobile app, or CLI (command-line interface) tool. "Client" here means client of the authorization server, not your browser.
The ID deskAuthorization serverwhere you authenticate and consent; it mints tokens (e.g. accounts.google.com)
The APIResource serverthe service holding your data; it accepts tokens instead of passwords (e.g. gmail.googleapis.com)
The valet-key analogy (the canonical one): a luxury car's valet key starts the engine but won't open the trunk or glovebox, and you can take it back anytime. An OAuth access token is a valet key for your data: scoped (only what was consented), revocable (cut off one app without touching others), and expiring (useless after ~an hour even if stolen). That triple — scoped, revocable, expiring — is the entire upgrade over password-sharing, and the key insight of this sheet.

3 · Flows — choreographies for getting a token

A flow (RFC term: grant type) is the choreography by which a client obtains tokens. Pick by what kind of client you are:

Authorization Code + PKCE — today's standard

For anything with a user and a browser: web apps, mobile apps, CLIs that pop a browser tab. The client never sees your password; it receives a one-time authorization code via browser redirect and swaps it for tokens over a back channel. PKCE (Proof Key for Code Exchange, RFC 7636, pronounced "pixy") adds a per-request secret so a stolen code is worthless — originally for mobile apps that can't keep a client secret, now recommended for every client.

Client Credentials — machine-to-machine

No human at all: a backend service authenticates as itself (client ID + secret, or a signed assertion) and gets a token for its own access. This is the shape of service-to-service auth everywhere — including GCP service accounts (see the GCP IAM sheet).

Device Flow — TVs & keyboards-from-hell

RFC 8628, for devices where typing a password is misery (TVs, consoles, IoT). The device shows a short code (WDJB-MJHT) and a URL; you finish login on your phone; the device polls until the authorization server says yes. You've done this every time you signed into a streaming app on a TV.

Implicit — deprecated, recognize and avoid

The old single-page-app shortcut: tokens returned directly in the redirect URL fragment, skipping the code exchange. Tokens leaked via browser history, referrer headers, and JavaScript. OAuth 2.0 Security Best Current Practice (RFC 9700) says: don't. Use Authorization Code + PKCE instead. If a tutorial shows response_type=token, it's a museum piece.

The authorization-code dance, step by step

You + browserresource owner
Appclient
Auth serverthe ID desk
APIresource server
1click "Connect"app redirects browser →
2redirect to auth serverwith client_id, scopes, redirect_uri, PKCE challenge

Why the two-channel split? The front channel (browser redirects, steps 2–4) is observable — URLs land in history, logs, referrers — so it only ever carries the short-lived, single-use code. The back channel (server-to-server HTTPS, steps 5–6) carries the actual tokens. PKCE welds the two halves together: the app invents a random secret, sends its hash in step 2, and reveals the secret in step 5 — so an attacker who steals the code from the front channel can't cash it in.

4 · Tokens — the three you'll meet

Access token — the concert wristband

Short-lived (commonly ~1 h), sent on every API call in the Authorization: Bearer … header. A bearer token means exactly what it sounds like: whoever bears it, wins — the API doesn't ask the holder to prove anything else, just like a festival wristband gets anyone wearing it through the gate.

Refresh token — the key to the mint

Long-lived, stored by the client, never sent to the resource server. Its only job: go back to the authorization server and mint fresh access tokens, so the user isn't re-prompted every hour. Revoking an app = invalidating its refresh token. (This is exactly what gcloud auth login stores on disk.)

ID token — the introduction card

An OIDC addition (§5): a JWT (§6) about who you are — name, email, issuer — addressed to the client itself, not to any API. Use it to log the user in; use the access token to call APIs. Sending an ID token to an API is the classic token mix-up.

Bearer risk → two mitigations. Since possession is everything, a stolen bearer token is a stolen identity for its lifetime. Defenses: ① short lifetimes (an hour of exposure, not a year — the refresh token, which never travels to APIs, absorbs the long-lived role) and ② TLS (Transport Layer Security) everywhere, so tokens are never readable in transit. This is also why OAuth 2.0 could drop OAuth 1.0's painful per-request cryptographic signatures: it outsourced confidentiality to HTTPS.

Docs: RFC 6750 — Bearer token usage · oauth.net — refresh tokens

5 · OIDC — the identity layer on top of OAuth

OIDC (OpenID Connect, 2014) is a thin layer on OAuth 2.0 that turns a delegation protocol into a login protocol. The slogan worth memorizing: OAuth proves the app MAY access something; OIDC proves WHO you are.

What OIDC adds to plain OAuth

ID tokena signed JWT carrying identity claims (who, issued by whom, when, for which app)
openid scoperequesting it is what makes an OAuth flow an OIDC flow; add profile, email for more claims
UserInfo endpointstandard API to fetch the user's profile with the access token
Discoverya well-known URL (/.well-known/openid-configuration) where the provider publishes all its endpoints and keys

"Sign in with Google" = OIDC

Every social-login button — Google, Microsoft, Apple, GitHub — is OIDC (or OIDC-shaped). The flow is the same authorization-code dance from §3; the app just asks for the openid email profile scopes and reads the ID token instead of (or in addition to) calling APIs. The provider is called an IdP (Identity Provider); the app is the RP (Relying Party). History hook: "OpenID" (2005) was a separate login protocol that lost; OIDC kept the brand and rebuilt the idea on OAuth 2.0's rails.

Docs: OIDC Core spec · Google — OpenID Connect

6 · JWT anatomy — header.payload.signature

A JWT (JSON Web Token, RFC 7519, pronounced "jot") is a self-contained token: three base64url-encoded segments joined by dots. Decode one at jwt.io — it's just JSON.

eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9 . eyJpc3MiOiJodHRwczovL2FjY291bnRzLi4uIn0 . SflKxwRJSMeKKF2QT4f…
└────────── header ──────────┘  └───────── payload ─────────┘  └──── signature ────┘

header   = {"alg": "RS256", "typ": "JWT"}          which signature algorithm
payload  = {"iss": "https://accounts.google.com",   the CLAIMS (see below)
            "sub": "10769150350006150715113082367",
            "aud": "example-client-id-123.apps.googleusercontent.com",
            "exp": 1767139200, "iat": 1767135600,
            "email": "alex@example.com"}
signature = RS256(header + "." + payload, issuer's private key)

The four claims to know

iss issuerwho minted the token — verify it's the IdP you expect
sub subjectstable unique ID of the user (use this, not email, as the primary key — emails change)
aud audiencewho the token is for — reject tokens addressed to another app (the "confused deputy" check)
exp expiryUnix timestamp after which the token is dead; iat = issued-at

Why JWTs at all?

Stateless verification. The resource server validates the signature against the issuer's published public keys (fetched once from the discovery URL) — no database lookup, no call back to the auth server, scales horizontally for free. Trade-off: a JWT can't be un-issued before exp, which is another reason lifetimes stay short.

Signed ≠ encrypted. A standard JWT is only base64-encoded — anyone who obtains it can read every claim with one decode call. The signature guarantees integrity (nobody altered it) and authenticity (the issuer really minted it), not confidentiality. Never put secrets in JWT claims, and never confuse "it's a JWT" with "it's safe to leak".

Docs: RFC 7519 — JWT · jwt.io — introduction

7 · Scopes & consent screens

A scope is a named slice of access the client requests — the unit of "how much valet key". Space-separated strings in the auth request:

scope=openid email https://www.googleapis.com/auth/calendar.readonly

How scopes behave

Requestedby the client in step 2 of the dance — ask for the minimum (least privilege applies to apps too)
Consentedby the user on the consent screen; the user (or provider) may grant fewer than asked
Baked into the access token; the resource server enforces them on every call
Not rolesscopes cap what the app may do on your behalf; they never expand beyond what you can do. App's power = your power ∩ granted scopes.

The consent screen is the UI of OAuth

That "App X wants to: see your email address, view your calendar" dialog is the resource owner's one moment of agency — it lists requested scopes in human words. Provider tip: scopes touching sensitive data (full Gmail, Drive) trigger heavier app-verification requirements from Google, which is why well-behaved apps ask for *.readonly slices. Review and revoke past grants at myaccount.google.com/permissions.

8 · The credential zoo — side by side

Five things that all "log you in" but work on different principles:

Identifies a project/app (for quota & billing) — not a person a user×app×scope grant one logged-in browser session whoever holds the private key a workload (code), not a human
Lifetime until manually rotated (∞ by default) minutes–hours hours–weeks (idle/absolute timeouts) until removed from authorized_keys tokens ~1 h; downloaded key files ∞ (the anti-pattern)
Revocation delete/rotate the key revoke the refresh token; access token dies at exp server deletes the session row — instant remove the public key from the server disable the SA or delete the key
Secret travels? yes, on every request yes (bearer) — hence TLS + short life yes, on every request (HttpOnly, Secure flags) no — only signatures travel (challenge–response) no for attached/federated identities; yes if a key file leaks
Pattern to notice: the security ladder runs static shared secret (API key) → expiring bearer (OAuth) → proof-of-possession, secret never travels (SSH keys, passkeys). Each rung removes a way to steal. GCP's push from SA key files toward attached identities and federation is a climb up this exact ladder — see the GCP mapping section.

9 · One-liners — terms you'll meet next

MFAMulti-Factor Authentication — require ≥2 of know (password) / have (phone, security key) / are (fingerprint); turns a stolen password into a non-event
Passkeys / WebAuthnpasswordless login standard (W3C WebAuthn + FIDO2): your device holds a per-site private keypair and signs a challenge — nothing shared to steal, nothing to phish, the SSH-key idea brought to the web
SAML vs OIDCSAML (Security Assertion Markup Language, 2005) = XML-based enterprise SSO (Single Sign-On), still everywhere in corporate IT; OIDC = the JSON/REST-native successor — same job ("the IdP vouches for you"), newer rails. Rule of thumb: legacy enterprise apps speak SAML, modern apps and mobile speak OIDC.
