Part 1 · Orientation — the shape of the platform

Azure = Microsoft's public cloud (2010). Same league as GCP/AWS; its trait is deep integration with the corporate Microsoft stack — Entra ID identity, Microsoft 365, Power Platform. In a SharePoint/Windows company you're already half inside Azure.
The resource hierarchy — policy, billing and access flow down this tree: 
Azure                                       GCP equivalent
Tenant (one Entra ID directory)        ≈    Organization
  └─ Management groups (nestable)      ≈      Folders
       └─ Subscriptions (billing)      ≈        Projects
            └─ Resource groups         ≈          (no GCP equivalent!)
                 └─ Resources          ≈            Resources
Tenant = one org's Entra ID directory. Subscription = billing/quota unit ≈ GCP project — but Azure shops have few subscriptions and many resource groups (mandatory lifecycle bags: delete the RG, delete the app). The RG has no GCP equivalent — that's the #1 translation gotcha. Day-to-day you live in the RG, the way you live in a GCP project.

Entra ID (formerly Azure AD) ≈ Cloud Identity + IAM

Microsoft Entra ID = the cloud identity directory, renamed from Azure Active Directory (Azure AD) in 2023 — colleagues and old docs still say "Azure AD". Holds users, groups, service principals (≈ GCP service accounts), app registrations; authenticates the portal, Microsoft 365, SharePoint, Power Platform. Roles via RBAC (Role-Based Access Control), inherited down the hierarchy like GCP IAM. Not the same as on-prem Active Directory — that's the classic domain controller, usually synced into Entra ID.

Regions, in one line

A region = a geographic datacenter cluster (60+, e.g. westeurope, uaenorth); resources are created in one, regions come in pairs for disaster recovery, zones exist too — maps 1:1 to GCP regions/zones.

Part 2 · The data landscape — drill-ins

Lightest → heaviest: SharePoint List → Dataverse → Azure SQL → Fabric, with Cosmos DB (NoSQL apps) and Blob (files) on the side. Click a card to expand.
SharePoint Lists lightweight tables, not a real DB · ≈ a Google Sheet used as a database

A table-with-a-form bolted onto a SharePoint site — columns, views, attachments, versioning. The default place corporate data accumulates, because anyone can build one. Not a database: no joins, no transactions, weak typing.

The 5,000-item view threshold. A list can store 30M items, but any view/query touching > 5,000 items is throttled or fails unless its filter/sort columns are indexed. Symptom: works for months, then "this view cannot be displayed". Fix: index filter columns before the list grows; keep views filtered below the threshold.

REST endpoint — every site exposes an OData (Open Data Protocol: the $filter/$select/$top REST query grammar) API:

GET https://contoso.sharepoint.com/sites/TeamSite/_api/web/lists/
      getbytitle('Tasks')/items?$select=Title,Status&$filter=Status eq 'Open'&$top=100
Accept: application/json;odata=nometadata

Automation: Power Automate (M365 workflow tool, ex-"Flow") has first-class triggers ("when an item is created…"); programmatic access increasingly via Microsoft Graph API: graph.microsoft.com/v1.0/sites/{site-id}/lists/{list-id}/items.

Dataverse Power Platform's real database · ≈ a relational, permissioned Firestore-as-app-backend

The managed relational backbone under Power Platform (Power Apps, Power Automate) and Dynamics 365. Real typed tables (ex-"entities"), real relationships (1:N, N:N), business rules, row- and column-level security roles, auditing. Runs on Azure SQL underneath but is sold as SaaS — you never see the server.

Microsoft Fabric analytics umbrella · ≈ BigQuery + Dataproc + Looker sold as one SaaS box

Microsoft's all-in-one SaaS analytics platform (GA 2023) — bundles the former Synapse, Data Factory and Power BI into one product, one storage layer, one capacity-based license. Key idea: one copy of data in OneLake, many engines over it — the same Delta table is queried by Spark, T-SQL and Power BI without copying.

CLI: az sql server create … then az sql db create --name mydb …. Query in ordinary T-SQL (Transact-SQL, Microsoft's dialect): SELECT TOP 10 * FROM Sales.Orders WHERE Total > 1000; — note TOP 10, not LIMIT 10, the classic T-SQL tell.

Cosmos DB globally distributed NoSQL · ≈ Firestore / Bigtable hybrid

One service, several APIs: document/NoSQL, MongoDB-compatible, Cassandra, Gremlin graph, table. Single-digit-ms reads, turnkey multi-region replication; throughput billed in RU/s (Request Units per second — Cosmos's currency for "how much work a query costs"). Think Firestore's document-DB developer experience + Bigtable's scale ambitions in one product — and like Bigtable, partition-key choice dominates cost and performance.

SELECT c.name, c.city FROM c WHERE c.city = "Lisbon" ORDER BY c.name
-- c = conventional alias for the container (≈ Firestore collection)
Storage accounts / Blob object storage · ≈ GCS

A storage account (globally unique name) wraps four primitives: Blob (objects ≈ GCS), Files (SMB shares), Queues (simple messaging), Tables (key-value, mostly superseded by Cosmos). Structure: account → container (≈ bucket) → blob (≈ object). Tiers hot / cool / cold / archive ≈ GCS standard / nearline / coldline / archive.

az storage account create --name stdemo123 --resource-group rg-demo --location westeurope
az storage container create --name docs --account-name stdemo123
az storage blob upload --account-name stdemo123 --container-name docs \
    --name report.pdf --file ./report.pdf   # ≈ gcloud storage cp

Part 3 · Which data store when — a corporate PM's decision walk

Walk down, stop at the first "yes":
1 · Team tracker / intake form, < a few thousand rows?→ SharePoint List. Free with M365. Index your filter columns on day one — the 5,000-item threshold is coming.
2 · Real app data — related tables, security roles, outlives the team?→ Dataverse. Accept the premium licensing; it's the price of not rebuilding on SQL later.
3 · Developers, custom app, full SQL + transactions?→ Azure SQL Database (Managed Instance if migrating on-prem SQL Server unchanged).
4 · Analytics across systems — dashboards, history, big aggregations?→ Fabric: land in OneLake, report with Power BI. Analytics = read workloads over copies; don't run reports on the operational store.
Side doorsplanet-scale NoSQL app → Cosmos DB; files/objects → Blob.

Rule of thumb: Lists are for tracking, Dataverse is for apps, SQL is for developers, Fabric is for questions about the past.

Part 4 · Teardown

az group delete --name rg-demodelete an app and everything in the resource group in one command — this lifecycle power is the reason resource groups exist ≈ gcloud projects delete
az logoutsign out of the CLI; clears cached credentials ≈ gcloud auth revoke
Remove a SharePoint Listsite owner: List settings → Delete; lands in the site Recycle Bin ~93 days. Low ceremony — whoever built it can usually kill it.
Delete Dataverse tablesneeds a System Administrator/Customizer security role in the environment; blocked while other tables/apps depend on it. Deliberately harder than Lists — it's app infrastructure.
Delete a Fabric workspaceWorkspace Admin → workspace settings → Remove; takes its items (lakehouses, reports) with it. OneLake data in other workspaces untouched.
Stop paying entirelycancel the subscription in the portal (or downgrade Power Apps/Fabric capacity licenses) — resources are disabled, then permanently deleted after a grace period
